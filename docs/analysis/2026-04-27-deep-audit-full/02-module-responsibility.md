# 维度 02：模块职责与文件边界

## 审核范围

检查所有 `packages/*/src/` 下源码文件的大小、职责单一性、入口文件纯度。

## 超大文件清单

### >700 行（必须拆分）

| 文件 | 行数 | 类型 | 状态 |
|------|------|------|------|
| `packages/flux-renderers-form-advanced/src/composite-field/object-field.test.tsx` | ~755 | 测试 | P2 |
| `packages/nop-debugger/src/controller-inspect.test.ts` | ~750 | 测试 | P2 |

### 500-700 行（需评估）

共 16+ 个测试文件超过 500 行，均为 `.test.ts` 或 `.test.tsx`。源码文件全部在 500 行以下。

### [维度02] object-field.test.tsx 超过 700 行

- **文件**: `packages/flux-renderers-form-advanced/src/composite-field/object-field.test.tsx:1-755`
- **严重程度**: P2
- **现状**: 测试文件 755 行，包含 object-field 的全面测试（渲染、验证、嵌套、联动等）。
- **风险**: 测试定位困难，维护成本高。
- **建议**: 按测试领域拆分（渲染测试、验证测试、嵌套测试、联动测试）。
- **为什么值得现在做**: 持续增长的测试文件，拆分成本不高但收益明显。
- **误报排除**: 非源码文件，但项目规则 >700 行必须拆分适用于测试文件。
- **历史模式对应**: 本仓库曾成功拆分多个超大文件（flux-core/src/index.ts、types.ts 等）。
- **复核状态**: 维度复核通过

### [维度02] controller-inspect.test.ts 超过 700 行

- **文件**: `packages/nop-debugger/src/controller-inspect.test.ts:1-750`
- **严重程度**: P2
- **现状**: 调试器控制器检查测试 750 行。
- **风险**: 同上。
- **建议**: 按检查功能域拆分。
- **为什么值得现在做**: 同上。
- **误报排除**: 同上。
- **历史模式对应**: 同上。
- **复核状态**: 维度复核通过

## 入口文件检查

所有包的 `index.ts` 均为纯 re-export barrel 文件，未包含具体实现逻辑。导出项数量合理（最大的 flux-core 约 80+ 导出，反映其作为基础类型包的广度）。

## 目录结构

- 各包 `src/` 顶层文件数量合理（<20 个）
- 无过度拆分的单文件子目录
- 文档与代码文件所有权基本一致

## 总结评估

源码文件全部在 500 行以下，职责划分良好。2 个测试文件超过 700 行需要拆分，16+ 个测试文件在 500-700 行需评估。入口文件纯度良好。
