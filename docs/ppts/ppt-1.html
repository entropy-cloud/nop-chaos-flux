高速生成下系统没有解体：效率只是副产品

1 名开发者 + AI，6 周（1.5 人月）

保守估计传统工作量：4-6 名高级前端工程师，3-4 个月 = 12-24 人月

真正罕见的不是快

包边界越来越硬：flux-compiler / flux-action-core / flux-runtime 三层分离

结构越来越纯：CompiledSchemaNode 等过渡结构被清除

审计越来越苛刻：deep audit 后经过 meta-review 和 re-verification 继续收敛

真正罕见的不是快，而是在这种快里系统没有解体。

核心问题：AI 生成是扩张力，没有收敛机制就会持续漂移

高带宽生成

AI 能在极短时间内产出大量代码、测试、文档

乐观偏差

AI 倾向于流畅地叙述"已经完成"，即使行为闭环并未真正建立

上下文漂移

旧前提在长上下文里不断被复用，整组文件一起偏离真实基线

最危险的地方

不是"写错一段代码"，而是局部都像对的，但整体在持续偏离。

局部都像对的，往往正是系统正在整体跑偏的时候。

我们找到的答案：把开发组织成一个受控收敛的动力系统

吸引子（Attractor）

一组能持续把实现拉回正确方向的稳定约束。架构基线 / owner doc / 文档权威链

扩张力（Divergence）

AI 生成代码 / 文档 / 方案 / 候选问题，把系统推向更大的状态空间

收敛力（Convergence）

Plan / lint / test / audit / closure / logs —— 把扩张结果压回吸引子附近

没有吸引子，AI 不是在开发系统，而是在高效采样错误解空间。

关键不是压制扩张，而是让扩张有去处。

什么是动力系统，什么是吸引子

动力系统

状态随时间演化、且下一步依赖当前状态的系统。关心的是轨迹。

吸引子

状态空间中的稳定区域——不管初始条件怎么抖动，轨迹最终会被限制在它附近。

混沌不等于随机；局部不可预测，不等于整体失控。

仓库真相带来的工程后果：生成和验收被真正拆开

传统开发

仓库 = 认知的外化。作者认知持有系统模型，仓库主要是表达媒介。

AI 深度参与后

仓库 = 系统真相的载体。代码、diff、logs、tests、docs 成为系统状态的主体。

对应关系

test / lint / audit = 测量
owner doc / plan / closure = 约束
logs / history / bugs = 轨迹记录

当作者不再持有完整模型，生成和验收就必须依赖仓库里的外部证据重新接起来。

这个模型和传统开发的关键区别

瀑布

先设计，再实现，再测试

蓝图优先，阶段分明

默认人类可同步把关

敏捷

迭代设计，快速反馈

设计与实现交替推进

默认团队仍能持续跟上系统演化

AI 深度参与

生成速度超出人脑自然收敛速度

生成与验收被拆开

重点变成系统轨迹控制与受控收敛

再用流水线思维管理 AI 高速生成，只会把混乱组织得更整齐。

吸引子的工程实现 1：架构基线与文档权威链

  • 可校验的架构约束

    七个原语闭合集、编译优先、数据/能力正交、统一渲染器合约、严格包边界

  • 固定在 docs/architecture/ 的 owner doc 中

    决定每次扩张的合法方向

文档权威链

docs/index.md 导航入口
docs/architecture/ 定义现行基线
docs/analysis/ 提供候选发现
docs/plans/ 负责执行收口
docs/logs/ 保存开发记忆

架构的价值不在于描述一切，而在于让错误结构无法合法存在。

吸引子的工程实现 2：保留澄清过程，而不只保留结论

  • 保留 docs/discussions/

    不只写最终结论，也保留澄清过程：原始表达、AI 复述与提问、用户补充、纠正记录、最终总结

  • 后续 AI 不用凭空猜"为什么这样设计"

    也能看到"为什么不能那样设计"

  • 早期理解偏差往往比编码错误更致命

    如果只保留最后结论，吸引子就会弱化

扩张力的角色定义：AI 负责展开，不负责最终裁决

AI 主要承担扩张

代码生成、文档整理、方案展开、审计候选生成

没有交给 AI 自主决定的

架构裁决、概念发明、边界重定义和方案淘汰

人机分工

人定义边界和收口标准

AI 高速扩张

多 agent 交叉验证把结果拉回基线

收敛规则由人工抽样校准

不让同一个上下文既做实现又做收敛裁判。

收敛力 1：Plan 从"路线图"变成"收口机制"

Plan 是局部收敛器

每个 Plan 至少要写清：Current Baseline、Goals/Non-Goals、Exit Criteria、Validation Checklist、Closure Audit Evidence

Plan 不是路线图，它是局部收敛器。

写计划前先核对 live repo

不沿用旧计划和旧 completion note

scope 太宽 → 拆 successor plan

每个 plan 只负责一个可关闭的结果面

接口存在 ≠ 行为落地

closure audit 必须区分 interface presence 和 behavior landing

Plan 作为收敛力的实际效果

漂移模式（Plan 机制解决前）

计划过宽，没人知道什么时候算完成

只记录最后 landing 的内容，不回看整份 plan

接口和类型出现了，就误判语义已经落地

实现者顺手把 plan 标成 completed，没有独立 audit

真实案例

Plan 145

closure/audit 后将新确认的跟进面分离为 Plan 146，避免假关闭

Plan 143

closure assumption 多次被 fresh audit 推翻，直到 2026-04-27 独立复核确认 live repo 全部过线才关闭

收敛力 2：开发日志作为外部化工作记忆

  • 日志最少记录四件事

    改了什么 / 为什么这么改 / 相关代码/文档/计划路径 / 下一步是什么

  • 隐藏价值：保留"为什么不这么做"

    某个收敛尝试回滚了，因为暴露了另一个 regression

  • 记忆缺失会让吸引子变弱

    如果信息不进仓库，后续 AI 容易重走同样的失败路线

日志不是收尾材料，而是系统记忆。

验证体系：自动化引力场与验证顺序

根级质量门

check:react19 / check:src-artifacts / check:oversized-code-files / check:i18n-keys / lint

高信号规则直接升到 error：react-hooks/exhaustive-deps、i18n 硬编码字符串

验证顺序 = 收敛逻辑

1. Focused verification（局部收敛）

2. 修当前切片

3. Workspace 级 typecheck/build/lint/test（全局收敛）

能靠机器提前失败的，就不要留给人类事后理解。

审计闭环：从多 Agent 发散到共识收敛

三阶段闭环

 发散：18 个 subagent 并发

 过滤：重叠/冲突/证据不足分类

 确认：reconciliation → 共识

多 agent 并发 ≠ 独立认知

真正有价值的是认知独立性和最后的 reconciliation

只有经过收敛的结论，才允许进入 owner doc / plan / backlog

高质量审计，不是堆问题，而是尽快排除假问题。

长期收敛力：Debugger、Bug 记忆与完成判定

Bug 记录保存的是排查路径

为什么难定位、先看了什么、排除了哪些假设、最后依据什么确认了问题

nop-debugger 按"AI 可消费"设计

以 live cid 为中心，暴露 automation-facing API

"完成"必须定义得非常苛刻

live code path 真经过
focused/full verification 收口
文档同步
leftover work 归属明确
独立 closure audit

闭环如何实际运转：一个完整周期

1. 定义吸引子

owner doc 固定当前基线

2. AI 扩张

实现代码、展开方案

3-6. 收敛

lint → 切片验证 → 日志 → 全局验证

7-8. 审计 & 关闭

多源审查 → closure audit

9-10. 更新 & 继续

更新吸引子，下一轮扩张

实例：编译优先和包边界吸引子 → AI 大量实现 → 审计发现运行时包仍混有 compiler 责任 → 拆出 flux-compiler 和 flux-action-core → 三层职责成为新基线

这不是流程图，是系统的呼吸方式。

从仓库历史看这个闭环的实际轨迹

03-20 初始吸引子 03-25~04-20 高速扩张+收敛 04-21~04-25 精确吸引子

结构收敛

CompiledSchemaNode 被消除，模板实例化成为真实基线

边界收敛

compiler / action-core / runtime 三层拆分

审计收敛

deep audit → meta-review → unresolved-only recheck

42 天真正了不起的不是产出快，而是在这种速度下，架构越来越清晰。

吸引子为什么能起作用——它解决了 AI 的三个核心偏差

AI 的三个核心偏差

乐观偏差

"看起来做完了"

上下文漂移

"旧前提不断被复用"

局部正确全局漂移

"在错误层做了正确修复"

对应的收敛机制

Exit Criteria + Closure Audit

验证行为闭环

日志 + 文档权威链

恢复正确基线

架构基线 + owner doc

限制错误方向

重点不是让 AI 不犯错，而是让错误不能长期存活。

这套方法的本质：Audit-and-Closure Driven Development

  • 审计-收口驱动开发

    不是传统 waterfall，也不是 AI 自由发挥，而是把实现和完成判定明确拆开

  • 把文档、discussion、plan、AI 实现、audit、closure、logs 组织成持续纠偏闭环

  • 当前主流 AI 擅长在既有知识边界内高效执行

    但受训练分布和偏置影响，很难自己长出真正新的架构观念

不要指望 AI 自己找到正确设计，要先定义一个不会把系统带偏的吸引子。

可复制结构与核心结论

你必须回答的五个问题：

谁定义方向？
冲突时信谁？
怎么发现漂移？
怎么保留记忆？
怎么判定完成？

可复制的不是表面做法，而是在当前主流 AI 条件下，人先定义新吸引子，AI 再围绕它高效收敛。

真正边界：当前主流 AI 擅长收敛执行，不擅长原创架构

AI 最适合承担

实现、展开、补全、验证、整理

AI 最不适合单独承担

核心概念发明、边界重定义、第一性原理层面的架构突破

案例

ActionScope 与 Data Scope 分离并采用词法作用域——这类方案不是让 AI 自由采样平均解能自然长出来的，而必须先由人定义吸引子。

当前主流 AI 最擅长把已有知识执行到极致，但很难自然长出真正新的架构观念。
