从基本假定出发推理agent设计的必然结果

1. 时间平移不变性：AI是状态空间中的流量远超人类的超级扩张力，并随机性很强。人脑不再具有完整蓝图，唯一真相只能由仓库自身来承担。今天在chat窗口与AI沟通，中途退出后，明天再通过一个新的chat窗口与AI沟通，AI对于项目事实的认知必须和昨天一致。

2. 信息的形式无关性：对于AI智能体而言，它可以理解和生成各种不同形式表达的相同的信息。智能体的一个本质性特征就是可以实现各种形式之间的信息等价转换。因为没有必要为了某种形式要求整理成固定格式的spec，必须按照某个文件格式或者结构存放信息等。典型的，对于一个真正的智能体而言，无论是按照特殊的SKILL格式表达，还是简单的在AGENTS.md中通过一个描述表达，都可以实现同样的能力路由功能。

但柔性不等于免费。形式语义等价，形式代价不等。

3. 领域知识的局域性： AI掌握的知识是通用领域的知识，它隐含理解的规律也是通用领域的规律。在特定领域中，存在着特殊的约定、规则，甚至可能是和通用域知识冲突的内容，这些必须通过外挂的信息存储机制来接入。这些特殊的知识很多还具有时效性，也不应该被纳入通用AI模型的权重记忆中，否则长期来说反而是一个误导。大量领域内知识还存在知识产权问题，也不可能纳入通用AI范畴。

4. 计算的不可压缩性： Wolfram提出的原则。 AI执行的每个任务内在的都存在很多逻辑判断和推导，必须在一定的上下文（计算量）下完成，因此如果其中包含的指导信息过多，特别是冲突性的表述，会导致无法充分考虑各种情况，出现遗漏或者加重幻觉。参数量小的模型天生的每次执行计算总量就更小，因此它们必然需要更明确的指示。或者将一个复杂的任务拆解为多个步骤，从而避免单次执行负担过重。

Wolfram 的观察在 LLM 上的具体体现：任何任务的内在推理步数有下界，不能用"请认真思考"压成一步。

这一条相对被理解，但常被低估的是它的边界条件。

很多人知道"复杂任务要拆"，但不知道判断"复杂"的标准是什么。标准不是任务的字数、不是涉及的文件数，是任务内在的逻辑判断点数量。

一个修 bug 的任务，如果同时要求："修 bug + 遵守代码风格 + 写测试 + 更新文档 + 保持向后兼容 + 考虑性能"——这看起来是 6 个要求，实际上是 6 个独立的判断维度。每一个维度在每个代码位置都要 AI 做决策。维度数 × 决策点数 = 总判断量。这个总量超过单次推理算力时，AI 不是"做不好"，是"做不到"——它会在某些维度上完全失守，而且通常以"看起来周到"的方式失守。

工程后果：

第一，任务拆分的颗粒度由判断维度决定，不由任务长度决定。

修 bug 的合理拆法是按维度拆：先只修 bug 验证修了，再调风格，再补测试，再更新文档。每步算力集中在一个维度。

第二，参数量小的模型不是"差的大模型"，是计算容量更小的执行器。

给 Claude Opus 一段含糊描述它能跑得动；给 Haiku 必须明确分步指令。这不是 Haiku"笨"，是它单次能动用的算力本来就少。选用小模型时，prompt 风格必须自动切换到"指令式"而非"描述式"——把人类该做的分解工作提前做完。

第三，OODA 循环不是工程优雅，是物理必然。

Agent 的 observe-plan-act-verify 循环结构常被当成"好的设计模式"。其实它是计算预算约束下的强制形态——单次推理装不下"完整任务"，必须分步，分步必须循环，循环必须有反馈。任何宣称"一步搞定"的 Agent 设计都在违反这一条。

5. AI 的失败有方向性，不是无偏的
   人们处理 AI 错误时常用统计直觉——"它有 X% 概率错"，所以"多采样几次取一致"。这套对付随机噪声的方法，对付不了 AI 的真实失败模式。

AI 的输出不是无偏的随机扰动，是系统性向训练分布的平均解漂移。它会朝主流方案漂、朝看起来合理的常见模式漂、朝它见过最多次的写法漂。这个漂移有方向。

工程后果：

第一，"多采样取一致"对付不了偏置。

如果偏置方向相同，10 次采样会一致地错向同一个方向。多采样能消除随机扰动（C4a 类问题），消除不了系统偏置（C4b 类问题）。后者只能靠外部锚点——人类给出的方向、可执行的约束、与训练分布不一致的明确指令。

第二，"让 AI 创新"通常失败。

你让 AI 设计一个新架构，它给出的是它训练数据里最常见架构的重组。你让它解决一个非主流问题，它把它扭成它见过的主流问题。这不是 AI"不够聪明"，是它的输出分布就是这样。

这条对工程的指导：真正新的架构方向不能依赖 AI 自发提出。AI 适合在已定义的方向上高速展开，不适合做方向定义本身。这把"人类该干什么"明确了：人类的稀缺价值不在写代码，在定义代码应该长成什么样

第三，验证机制必须用 AI 看不到的视角。

如果验证 Agent 和实现 Agent 用同一个模型、同一种 prompt 风格、同一份上下文，它们会有同向偏置——实现错了，验证也认为对。有效验证必须引入异质性：不同模型、独立 session（不共享上下文）、可执行检查（lint / test，根本不受 LLM 偏置约束）、或者人类。

一个复杂任务里面包含：

需求理解；
事实检索；
冲突消解；
计划拆解；
代码阅读；
方案比较；
风险评估；
执行；
验证；
修复；
总结。
这些步骤不是靠一句“请认真思考”就能压缩掉的。

所以小模型如果上下文少、推理预算低，就需要：

更明确的指令；
更结构化的任务；
更小的步骤；
更多工具辅助；
更多外部验证；
更强的约束格式。
大模型也不能无限压缩复杂任务。上下文塞太多信息，尤其是冲突信息，会导致：

注意力稀释；
优先级混乱；
幻觉；
遗漏；
错误引用；
计划不稳定；
看似合理但不可执行的答案。
所以我同意你说的：

复杂任务必须拆解，而不是试图把所有指令都塞进一次 prompt。

但我会补充一点：

计算不可压缩不等于信息不可组织。

也就是说，推理过程本身可能不可省略，但我们可以通过更好的表示来降低无效计算：

用索引减少检索成本；
用 schema 减少歧义；
用测试减少验证成本；
用类型减少状态空间；
用计划减少搜索空间；
用缓存避免重复计算；
用工具外包精确计算；
用子任务隔离上下文污染。
所以更完整的原则是：

推理不可任意压缩，但任务空间可以通过表示、工具和分解来重构。

Agent 是一个在有限计算下，面对重复但变化的任务分布，通过外部状态、工具调用、验证反馈和计算摊销结构来最小化长期任务成本的系统。

原理 P2:知识局域性
推导:P1 + C1
P1 要求事实跨时间一致。C1 告诉我们 LLM 无状态。两者结合直接强制:事实必须存在于模型权重之外的某个权威载体中——这就是仓库即真相。

但还不够。事实本身会变。如果把"快变量"烧进模型权重,模型版本一更新事实就跳变,违反 P1;多个 Agent 跑在不同模型版本上事实就分裂,违反 P1。

通用模型权重只承载慢变量(语言、推理、常识);快变量(领域规则、时效事实、私有约定)必须在权重之外。

凡是会变的,必须可被替换;凡是必须可被替换的,就不能烧进权重。

更进一步:外置知识之间必须有权威优先级。当不同来源冲突时,如果没有明确的"谁说了算",AI 会把它们平均成一个错误现实。因此外置知识不是平铺的,而是分层的——架构基线 owner doc 高于分析文档,分析文档高于历史记录,历史记录高于临时讨论。

原理 P3:Skill 的必然性
推导:C2 + C6 + P1
C2 说计算不可压缩——已走过的路径其计算无法被压缩,但其结果可以被复用。C6 告诉我们问题分布有重复模式,所以复用是有意义的。两者联立:

Skill 是过去计算所探索出的局部最优路径的固化。它的存在不是为了"组织代码",而是为了把不可压缩的计算成本一次性付清,避免每次推理重复支付。

Skill 之于 Agent,相当于缓存之于 CPU、肌肉记忆之于人脑——都是 C2 与 C6 共同作用下的必然产物。一个没有 Skill 机制的 Agent 等于每次都让 CPU 不带缓存跑。

Skill 同时必须服从 P1:环境变了,旧路径会失效,Skill 本身也是快变量,必须外置、必须可版本化、必须可被淘汰。

Skill 的来源不止人工编写。在长期运行的 Agent 系统中,高信号的审计发现、反复出现的诊断路径、被多次重新发明的解法,都应当被识别并下沉为 Skill。这是把扩张过程中的偶发智慧固化为系统能力的标准路径。

原理 P4:柔性接口
推导:C3 + C2
传统软件接口是刚性的,因为传统软件没有 C3——它只能逐字节匹配。整个传统软件工程的复杂性,相当大一部分来自于维护这种刚性。

Agent 有 C3。一段能力描述写成 SKILL 格式、AGENTS.md 中的自然语言、还是 README 的一节,对 Agent 都是等价的。

承载相同语义的任意形式表达对 Agent 能力等价。接口因此承载语义而非锁定形式。

但柔性不等于免费。C3 给了能力,C2 限制了预算。结构化形式更省 token、触发更稳定;自然语言形式更灵活、更易演化。形式选择是 C2 约束下的代价权衡。

原理 P5:元认知必然性
推导:C4 + P1
P1 定义中已经包含"置信状态"作为真相的一维。C4 告诉我们 LLM 输出本质上是概率分布,且有系统性偏置。两者联立:一个只报告"最可能的答案"而不报告置信度的 Agent,隐藏了真相的一个维度,直接违反 P1。

概率性系统必须显式输出其不确定性边界:已知什么、未知什么、信息何时冲突、何时需要裁决。

每个推理步骤应附带置信度或边界标记;
遇到知识冲突时显式暂停并请求裁决,而非自行消解;
Agent 必须有"未知"作为一个一等公民的答案选项,能说"这个问题我无法在当前仓库状态下确定"。
元认知不是工程礼貌,是概率系统的真相完整性要求。

D9 信任工程(最小惊讶)
由 P1 + C4 + C5 推出。当人脑不再持有完整蓝图(C5),又面对概率性的 Agent(C4),信任不会自动产生,必须被设计进去:破坏性操作事前确认或沙盒预览,行为应可被预估,自主性必须与可预测性同步增长。

最强点二：真相包含置信状态
这点也非常重要。

传统软件系统里，真相常常被理解成：

text
数据库里有什么
但 Agent 系统里的“真相”必须包含：

text
我们不知道什么
我们为什么不确定
哪些来源冲突
哪些结论只是猜测
哪些需要人类裁决
因为 LLM 是概率系统，如果它只输出一个最可能答案，就会把不确定性隐藏掉。

隐藏不确定性不是表达不完整，而是真相不完整。

最强点三：局部正确、全局漂移
P6 非常有价值。

很多系统失败不是因为某一步明显错了，而是：

text
每一步都看似合理
每个 PR 都能解释
每次修改都有理由
每个测试都通过
但整体方向已经偏离原目标
这就是“局部正确，整体漂移”。

这类问题靠 step-level confidence 解决不了。

必须有轨迹层机制：

定期 global review；
与 architecture baseline 对照；
与 initial intent 对照；
与 negative space 对照；
独立 session 复核；
对“系统是否正在变成另一个东西”做判断。
这确实是 Agent 长期运行的核心风险。

最强点四：把流程理解成“力”，而不是节点
这句很关键：

Plan、lint、audit、closure 不是流程，而是力。

这能纠正一个常见误区：

text
加更多流程节点 = 更可靠
其实不是。

如果这些节点同源、同上下文、同偏置，它们只是重复同一种错误。

真正有效的是：

text
独立的回拉力
例如：

实现 Agent 和审计 Agent 上下文隔离；
reviewer 不复用 executor 的结论；
audit 直接查 live state；
baseline 独立于当前任务上下文；
人类只裁决关键边界，而不是审核所有 token。
所以不是“检查越多越好”，而是“回拉力是否独立、方向是否正确、作用点是否有效”。

7. “尺度分离原则”

Agent 系统里有不同变化尺度：

层级 变化速度
模型权重 慢
领域知识 中
项目状态 快
当前任务上下文 很快
scratchpad 瞬时
skill 中快
架构原则 慢
生产数据 快
很多错误来自把不同时间尺度的信息放错位置。

例如：

把快变量烧进模型权重；
把临时假设写入长期记忆；
把一次性 workaround 固化成 skill；
把过期注释当真；
把旧 audit 结果变成永久 backlog；
把当前实现细节提升成架构原则。

尺度分离原则：信息应存放在与其变化速度、作用域和权限范围匹配的载体中。

真相必须包含事实、因果、置信和否定空间；否则长期 Agent 必然重复犯错和漂移。
Agent 不是静态模块组合，而是动态控制系统；验证、审计、skill、日志、版本化都是控制轨迹的力。

5. 不是“更严流程”

这套东西不是：

text
多写文档
多写 plan
多做 audit
多设规则
而是：

text
为了让系统轨迹不漂，必须有吸引子、测量、回拉和记忆。
所以 Plan、lint、audit、closure 不是流程节点，而是控制力。

6. “吸引子模型”
   AI 是充满随机性的快速扩张力：

架构基线定义吸引子：

Plan 定义局部收口；
lint/check/test 提供势垒；
logs 保留轨迹；
audit 观测漂移；
closure 防止假关闭；
meta-review 收敛审计噪声。

“AI 大规模开发 = 受控动力系统”

7. 验证本身也会污染系统
   “验证的二阶收敛”，非常重要。

普通 Agent 设计会说：

text
LLM 会错，所以加 verifier。
但是：

text
verifier 如果也是 AI，也会错，也会制造噪声。
这一步很关键。

AI 审计结果不能直接变成 backlog。

这是一个非常实用、非常硬的原则。

否则系统会出现一种很危险的循环：

text
AI 生成问题
AI 审计出更多问题
AI 把假问题写进 backlog
AI 修复假问题
系统复杂度上升
新的审计继续发现更多伪偏离
最后系统会被自己的治理机制拖垮。

所以高质量审计的目标不是“发现更多问题”，而是：

让真正的问题留下来，让伪问题尽快死亡。

8. “Plan 是局部收敛器”
   普通项目里的 plan 是任务清单。

但在 AI 大规模开发里，Plan 的作用变了。

它不是：

text
TODO list
而是：

text
当前基线 + 本轮目标 + 非目标 + 完成条件 + 验证证据 + closure 判定
也就是说，Plan 是一个局部吸引盆地。

它定义：

这一轮扩张允许到哪里；
什么算落地；
什么不属于本轮；
怎么判断完成；
谁来独立确认。
所以我现在认为：

没有 Plan 机制的 Agent，很容易把“生成了很多东西”误判成“完成了一个结果面”。

9. “完成判定”是 AI 工程里的一级问题
   以前很多人把 完成判定 看成流程末端的小问题。

在 AI 开发里，done 是核心问题。

因为 AI 非常擅长制造“完成感”：

文件有了；
类型有了；
测试有了；
文档有了；
checklist 勾了；
summary 写得很流畅。
但这些都不等于行为真的落地。

所以 completed 必须靠证据，而不是靠声明。
完成判定必须从实现上下文中分离出来。

同一个 session 不能既实现又验收。

否则它会倾向于证明自己完成了。

10. 对“人类角色”的认知的变化

AI 大规模开发不是简单地让 AI 替人写代码，而是把人类的角色上移。

人的关键职责变成：

定义吸引子；
定义概念边界；
裁决冲突；
校准收敛标准；
判断哪些问题是真问题；
决定哪些新约束升格为基线；
维护系统方向。
AI 的职责是：

扩张；
实现；
补齐；
检查；
复核；
生成候选；
做局部收敛。

当前 AI 时代，高级架构师的价值不是被削弱，而是从“亲自构造所有细节”转为“设计系统的吸引子和收敛机制”。

项目真相的四个维度
时间平移不变性要求“真相”不仅仅是最终代码或最终状态。对 Agent 而言，项目真相至少包含四个维度。

1. 事实内容
   即项目现在是什么：

代码是什么；
接口是什么；
数据结构是什么；
当前任务状态是什么；
当前架构基线是什么；
当前测试结果是什么。
这是最常见意义上的“事实”。

2. 因果来源
   即为什么会变成这样：

谁做了这个决策；
基于什么约束；
解决了什么问题；
替代了什么旧结构；
哪个 plan、issue、commit、discussion 导致了当前状态。
如果系统只保存最终状态，不保存因果来源，下一个 Agent 就无法判断：

text
这是深思熟虑后的架构基线？
还是一次临时 workaround？
还是过渡阶段的残留？
缺失因果来源，会导致项目认知在时间上漂移。

3. 置信状态
   即我们知道什么、不知道什么、哪些仍然存疑：

已验证事实；
未验证假设；
来源冲突；
推测性结论；
需要人类裁决的问题；
过期但尚未清理的信息。
概率性系统如果只输出“最可能答案”，却隐藏不确定性边界，那么它隐藏了真相的一部分。

因此，置信状态不是附加说明，而是真相本身的一维。

4. 否定空间
   即哪些路已经被排除：

哪些方案被否决；
哪些假设被证伪；
哪些 bug 路径已经查过；
哪些重构方向被证明代价过高；
哪些审计发现已经被判定为非问题；
哪些实现曾经尝试但被回滚。
这是长期 Agent 最容易缺失的一维。

如果否定空间不被保存，后续 Agent 会反复提出已经被否决的方案。表面上看，它是在“探索新方案”；实际上，它是在破坏时间一致性。

所以：

否定空间是项目真相的一部分。

C3：语义可理解性
LLM 与传统软件的根本差异之一，是它能在不同形式之间识别相同语义。

P3：知识局域性与尺度分离原则
信息应存放在与其变化速度、作用域和权限范围匹配的载体中。
这就是知识局域性的本质。
模型权重承载慢变量。
项目状态源承载快变量。
当前上下文只承载瞬时变量。

Skill 是过去不可压缩计算探索出的局部最优路径的固化。

P5：柔性语义接口原则
Agent 能理解不同形式表达的相同语义。

因此，能力描述不必被某一种固定格式垄断。
对 Agent 而言，它们都可能承载同一语义。
但 C2 告诉我们，计算资源有限。不同形式虽然语义可等价，处理成本却不同：

熵增背景 → 解释为什么"维持"是有代价的；
最小作用量 → 解释为什么"复用"是必然的；
信息局域 → 解释为什么"协同"必须经过介质；
动力系统 → 解释为什么"修正"需要吸引子。

Agent 的第一性原理：Agent 是一个在目标约束下，跨时间维持、验证、修正、复用可行动认知结构的系统。

An agent is a system that persistently maintains, verifies, revises, and reuses an actionable cognitive structure under goals.

维持 → 让认知结构跨时间存在
验证 → 判断它是否仍然接地
修正 → 让它在偏离后回到正确轨道
复用 → 让历史计算转化为未来能力

维持 连续性 失忆、时间漂移
验证 接地性 自洽幻觉
修正 适应性 僵化、错误滞留
复用 积累性 重复付费、不成长
行动 有效性 退化为知识库

一、上下文是计算资源，不是信息容器
这是最常被搞错的一条。

大多数人把上下文当成"装信息的桶"——能装多少装多少，能装下就是好。所以他们的 CLAUDE.md / AGENTS.md 越写越长，prompt 越塞越满，"反正模型能读到"。

错。

上下文的本质是计算预算。每个 token 都在消耗注意力分配，每条信息都在和其他信息竞争 AI 的处理能力。一个 50K token 的上下文不是"装了 50K 信息的容器"，是"50K token 之间互相干扰、共同决定一次推理质量的计算场"。

这有两个直接后果。

第一，冲突信息的成本极高，远高于缺失信息。

"用 A 模式，但有时也用 B" 比 "用 A 模式" 贵一个量级——前者强制 AI 在每次决策时分配算力判断"现在算不算那个有时"。指令书写应当追求确定性，而非完整性。宁可说"统一用 A"再在极少例外处单独标注，也不要在通用规则里堆条件分支。

第二，相关性比完备性重要。

把项目所有规则都塞进 system prompt 是反模式。正确的做法是分层：核心约束（不能违反的死线）+ 高频偏好（每次都要用的）进入常驻上下文；其他全部按需加载——SKILL 触发式、owner doc 引用式、历史日志检索式。

这条原则的更深推论：Agent 系统的核心设计问题，是知识加载策略，不是知识总量。一个仓库有多少文档不重要，重要的是 AI 进入任何任务时能在前几百个 token 内拿到决策所需的最小相关集。

判断一个 Agent 系统设计水平的简单标准：看它的 AGENTS.md 多长。超过 1000 行的，基本可以判定团队没理解这条。
