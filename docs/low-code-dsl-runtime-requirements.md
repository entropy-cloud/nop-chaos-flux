# 声明式低代码 DSL 运行时 — 需求说明

> **文档性质**: 实现无关的需求规格说明。描述系统应具备的能力和行为约束，但不指定具体技术栈、框架选择或内部架构设计。
>
> **用途**: 作为基准输入，让不同实现方案在同一份需求描述下进行技术对比。

---

## 1. 项目定位

本系统是一个 **声明式低代码 DSL 运行时**，它接收一份结构化的 JSON/DSL schema 作为输入，在浏览器端将其解释执行为一个可交互的前端应用。

该系统的核心角色是 **schema 引擎**：它不直接生成业务代码，而是将一份描述页面结构、数据流、交互逻辑和校验规则的声明式 schema 转化为运行时行为。schema 是该平台的**一级制品**——它不仅仅是运行时的配置文件，而是一个独立可操作的结构层，在进入运行时之前就可以被编辑、组合、裁剪和变换。

---

## 2. 核心能力需求

### 2.1 Schema 解析与编译

系统必须能够：

1. **接收声明式 schema**：输入为 JSON 或等价结构化格式，描述一棵由节点组成的树。每个节点拥有 `type`（如 page、form、input-text、table、button 等）以及一系列属性字段。

2. **编译 schema 为内部执行模型**：在进入实际执行前，schema 应被编译为一个不可变的、面向执行优化的内部表示。编译阶段负责：
   - 解析节点类型并关联对应的渲染组件
   - 将 schema 值字段分类为静态值、动态表达式、模板字符串、子节点区域等不同类别
   - 提取子区域（region），如页面的 body、表单的 actions、表格的 columns
   - 组装校验规则、事件处理器、数据源声明等
   - 静态部分应保持零运行时开销

3. **支持渐进式值语义**：schema 字段的值应当支持从简单到复杂的渐进表达：
   - 字面量（`"Hello"`）
   - 表达式（`${user.name}`）
   - 模板字符串（`Hello ${user.name}, today is ${date}`）
   - 基于动作的值生产者（异步获取数据）
   - 命名数据源（带生命周期管理的持续值发布）

### 2.2 表达式引擎

系统必须提供表达式求值能力：

1. **变量解析**：表达式通过路径（如 `user.name`、`items.length`）访问当前数据环境中的值
2. **运算与函数**：支持算术、比较、逻辑运算及内置函数/过滤器调用
3. **统一求值上下文**：所有表达式在同一个上下文模型中求值，通过 `resolve(path)` 和 `has(path)` 接口访问变量
4. **编译一次，多次执行**：表达式应在首次使用时编译，后续执行复用编译结果
5. **禁止动态代码生成**：不允许使用 `new Function`、`eval`、`with(scope)` 等动态代码执行机制

### 2.3 词法数据环境（Scope）

系统必须支持分层的数据环境：

1. **词法继承**：子数据环境默认继承父环境的所有可见数据，同名字段在子级中遮蔽父级
2. **显式隔离**：支持将子环境标记为隔离（不继承父级），用于高频重复场景的性能优化
3. **数据初始化**：每个容器节点（page、form、dialog 等）可以声明初始数据，作为该节点 own 数据环境的起始补丁
4. **定点写入**：支持通过路径精确写入数据环境的某个位置，而非总是整体替换
5. **变更传播**：数据变更应向订阅者通知具体变更了哪些路径，而非仅通知"发生了变化"

### 2.4 依赖追踪与响应式更新

系统必须实现精确的依赖追踪机制：

1. **隐式依赖收集**：当表达式求值读取了某个数据路径时，该读取应被自动记录为依赖，无需显式声明
2. **精确失效**：当数据变更时，只有实际依赖了变更数据的那部分计算才被标记为需要重新求值
3. **三类消费者共享同一依赖模型**，但变更后果不同：
   - **值的读取**：依赖变更 → 重新计算该值
   - **命名数据源**：依赖变更 → 按策略刷新（重新请求/重新计算）
   - **副作用观察者**：依赖变更 → 重新评估，可能触发动作
4. **自写保护**：命名数据源写入的数据变更不应触发该数据源自身的刷新循环
5. **读写分离**：所有对数据的读取通过数据环境进行；所有可见的副作用（状态写入、API 调用、导航等）只通过统一的能力派发通道进行

### 2.5 渲染与组件系统

系统必须将编译后的 schema 树渲染为用户界面：

1. **组件注册与解析**：schema 中的每个 `type` 对应一个注册的渲染组件。组件注册表支持在运行前或运行时扩展
2. **Props 与 Meta 分离**：每个节点在运行时被解析为：
   - **业务属性（props）**：组件的直接输入，如 label、placeholder、options
   - **控制元数据（meta）**：运行时控制信息，如 visible、disabled、className
   - **子区域（regions）**：预编译的子节点渲染句柄，如 body、header、actions
   - **事件处理器（events）**：声明式事件到运行时动作的映射
3. **容器与叶子分离**：
   - **布局类组件**（page、container、flex）仅输出语义标记类名，不包含内置视觉样式，样式完全由 schema 驱动
   - **控件类组件**（input、table、select、code-editor）是完整的、自包含的 UI 控件，内置视觉样式
4. **片段渲染**：支持将一段 schema 渲染为独立片段，可以注入局部数据覆盖、创建子数据环境
5. **参数化区域**：子区域支持声明参数名（如 `item`、`index`），渲染时通过绑定传入，在表达式中通过 `$slot` 访问

### 2.6 动作系统与控制流

系统必须支持声明式的交互逻辑编排：

1. **单步派发**：最基本的动作形式是 `{ action, args }`，直接派发一个命名动作
2. **条件守卫**：`when` 表达式决定动作是否执行，跳过时返回 `skipped` 结果
3. **成功续接**：`then` 在动作成功后执行后续动作，可读取前一步的结果
4. **失败续接**：`onError` 在动作失败时执行错误处理
5. **并发聚合**：`parallel` 同时派发多个子动作，收集全部结果后合并为聚合结果
6. **重试与超时**：支持配置重试策略和超时
7. **防抖**：支持配置动作防抖
8. **结果分类**：动作执行结果分为三类——成功、失败、跳过（中性），控制流分支基于此分类
9. **链式结果上下文**：`then`/`onError` 中的表达式可以访问 `result`（前一步结果）、`error`（错误信息）、`prevResult`（更早的结果）

### 2.7 动作能力的三层解析

动作派发应支持三种解析路径：

1. **平台内置动作**：如 `setValue`（设置数据）、`ajax`（发起请求）、`dialog`（打开对话框）、`submitForm`（提交表单）、`closeDialog`（关闭对话框）
2. **组件实例动作**：`component:<method>` 格式，通过组件实例注册表查找并调用特定组件实例的方法（如表格的刷新、获取选中行）
3. **命名空间动作**：`namespace:method` 格式，通过词法作用域的命名空间注册表查找（如 `designer:export`），用于复杂宿主控件的能力暴露

### 2.8 表单与校验

系统必须支持表单场景：

1. **表单运行时**：表单节点创建独立的数据环境和表单运行时，管理表单值、脏状态、已访问状态、提交状态
2. **声明式校验规则**：校验规则在 schema 中声明，编译时提取为校验图，包括：
   - 字段级规则（required、minLength、pattern 等）
   - 对象级规则（跨字段联合校验）
   - 数组级规则（数组长度等）
   - 条件规则（根据其他字段值决定是否生效）
3. **校验时机**：支持多种触发时机（提交时、值变更时、失焦时），UI 显示策略与规则执行分离
4. **异步校验**：支持异步校验规则，可取消
5. **部分校验**：支持按路径或子树进行局部校验
6. **草稿隔离**：支持将子表单或编辑区域标记为草稿模式，其校验状态和脏状态独立于父级，直到显式提交

### 2.9 API 请求与数据源

系统必须支持声明式的远程数据获取：

1. **声明式 API 定义**：schema 中声明请求 URL、方法、参数、请求体、响应适配器
2. **作用域注入**：可配置将当前数据环境的变量自动注入到请求参数中
3. **命名数据源**：支持声明命名的数据生产者，具备：
   - 值发布：将获取到的数据发布到数据环境
   - 刷新策略：支持手动刷新、定时轮询、依赖变更自动刷新
   - 生命周期：随所在数据环境的挂载/卸载自动管理
   - 加载状态：暴露 loading/error 状态到数据环境
4. **副作用观察者（Reaction）**：支持声明数据变化观察者，当被观察的值变化且满足条件时，派发指定动作

### 2.10 表面对话系统

系统必须支持弹出层的统一管理：

1. **统一表面模型**：对话框和抽屉共享同一套表面管理模型，通过稳定类型标记（dialog/drawer）区分
2. **独立数据环境**：每个打开的表面拥有独立的数据环境，不复用页面数据环境
3. **栈式管理**：多个表面可以叠加，后打开的在上层，只有最顶层表面拥有焦点和键盘事件处理
4. **关闭恢复**：关闭顶层表面后，前一个表面恢复为活动状态

### 2.11 表格与集合渲染

系统必须支持高性能的集合数据展示：

1. **行级数据环境**：表格为每行创建独立的数据环境，包含 `record`、`index` 等行级变量
2. **行级隔离**：行数据环境默认隔离，不继承父级数据环境，以保证高频渲染性能
3. **行级依赖隔离**：修改一行的数据不应导致其他行重新渲染
4. **显式行数据投影**：隔离的行环境若需要少量外部数据，应通过显式投影传入，而非打开任意父级穿透

### 2.12 循环与递归结构

系统必须支持：

1. **循环结构展开**：`loop` 节点根据集合数据动态展开子节点，每个迭代项拥有自己的数据环境，包含 `item`、`index` 等变量
2. **递归渲染**：支持节点引用自身或祖先的结构定义进行递归渲染
3. **编译一次，实例化多次**：循环/递归的子树结构只编译一次，运行时为每个迭代项创建独立的实例状态

## 3. 宿主集成需求

### 3.1 宿主环境边界

系统必须通过明确的边界与宿主应用集成：

1. **数据注入**：宿主通过显式参数注入初始数据和运行时环境
2. **网络请求委托**：实际的 HTTP 请求由宿主通过回调接口提供，运行时本身不直接发起请求
3. **通知委托**：toast/notification 等全局通知由宿主提供
4. **导航委托**：页面导航由宿主提供
5. **环境稳定性**：运行时不应因为宿主传入的环境对象引用变化而重建内部状态

### 3.2 复杂域控嵌入

系统必须支持嵌入复杂的域专用编辑器（如流程设计器、电子表格、报表设计器、文档编辑器）：

1. **只读快照投影**：域控件将自身的只读状态快照投影到 schema 可见的数据环境中
2. **命名空间命令**：域控件通过命名空间注册自己的能力方法（如 `designer:export`），schema 通过统一的动作派发通道调用
3. **域私有通道**：域控件的内部状态机、协议、协作引擎等不进入 schema 可见的数据环境
4. **静态契约声明**：域控件可以声明自己暴露的投影字段和能力方法的类型信息，供编译期校验

### 3.3 主题与样式兼容

1. **无内置主题系统**：运行时不提供 ThemeProvider 或运行时主题切换机制，通过 CSS 变量与宿主主题系统对接
2. **标记类名约定**：布局类组件输出稳定的语义标记类名，宿主通过这些类名在外部定义样式
3. **Schema 驱动样式**：组件的 className、间距等视觉属性由 schema 声明，运行时仅透传

## 4. 跨领域约束

### 4.1 安全约束

1. 禁止运行时执行权限判断——权限裁剪应在 schema 进入运行时之前完成
2. 禁止动态代码生成（`new Function`、`eval`、`with`）
3. 表达式执行必须通过受控的编译/求值路径
4. 命名空间动作边界必须显式，不允许绕过作用域解析的全局注册

### 4.2 性能约束

1. **静态零开销**：不含表达式的 schema 部分在运行时不应引入额外计算开销
2. **引用复用**：动态求值结果未发生变化时，应复用上一次的引用
3. **选择器式订阅**：只关心部分数据的组件不应因无关数据变更而重新执行
4. **编译一次执行多次**：schema 编译结果应为不可变模板，运行时仅实例化
5. **高频子树隔离**：表格行等高频重复场景必须有机制避免无关变更导致的扇出刷新

### 4.3 渐进式复杂性

系统应支持从简单到复杂的渐进式使用方式：

| 关注点 | 简单形式 | → | 复杂形式 |
|--------|---------|---|---------|
| 值生产 | 字面量 → 表达式 → 动作型值生产者 | → | 命名数据源（带生命周期） |
| 动作编排 | 单步派发 | → | 条件守卫 → 成功/失败续接 → 并发聚合 |
| 结构控制 | 显隐控制 | → | 条件激活/生命周期参与 → 集合展开 → 远程片段动态加载 |
| 数据写入 | 语义命令 | → | 结构化补丁写入 |

### 4.4 编写态与执行态分离

1. Schema 在进入运行时之前应经过编译/组装阶段
2. 编译阶段的优化目标是可理解性、编辑保真、结构组合
3. 执行阶段的优化目标是性能、概念简化、开销最小化
4. 如果一个问题能在编译阶段解决，就不应推迟到运行时

---

## 5. 国际化需求

1. 支持多语言（至少中文 zh-CN 和英文 en-US）
2. 所有国际化键使用统一前缀
3. 国际化字符串替换在编译阶段完成，不影响运行时行为

---

## 6. 开发工具支持需求

系统应支持调试和开发工具集成：

1. **节点标识**：每个运行时节点拥有唯一标识，可通过 DOM 属性关联到编译时 schema 位置
2. **运行时检查**：支持检查当前数据环境快照、节点属性、校验状态等
3. **诊断信息**：编译期可以产出诊断信息（类型不匹配、无效引用等）
4. **错误处理回调**：宿主可以注册全局错误处理回调

---

## 7. 质量属性

1. **可测试性**：核心逻辑不依赖特定 UI 框架，可以在无 DOM 环境下测试
2. **可嵌入性**：可以作为子组件嵌入到任意外部应用中，不污染全局状态
3. **可扩展性**：新的组件类型、动作类型、域控件可以通过注册机制接入，无需修改核心
4. **类型安全**：所有公开接口拥有类型定义，表达式和 schema 结构可以在编译期进行静态校验

---

## 8. 不属于本系统的范围

以下能力虽然对最终产品重要，但不属于本 DSL 运行时的核心职责：

- Schema 编辑器/设计器的具体实现
- Schema 的继承、覆写、权限裁剪等编写态结构操作
- 协作编辑协议（CRDT/OT）
- 具体的图表渲染引擎
- 具体的电子表格引擎
- 具体的流程图引擎
- 具体的文档编辑器引擎
- 后端 API 服务
- 用户认证与权限管理系统
- 工作台 Shell 和会话管理
