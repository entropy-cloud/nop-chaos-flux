nop-chaos-flux 是什么

  • 一个 Schema 驱动的前端运行时和渲染框架

    "Schema that executes, not just renders"——Schema 不仅描述 UI，还参与执行

  • 为 NOP 低代码体系提供下一代前端运行时基础设施

    不是应用框架，不是 UI 组件库

  • 围绕 七个原语构建：Template、ScopeRef、Value、Resource、Reaction、Capability、Host Projection

    表单、表格、对话框、设计器、验证等都是这七个原语之上的派生系统

  • 在 NOP 体系中的位置：nop-chaos 过去基于 AMIS，Flux 是下一代独立重写

核心问题：AI 生成是扩张力，没有收敛机制就会持续漂移

高带宽生成

AI 能在极短时间内产出大量代码、测试、文档

乐观偏差

AI 倾向于流畅地叙述"已经完成"，即使行为闭环并未真正建立

上下文漂移

旧前提在长上下文里不断被复用，整组文件一起偏离真实基线

最危险的地方：不是"写错一段代码"，而是局部都像对的，但整体在持续偏离。

仓库实例：Plan 76 尝试移除本地状态镜像，结果打出 11 个测试失败——测试也绑定了旧实现时序。

局部都像对的，往往正是系统正在整体跑偏的时候。

我们找到的答案：受控收敛的动力系统

没有吸引子，AI 不是在开发系统，而是在高效采样错误解空间。

关键前提：在当前主流模型的训练分布和偏置下，新吸引子通常不会由 AI 自己自然长出来。真正新的概念切分、边界定义和结构语言，仍然需要人先给出。

什么是动力系统，什么是吸引子

动力系统

状态随时间演化、且下一步依赖当前状态的系统。我们关心的是轨迹——从哪里出发，往哪里演化。

吸引子

状态空间中的一个稳定区域——从足够近的地方出发的轨迹，最终会被它吸引并长期停留在它附近。

对 AI 开发的映射

  • 状态空间 = 所有合法实现状态的集合

  • 吸引子 = 架构基线允许的结构空间

  • 轨迹 = 每一轮 AI 生成 + 收敛的演化路径

混沌不等于随机；局部不可预测，不等于整体失控。

仓库真相带来的工程后果：生成和验收被真正拆开

传统开发

仓库 = 认知的外化

作者认知持有系统模型
仓库主要是表达媒介

系统状态的主要载体转移

AI 深度参与后

仓库 = 系统真相的载体

代码 / diff / logs / tests / docs / audit
人：观察 · 约束 · 校正

对应关系：

test / lint / audit = 测量
owner doc / plan / closure = 约束
logs / history / bugs = 轨迹记录

案例：CompiledSchemaNode 被清除——仓库持续排除已不符合新基线的旧结构。

案例：Plan 76 回滚——仓库保留了失败尝试和失败原因。

当作者不再持有完整模型，生成和验收就必须依赖仓库里的外部证据重新接起来。

这个模型和传统开发的关键区别

瀑布

强蓝图 / 长流水线

  • 先设计，再实现，再测试
  • 蓝图优先，阶段分明
  • 默认人类可同步把关

敏捷

弱蓝图 / 短流水线

  • 迭代设计，快速反馈
  • 设计与实现交替推进
  • 默认团队仍能持续跟上

AI 深度参与

动力系统 / 轨迹控制

  • 生成速度超出人脑自然收敛
  • 生成与验收被拆开
  • 重点：系统轨迹控制与受控收敛

按图施工 → 迭代施工 → 控制系统轨迹

再用流水线思维管理 AI 高速生成，只会把混乱组织得更整齐。

吸引子的工程实现 1：架构基线与文档权威链

  • 吸引子 = 可校验的架构约束

    七个原语闭合集、编译优先、数据/能力正交、统一渲染器合约、严格包边界

  • 固定在 docs/architecture/ 的 owner doc 中，决定每次扩张的合法方向

文档权威链

导航入口

docs/analysis/

候选发现

docs/plans/

执行收口

开发记忆

案例：CompiledSchemaNode 被移除，因为它已经不属于模板/实例分离的新基线。

架构的价值不在于描述一切，而在于让错误结构无法合法存在。

吸引子的工程实现 2：保留澄清过程

  • 专门保留 docs/discussions/

    不只写最终结论，也保留澄清过程：原始表达 → AI 复述与提问 → 用户补充 → 纠正记录 → 最终总结

  • 后续 AI 不用凭空猜"为什么这样设计"，也能看到"为什么不能那样设计"

  • 对复杂系统来说，早期理解偏差往往比编码错误更致命

如果只保留最后结论，后来的 AI 看不到哪些前提曾经被证伪过，哪些术语是为了消歧才改的——吸引子就会弱化。

扩张力的角色定义：AI 负责展开，不负责最终裁决

AI 负责

  • 代码生成

  • 文档整理

  • 方案展开

  • 审计候选生成

不交给 AI 自主决定

  • 架构裁决

  • 概念发明

  • 边界重定义

  • 方案淘汰

核心原则：不让同一个上下文既做实现又做收敛裁判。用多个独立 session 交叉验证，人校准收敛标准。

收敛力 1：Plan 从"路线图"变成"收口机制"

  • Plan 不是路线图，而是局部收敛器

    定义吸引子（baseline + goals）→ AI 扩张 → 收敛验证 → 关闭

  • 每个 Plan 至少写清

    Current Baseline
    Goals / Non-Goals
    Exit Criteria
    Validation Checklist
    Closure Audit Evidence

  • 硬规则：写计划前先核对 live repo，不沿用旧计划和旧 completion note

Plan 不是路线图，它是局部收敛器。

Plan 作为收敛力的实际效果

漂移模式（Plan 机制之前）

  • 计划过宽，没人知道什么时候算完成

  • 只记录最后 landing 的内容，不回看整份 plan

  • 接口和类型出现了，就误判语义已经落地

  • 实现者自己把 plan 标成 completed

制度化的解决

  • scope 太宽 → 拆 successor plan

  • 接口存在 ≠ 行为落地 → closure audit 区分

  • completed 必须来自独立 closure audit

真实案例：Plan 145 在 closure/audit 后将新跟进面分离为 Plan 146——不同结果面混在一起只会制造假关闭。

真实案例：Plan 143 的 closure assumption 被独立审计连续推翻，直到 2026-04-27 live repo 全部过线才标记 completed。

收敛力 2：开发日志作为外部化工作记忆

日志最少记录四件事

改了什么

为什么这么改

相关代码/文档/计划路径

下一步是什么

隐藏价值：保留"为什么不这么做"

某个收敛尝试回滚了，因为暴露了另一个 runtime regression
某个架构方向先只收敛文档语义，不强推 runtime 重写
某个能力先作为 docs-only alignment 落地，不扩大 scope

如果这些信息不进仓库，后续 AI 容易重走同样的失败路线——记忆缺失会让吸引子变弱。

日志不是收尾材料，而是系统记忆。

验证体系：自动化引力场

  • 验证体系 = 自动化、持续的收敛力

  • 根级质量门

    check:react19 — 阻断旧 API
    check:src-artifacts — 阻断构建产物污染
    check:oversized-code-files — 阻断超大文件
    check:i18n-keys — 阻断硬编码字符串
    lint — 包括 react-hooks/exhaustive-deps、react-compiler 等 error 级规则

  • 验证顺序也是收敛逻辑：先 focused verification → 修当前切片 → workspace 级 typecheck/build/lint/test

能靠机器提前失败的，就不要留给人类事后理解。

审计闭环：从多 Agent 发散到共识收敛

1. 发散

18 个 subagent 并发发散
覆盖不同维度

2. 过滤

meta-review
重叠/冲突/证据不足分类
删除伪问题

3. 确认

re-verification
核对 live repo
只保留仍成立的问题

实际数据：2026-04-17 全量审计 28 项初审判定中，经独立复核后 11 项被归为非问题，假阳性率约 39%。

  • 审计终态只允许两种："已解决" 或 "明确无需行动"。禁止"延后""待跟进"等模糊终态。

  • 高信号审计发现会逐步下沉为 lint 规则、check 脚本或测试守卫——把人工收敛力转化为自动化收敛力。

高质量审计，不是堆问题，而是尽快排除假问题。

长期收敛力：Debugger、Bug 记忆与完成判定

Bug 记忆

保存的不是 diff，而是排查路径：为什么难定位、先看了什么、排除了哪些假设、最后依据什么确认了问题

nop-debugger

按"AI 可消费"设计：以 live cid 为中心，暴露 automation-facing API
inspectByCid() / getSnapshot() / explainNodeValue()

完成判定

live code path 真经过 • focused/full verification 收口 • 文档同步 • leftover work 归属明确 • 独立 closure audit

completed 不是"我写完了"，而是"这一轮工作确实已经收住了"。

闭环如何实际运转：一个完整周期

完整周期（10 步）

① 定义吸引子 → ② AI 扩张 → ③ 实时收敛(lint/check) → ④ 切片收敛 → ⑤ 记忆沉淀(logs) → ⑥ 全局收敛(full repo verification) → ⑦ 审计收敛 → ⑧ 关闭判定(closure audit) → ⑨ 吸引子更新 → ⑩ 下一轮

典型实例

编译优先+包边界吸引子 → AI 大量实现编译/动作/运行时逻辑 → 审计发现运行时包仍混有 compiler 责任 → 最终拆出 flux-compiler 和 flux-action-core → 三层职责固定为新基线

这不是流程图，是系统的呼吸方式。

这套方法的本质：Audit-and-Closure Driven Development

Audit-and-Closure Driven Development（审计-收口驱动开发）

  • 不是传统 waterfall，也不是 AI 自由发挥

    把实现和完成判定明确拆开：人先定义吸引子，再让扩张和收敛交替发生

  • 把文档、discussion、plan、AI 实现、audit、closure、logs 组织成一个持续纠偏闭环

  • 承认一个事实：当前主流 AI 擅长在既有知识边界内高效执行，但受训练分布和偏置影响，很难自己长出真正新的架构观念

不要指望 AI 自己找到正确设计，要先定义一个不会把系统带偏的吸引子。

可复制结构与核心结论

你不一定要照搬这个仓库的具体做法，但你很可能必须回答五个问题：

1

谁定义方向？

2

冲突时信谁？

3

怎么发现漂移？

4

怎么保留记忆？

5

怎么判定完成？

这五个问题都有答案，你就有了闭环；没有答案，系统就很容易在高速推进中偏掉。

可复制的不是表面做法，而是在当前主流 AI 条件下，人先定义新吸引子，AI 再围绕它高效收敛。

真正边界：当前主流 AI 擅长收敛执行，不擅长原创架构

AI 最适合承担

实现 · 展开 · 补全 · 验证 · 整理

在既有知识边界内做高质量执行

最不适合单独承担

核心概念发明 · 边界重定义 · 第一性原理层面的架构突破

在当前训练分布下，AI 会朝平均情况收敛

案例：ActionScope 与 Data Scope 分离并采用词法作用域——这类方案不是让 AI 自由采样平均解能自然长出来的，必须先由人定义吸引子。

当前主流 AI 最擅长把已有知识执行到极致，但很难自然长出真正新的架构观念。
