# nop-chaos-flux UI/UX 设计合规性审查提示词

> **定位**: 本文档是一份面向 AI 的多轮递归式 UI/UX 设计合规性审查提示词。目标是发现渲染器组件中不符合行业常见设计模式、视觉不一致、交互缺陷、可访问性不足等问题。
> **执行方式**: 多轮递归发现，每轮在上一轮基础上扩展搜索范围，直到收敛。
> **输出格式**: 每个发现按统一格式输出（见附录 A）。

---

## 审查定位

本审查不同于 `deep-audit-prompts.md` 的维度 09（渲染器契约）和维度 10（样式系统合规性）。那些维度关注的是**代码架构契约**（RendererComponentProps 遵循、marker class 纯标记、BEM 残留等）。

本审查关注的是**用户可见的视觉和交互质量**——用户看到什么、点击什么、感受到什么。即使代码在架构层面完全合规，也可能在视觉和交互层面存在设计缺陷。

### 本审查覆盖的范围

| 类别          | 示例                                                                  |
| ------------- | --------------------------------------------------------------------- |
| 图标语义      | 同一操作使用不同图标；图标选择不符合行业惯例（如用 chevron 代替漏斗） |
| 按钮一致性    | 同一语义操作在不同组件中使用不同 variant/size/颜色                    |
| 状态指示      | 排序方向无区分、选中状态不明显、禁用状态不可感知                      |
| 表单交互      | 标签不可点击、缺少 indeterminate 状态、radio 用 checkbox 模拟         |
| 视觉反馈      | 可聚焦元素无 focus ring、loading 无 spinner、空状态无内容             |
| 对话框/弹出层 | 缺少关闭按钮、trigger 文案误导、不可见原生控件                        |
| 颜色使用      | 硬编码颜色而非设计令牌、颜色语义不当（红色用于非破坏性操作）          |
| 间距对齐      | 删除按钮浮动而非对齐、装饰元素重叠、未使用 ml-auto 靠右               |
| 无障碍可见性  | role 语义错误、缺少 aria-label、图表使用 role="button"                |
| 跨组件一致性  | 分页 UI 不统一、同功能不同交互模式                                    |

### 本审查不覆盖的范围

以下由 deep-audit 已有维度覆盖，不要重复报告：

- 渲染器是否遵循 RendererComponentProps 契约（维度 09）
- marker class 是否带视觉样式、BEM 残留（维度 10）
- 是否用了原生 HTML 而非 @nop-chaos/ui（维度 11）
- field metadata / value-or-region 建模（维度 12）
- 可访问性的全面 WCAG 合规（维度 20）— 但如果某个可访问性问题同时构成了**明显的用户交互障碍**（如不可见的原生 select 导致用户无法操作），属于本审查范围

## 行业设计基准

审查时以以下行业基准为参照。不需要在发现中引用具体来源，但判断"是否符合常规"时应以这些系统的实际做法为准：

| 交互场景       | 行业标准做法                                                                                           |
| -------------- | ------------------------------------------------------------------------------------------------------ |
| 可排序列头     | asc = `ArrowUp`（高亮），desc = `ArrowDown`（高亮），未排序 = `ArrowUpDown`（极淡），不可排序 = 无图标 |
| 删除操作       | 行级删除 = ghost + Trash2Icon + hover:text-destructive；批量/危险删除 = destructive variant + 确认     |
| 新增操作       | ghost + PlusIcon 或 outline 按钮，统一风格                                                             |
| 表格全选       | 部分选中 = indeterminate 状态（短划线）                                                                |
| 单选行         | 使用 radio 语义，非圆形 checkbox                                                                       |
| 表格筛选       | 漏斗图标（ListFilterIcon），非 chevron                                                                 |
| Loading 状态   | Spinner 组件 + 可选文本，非纯文本                                                                      |
| 空状态         | 有意义的提示内容（"暂无数据" / 图标），非空白                                                          |
| 对话框         | 右上角可见关闭按钮（X），不隐藏                                                                        |
| 表单标签       | 点击标签切换控件状态（Label + htmlFor）                                                                |
| 可聚焦交互元素 | focus-visible ring，非仅 hover 变色                                                                    |
| 颜色语义       | 硬编码色值应改为 CSS 自定义属性（设计令牌）                                                            |
| 分页           | 同一 CRUD 系统内分页 UI 保持一致                                                                       |
| 图表           | role="img"，非 role="button"                                                                           |
| 按钮文案       | 无 label 时 fallback 为有意义的文本或 aria-label，非字面 "Button"                                      |

## 多轮递归执行模型

### 执行方式

本审查采用**开放式多轮递归发现**模型：

#### 阶段一：迭代发现（多轮）

1. **第 1 轮（初扫）**：按"检查视角"（见下文）全面扫描所有渲染器组件，产出的发现列表保存到 `docs/analysis/{日期}-ux-audit/round-01.md`。
2. **第 2 轮（递归扩展）**：读取第 1 轮全部发现，基于已有发现暴露的模式、组件、关联路径，继续深挖：
   - 已有发现涉及的组件是否有同类型问题未被检出？
   - 已有发现的模式是否在其他组件中也存在？
   - 第 1 轮扫描中是否有尚未覆盖的组件？
   - 是否有跨组件交互场景（如"表格 + 筛选 + 排序组合使用时"）未被检查？
3. **第 N 轮（继续递归）**：重复第 2 轮逻辑，以第 N-1 轮发现为输入继续扩展。
4. **终止条件**：
   - 某轮输出"未发现新的问题"或新发现数量为 0
   - 或新发现仅为低价值、机械重复、缺少明确用户影响的问题
   - 上限保护：每轮审查最多 **10 轮**
5. **价值收敛条件**：递归不是为了凑数量。如果已有发现已经覆盖了所有主要交互模式，继续搜索只能得到极低价值的问题，应输出"未发现新的高价值问题。审查结束。"

#### 阶段二：复核

深挖阶段结束后，进入复核阶段：

1. **维度复核**：派发一个独立子 agent，读取全部轮次的完整发现文件，重新对照 live code 逐条核对，输出"保留 / 降级 / 驳回"的逐条判断。
2. **子项复核（分层）**：对高风险或存在争议的发现项逐项复核；低风险项可按组件或按模式批量复核。

#### 核心要求

- 复核 agent 不能复用发现 agent 的结论当作事实，只能把发现结果当作"待验证线索"。
- 最终汇总报告只能使用"已通过独立复核"的结论。
- **发现条目完整性**：所有子 agent 的输出都必须遵守"发现条目完整性守则"（见附录 A）。

### 调度模式

```
主 Agent（协调者）
  │
  │  ── 阶段一：迭代发现 ──
  │
  ├── Task → Round 01（初扫全部组件） → 保存到 round-01.md
  ├── Task → Round 02（基于 R01 递归扩展） → 保存到 round-02.md
  ├── ...（直到无新发现或达到 10 轮上限）
  │
  │  ── 阶段二：复核 ──
  │
  ├── Task → 全量维度复核（读取所有轮次，逐条保留/降级/驳回）
  ├── Task → 高风险逐项复核
  ├── Task → 低风险批量复核
  │
  └── 汇总已复核通过的结果 → summary.md
```

### 递归扩展提示词模板

从第 2 轮起，主 agent 在基础提示词后追加以下内容：

```text
---

## 递归扩展指令（第 N 轮）

以下是本审查前 N-1 轮已保存的全部发现：

[粘贴前 N-1 轮的完整发现文本]

你的任务：

1. 读取上述已有发现，理解已经覆盖了哪些组件、模式和交互路径。
2. 基于已有发现暴露的组件、模式和关联路径，**继续深挖**尚未覆盖的盲区：
   - 已有发现涉及的组件是否有同类型问题未被检出？
   - 已有发现的模式是否在其他组件中也存在？
   - 第 1 轮扫描中是否有尚未覆盖的组件或交互场景？
   - 是否有跨组件交互场景（如"表格 + 筛选 + 排序 + 分页组合使用时"）未被检查？
   - 是否有第 1 轮未检查到的边缘场景（如 disabled 状态、空数据、加载中、错误状态）？
   - 是否有响应式/移动端场景的视觉问题？
3. **只输出新发现的条目**。不要重复已有发现中已经报告的内容。
4. 如果经过仔细检查后确实没有发现新的问题，输出："未发现新的问题。审查结束。"
5. 如果仍能找到零散细节，但这些细节只是低价值、机械重复、缺少明确用户影响，不要为了凑数量继续报告；输出："未发现新的高价值问题。审查结束。"
6. 每个新发现仍需遵守统一的发现格式（见附录 A）。
```

### 保存与追加规则

1. **每轮结果独立文件**：`round-01.md`, `round-02.md`, ... 不覆盖、不合并。
2. **复核结论**：`review.md` — 引用轮次和发现编号，不重写发现内容。
3. **汇总报告**：`summary.md` — 仅包含已通过独立复核的结论。

## 共享提示词前缀

主 agent 在派发任一轮次前，应先附加以下共享前缀：

```text
你正在审查 nop-chaos-flux 项目的 UI/UX 设计合规性。这是一个 React 19 + Zustand + TypeScript 的低代码渲染引擎 monorepo（pnpm workspace）。

执行前先阅读：
1. docs/index.md
2. AGENTS.md
3. docs/architecture/styling-system.md
4. docs/architecture/renderer-markers-and-selectors.md
5. docs/architecture/theme-compatibility.md
6. packages/ui/src/index.ts（可用组件清单）

审查范围：
- packages/flux-renderers-form/src/
- packages/flux-renderers-form-advanced/src/
- packages/flux-renderers-data/src/
- packages/flux-renderers-basic/src/
- packages/ui/src/components/（作为"正确用法"参照）
- apps/playground/src/（实际使用场景）

审查标准：
- 以用户可见的视觉和交互质量为准，不是代码架构合规性
- 以行业常见设计模式为参照（Ant Design、shadcn/ui、MUI、AG Grid 等）
- 同一语义操作在不同组件中的视觉表现应一致
- 交互状态（hover、focus、active、disabled、loading、empty）应有恰当的视觉指示
- 不要求追求"最完美"的设计，但要求没有明显违背行业惯例的问题

通用口径：
1. 以当前代码为准，不与历史版本对比。
2. 不重复报告架构契约问题（由 deep-audit 维度 09-12 覆盖）。
3. 不把"不够美观"当问题，必须有明确的用户交互障碍或行业惯例偏离。
4. 每个发现必须可定位：文件路径 + 行号范围 + 3-10 行证据片段。
5. 区分"功能缺陷"和"设计不佳"：功能缺陷（如点击标签无法切换 checkbox）优先级高于纯视觉问题。
6. 所有结论都必须经过独立复核。

严重程度判级：
- HIGH: 当前已构成用户交互障碍、功能缺陷、或严重违背行业惯例
- MEDIUM: 不一致性问题、缺失的视觉反馈、可改进的交互模式
- LOW: 视觉细节优化、边缘场景的微调

【发现条目完整性 — 强制规则】：
1. 每个发现必须按统一格式（附录 A）完整输出。
2. 禁止将发现压缩为一行标题或表格单元格。
3. 禁止用摘要替代完整条目。
4. 证据代码片段不可省略。
5. 每个发现条目最少 8 行。
```

## 第 1 轮（初扫）提示词

````
以下为第 1 轮初扫的维度正文。派发时必须与"共享提示词前缀"拼接。

## 检查视角

按以下 10 个视角逐个扫描所有渲染器组件。每个视角覆盖一类 UI/UX 问题：

### 视角 1：图标语义正确性

1. 在所有渲染器中搜索 lucide-react 图标的使用。
2. 对每个图标判断其语义是否匹配当前操作：
   - 删除操作是否使用 Trash2Icon 或 XIcon（而非文本字符如 ×）
   - 新增操作是否使用 PlusIcon
   - 筛选操作是否使用 ListFilterIcon 或 FilterIcon（而非 ChevronDownIcon）
   - 展开/折叠是否使用方向一致的图标（ChevronDown/ChevronRight 或 ChevronUp/ChevronDown）
   - 排序指示是否区分方向（ArrowUp/ArrowDown/ArrowUpDown 三态）
3. 同一语义操作是否在不同组件中使用了不同图标。

### 视角 2：按钮样式一致性

1. 搜索所有 Button 和 WrappedFieldAction 的使用。
2. 按语义操作分组（删除、新增、编辑、确认、取消等）。
3. 同一组内的 variant、size、颜色是否统一。
4. 同一组内的交互模式是否统一（如有的 hover 变红、有的始终红色）。

### 视角 3：状态指示完整性

1. 检查所有可交互元素的以下状态是否有恰当的视觉表现：
   - 默认态
   - hover 态
   - focus-visible 态（必须有可见 ring）
   - active 态
   - disabled 态
   - loading 态
   - selected/active 态
2. 特别关注：
   - `tabIndex={0}` 的非 button 元素是否有 focus ring
   - 当前排序/选中/过滤状态是否有明确指示
   - indeterminate 状态是否被处理

### 视角 4：表单交互模式

1. 检查所有表单控件：
   - checkbox/radio/switch 的标签是否可点击（Label + htmlFor）
   - select 是否有正确的 placeholder 和清空机制
   - number input 的 stepper 和 suffix 是否会重叠
   - 表单验证错误的展示是否统一
2. 检查表格行选择：
   - checkbox 全选是否有 indeterminate 状态
   - radio 选择是否使用真正的 radio 语义

### 视角 5：Loading 和空状态

1. 搜索所有 loading/loading/isLoading 相关的渲染逻辑。
2. loading 状态是否有 Spinner 组件（非纯文本）。
3. 搜索所有空数据/empty 渲染。
4. 空状态是否有有意义的提示内容（非空白或空字符串）。
5. 虚拟化组件和非虚拟化组件的空状态处理是否一致。

### 视角 6：对话框和弹出层

1. 搜索所有 Dialog/Sheet/Drawer/Popover/DropdownMenu 的使用。
2. 对话框是否有关闭按钮（showCloseButton 不应为 false，除非有充分理由）。
3. 弹出层的 trigger 文案是否准确描述其行为。
4. 是否有使用不可见的原生控件（如 opacity-0 的 native select）作为交互触发器。
5. 确认/取消按钮的顺序和样式是否统一。

### 视角 7：颜色和设计令牌

1. 搜索所有硬编码的 Tailwind 颜色类（如 orange-300、blue-500 等）。
2. 判断这些颜色是否应该改为 CSS 自定义属性（设计令牌）：
   - 语义状态颜色（warning、success、danger）应使用令牌
   - 品牌色/强调色可以是硬编码（如果有意为之）
3. 检查颜色语义是否正确：
   - 红色是否仅用于破坏性操作
   - 绿色是否仅用于成功/确认
   - 蓝色是否仅用于主要操作/信息

### 视角 8：间距和对齐

1. 检查操作按钮的定位方式：
   - 是否使用 absolute 浮动定位（应改为 flex + ml-auto 对齐）
   - 删除/关闭按钮是否垂直居中
   - 操作按钮是否靠右对齐
2. 检查嵌套组件的间距是否一致：
   - 列表项之间的 gap
   - 分组标题与内容的间距
   - 操作栏的内边距

### 视角 9：ARIA 语义和角色

1. 搜索所有 role 属性的使用：
   - 图表是否使用了 role="img"（非 role="button"）
   - 列表项是否使用了 role="listitem"
   - 自定义控件是否使用了正确的 role
2. 搜索所有 aria-label：
   - icon-only 按钮必须有 aria-label
   - 表单控件在无可见标签时必须有 aria-label
   - fallback 文案是否有意义（非字面 "Button"）

### 视角 10：跨组件交互一致性

1. 比较 CRUD renderer 和 Table renderer 的分页 UI。
2. 比较不同筛选/排序组件的交互模式。
3. 比较不同删除场景的确认流程。
4. 比较 loading 状态的展示方式。
5. 比较空状态的展示方式。

## 执行策略

1. 从 packages/flux-renderers-form/src/renderers/ 开始，逐个读取所有渲染器文件。
2. 然后扫描 packages/flux-renderers-form-advanced/src/。
3. 然后 packages/flux-renderers-data/src/。
4. 然后 packages/flux-renderers-basic/src/。
5. 对每个文件，按上述 10 个视角逐一检查。
6. 发现问题时，继续检查该文件是否有同类问题（不要只报一个就跳过）。

## 输出格式

按视角分组输出。对每个发现：

### [视角N-序号] 简短标题

- **文件**: `packages/xxx/src/yyy.tsx:行号范围`
- **证据片段**:
  ```tsx
  // 贴出 3-10 行与结论直接相关的原文
````

- **严重程度**: HIGH / MEDIUM / LOW
- **现状**: 当前实现是什么
- **行业惯例**: 行业标准做法是什么
- **用户影响**: 用户会看到什么、遇到什么困难
- **建议**: 修复方向
- **复核状态**: 未复核

最后输出：

1. 按严重程度排序的问题清单
2. 按组件分组的问题清单
3. 总体评估

```

## 结果落盘与归档

### 目录结构
```

docs/analysis/{YYYY}-{MM}-{DD}-ux-audit/

````
### 保存规则

1. 每轮结果保存为独立文件，不覆盖、不合并。
2. 主 agent 保存前必须检查每个发现条目的完整性（≥8 行、有证据片段、有严重程度、有建议）。
3. 不合格的输出必须要求子 agent 重新生成后再保存。

### 汇总报告格式

```markdown
# UI/UX 设计合规性审查汇总报告

## 审查范围
- 扫描的包：[列表]
- 审查日期：YYYY-MM-DD
````

## 附录 A：发现条目格式

每个发现必须按以下完整格式输出。**不得压缩、省略或简化。**

````markdown
### [视角N-序号] 简短标题

- **文件**: `packages/xxx/src/yyy.tsx:行号范围`
- **证据片段**:
  ```tsx
  // 3-10 行与结论直接相关的原文代码
  ```
````

- **严重程度**: HIGH / MEDIUM / LOW
- **现状**: 当前实现是什么（一两句话）
- **行业惯例**: 行业标准做法是什么（引用具体系统的做法）
- **用户影响**: 用户会看到什么、遇到什么困难
- **建议**: 具体的修复方向（不是"应该改进"，而是"应改为 X 图标 + Y 样式"）
- **复核状态**: 未复核 / 复核通过 / 已降级 / 已驳回

```

**反压缩规则**：

| 反模式 | 正确做法 |
|--------|----------|
| 只写标题不写详情 | 必须有完整的 8 行以上条目 |
| 省略证据片段 | 必须贴 3-10 行代码 |
| 只写"应改进"不写具体方向 | 必须写出具体的图标名/variant/class 建议 |
| 不说明行业惯例 | 必须说明行业标准做法（可引用具体系统） |
| 不说明用户影响 | 必须描述用户会看到什么、遇到什么困难 |

## 附录 B：执行注意事项

1. **子 agent 类型**: 使用 `subagent_type="explore"`。
2. **派发时必须拼接共享前缀**: 第 1 轮使用"共享前缀 + 第 1 轮正文"；后续轮次使用"共享前缀 + 递归扩展指令"。
3. **复核是强制步骤**: 不能跳过复核直接汇总。
4. **复核 agent 必须独立**: 不能让发现 agent 自己复核自己。
5. **增量审查**: 可以只聚焦特定包（如只审查 flux-renderers-form），不必每次全量。
6. **参考已有分析**: 如果 `docs/analysis/` 中已有 UI/UX 审查报告，先快速浏览以避免重复，但不以此为限制。
7. **多轮递归不是线性**: 第 N 轮可以跳到任何上一轮发现中涉及的文件或模式，不限于第 N-1 轮。
8. **每轮上限**: 最多 10 轮。达到上限后强制进入复核阶段。
```
