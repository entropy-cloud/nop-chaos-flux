# 文稿去噪审稿规则

> **定位**: 用来清除文稿里的废话、聊天残留、AI 套话和伪深刻表达。
> **边界**: 这份规则只审表达，不替作者重写理论。理论错了另开架构或内容审查。

## 语感初筛（每次审稿必须最先执行）

在进入任何规则扫查之前，先做一件事：

从头到尾默读一遍全文。读到任何一个地方让你觉得：

- “话好像不是这么说的”
- “这句子得在心里翻译成英文才通”
- “书面语和口语奇怪地混在一起”
- “名词摞名词，断句费劲”
- “读起来要回头找主语才懂”
- “语法没错，但中国人不这么讲话”

就立即标黄。这一步只凭语感，不需要讲道理，不需要归类。

扫完之后，再对照后面的规则，把标黄的句子逐一归类修改。

这一步是整份规则最重要的前置动作。绝大多数翻译腔和生硬表达会在这一步被一次性捕获，不会被后续的"找废话"夺走注意力。

## 审稿目标

把文稿从"回答得很完整"改成"文章真的站得住"。

要删的是噪音，不是锋芒。不要把文本磨成四平八稳的说明书。好文章允许有钉子句、短句、断裂感和明确判断；真正该清掉的是没有信息增量的顺滑。

## 第一判断

先问一句：这句话删掉后，读者会少知道什么？

- 少一个事实，保留。
- 少一个边界，保留。
- 少一个例子，保留。
- 只少一点"过渡感""礼貌感""总结感"，删。

多数 AI 味不是错在句子不通，而是错在句子太通。它们把文章铺得很平，却没有让判断往前走。

## 必须删除的内容

### 写作动作暴露

这些话是在告诉读者"我要开始写了"，不是在写：

- `下面我们来看`
- `这里需要先说明`
- `说到这里`
- `到这里还有一个问题`
- `把话说得更直白一点`
- `换句话说`
- `在继续往下讲之前`
- `如果让我用一句话概括`

处理方式：能删就删。删了断，就把后面的实义内容提到句首。

差：`这里需要先说明，日志不是 source of truth。`

好：`日志不是 source of truth。`

#### 论证自述

这些句子不描述转场，而是描述"我正要做什么论证动作"：

- `答案是否定的`
- `答案是...`
- `引出一个更强的追问`
- `另一个更难的问题是`
- `类比点：`
- `核心差异在于`
- `三者共享同一个起点`

处理方式：删掉自述部分，只保留实义内容直接陈述。

差：`数据流问题引出一个更强的追问：RMSP Agent 能否内化 AGE？答案是否定的，原因在于时间结构不匹配。`

好：`RMSP Agent 能否内化 AGE？不能。原因在于时间结构不匹配。`

差：`另一个更难的问题是两者冲突时谁优先。答案是 attractor 优先。`

好：`两者冲突时 attractor 优先。`

### 模板化正确废话

这些句子看起来稳，但不能排除任何误解：

- `这不仅是技术问题，更是工程问题`
- `这背后反映了更深层的问题`
- `本质上是一个系统性问题`
- `值得注意的是`
- `需要指出的是`
- `从某种意义上说`
- `共享同一个起点` / `共享同一条X`
- `遵守同一条纪律` / `遵循同一个原则`

处理方式：后面没有具体机制，整句删。后面有机制，就让机制直接开头。

差：`这背后反映了更深层的问题：测试没有证明真实路径。`

好：`测试没有证明真实路径。`

### 会话腔缓冲

聊天里有用，文章里常常拖慢：

- `其实`
- `当然`
- `你会发现`
- `很多人会说`
- `这很自然`
- `某种程度上`

处理方式：删掉后逻辑不变，就删。连续几段都靠这些词开头，说明文章在靠口气推进，不是靠内容推进。

这些词不是禁词。`当然` 可以承担让步，`其实` 可以承担反预期，`很多人会说` 可以引入真实反方。保留条件很简单：后文兑现了这个转折。如果只是垫嗓子，删。

### 假有力动词

这些词常被用来伪装判断力：

- `拉住`
- `钉死`
- `压回去`
- `兜住`
- `立住`
- `边界变硬`

不是禁用。问题在于它们经常没有对应机制。

先问：谁通过什么机制把什么约束住了？答不出来，就换成结构性动词：`定义`、`固定`、`排除`、`约束`、`收敛到`、`拒绝`。

差：`owner docs 把系统压回正确方向。`

好：`owner docs 固定长期语义边界，排除与当前基线冲突的实现。`

### 伪反叛语气

另一种 AI 味会伪装成"不 AI"：句子很凶，判断很空。

常见模式：

- `真正的问题不是 X，而是 Y`
- `这句话必须说重一点`
- `很多人不愿意承认`
- `残酷地说`
- `这不是选择题`

处理方式：删掉姿态词，只保留判断成本。

先问：这句话得罪了哪个具体误解？排除了哪个具体做法？指出了哪个具体代价？答不出来，狠话只是装饰。

差：`残酷地说，很多团队根本没有真正理解 AI 工程。`

好：`很多团队只把 AI 接进代码生成环节，没有建立独立验收和缺陷记忆。`

### 空转对比句

`不是 A，而是 B` 可以是钉子，也可以是空转。

保留条件：它明确排除了一个常见误解。

删除条件：A 和 B 都是抽象词，只是在换说法加气势。

差：`这不是普通流程，而是更深层的工程能力。`

好：`它不是任务清单，而是判断一次变更能否关闭的证据结构。`

### 句子片段

半截问句或从句独立成句，用句号结尾。读起来像注释或笔记混入正文，不是完整的文章语言。

常见模式：

- `谁优先。`
- `为什么不能。`
- `怎么办。`
- `如何取舍。`

处理方式：把碎片整合到上下文里，改成一个完整句子。

差：`两者冲突时谁优先。如果 Agent 的 aspect 图漂移，attractor 优先。`

好：`两者冲突时 attractor 优先——跨 session 的项目一致性权重高于单次 Agent 适应的局部最优。`

### 翻译腔和硬翻译

翻译腔常常不是语法错误，而是中文读起来别扭。它会让文章像英文材料转写，不像中文作者直接表达。

常见症状：

- 英文语序硬搬到中文里。
- 名词串太长，动词太少。
- `这种做法并不一定...`、`它是一个...的问题` 这类英文句式痕迹太重。
- 半中半英不是术语需要，而是懒得转成中文。
- `被` 字句过多，责任和动作主体都变模糊。
- 抽象名词堆叠：`维护策略的系统化方案的缺乏`。
- 动词悬空缺少工具格：`借一个类比`。
- 动宾搭配在中文里不成立：`承担载体`。
- 被动结构缺少主语：`方向已经知道了`。
- 形容词与名词的搭配在中文里不存在：`高度像对`。

处理方式：不是"找个近义词替换"，而是把整句打散，用中国人讲同一件事时会用的句型和节奏重写。如果改完的句子读起来像在说人话，就算过。

差：`这种开发方式并不一定马上坏。`

好：`这种开发方式短期内未必出问题。`

差：`长期维护策略缺乏系统化的方案。`

好：`长期维护缺少一套系统方案。`

差：`如果借一个数学类比。`

好：`如果借用一个数学类比。`

差：`危险不在明显胡说，而在局部信号高度像对。`

好：`危险不在于明显胡说，而在于局部信号看起来高度合理。`

差：`仓库开始承担系统真相的载体。`

好：`仓库开始成为系统真相的载体。`

差：`系统正确的方向已经知道了。`

好：`系统正确的方向是已知的。` 或 `我们已经清楚系统应当收敛到怎样的结构。`

## 不能误删的东西

不要为了"去 AI 味"把文章削成无菌文本。

必须保留：

- 概念定义第一次出现时的硬句。
- 防止误读的反定义。
- 能改变读者判断的例子。
- 文件路径、计划编号、bug 记录、测试名这类抓手。
- 必要的重复，前提是每次重复新增一层含义。

一句话如果删掉后文章更顺，但边界更软，这句话应该留。

## 整篇坏味道

逐句清理不够。AI 味经常藏在结构里。

### 结构过于平均

模型生成的文章常常不是乱，而是太整齐。

坏症状：

- 每节都是"定义 -> 三点 -> 小结"。
- 段落长度接近，读起来没有轻重。
- 所有条目语法结构相同，像批量生成。
- 每个标题下面都先铺一句"本节将说明"。
- 清单里的每项都差不多长、差不多抽象、差不多正确。

处理方式：打破等权结构。重要判断展开，不重要判断合并。能用一个反例击穿的，不要列三条平行解释。

清单不是错。错的是让所有条目看起来拥有同等重量。

### 讨论稿没有变成文章

坏症状：

- `前面已经说过`
- `在继续往下讲之前`
- `这里先把一个误解排掉`
- `很多人听到这里会觉得`

处理方式：把讨论过程改成文章结论。除非历史过程本身是证据，否则不要把聊天现场搬进正文。

### 概念先用后定义

核心概念第一次承担论证重量之前，必须先定义。不要让读者先接受结论，再到后面才知道术语是什么意思。

如果用了物理、数学、哲学类比，马上给工程映射。类比不能单独承担论证。

### 同一个层级关系反复铺开

方法论文章最容易反复解释"先后""层级""顺序不能反"。一个 hierarchy 只完整定义一次。后文引用它带来的工程后果，不要重新铺一遍。

### 句句正确，但没有抓手

这种最像 AI：每段都能点头，读完什么都抓不住。

补抓手：文件、路径、计划编号、测试名、失败案例、反例、术语边界。一个段落删掉具体名词后仍然成立，通常太空。

### 只赢弱反方

AI 文稿常常能把正方观点讲顺，但只反驳了对方最弱的版本。读者一旦把反方方案强化到合理形态，文章就失去说服力。

坏症状：

- 把反方写成明显愚蠢、懒惰或不完整的做法。
- 只说某机制“可能没执行”，没有处理它被强制执行后的版本。
- 用“这就是降级”“这就是本质区别”收尾，却没有写清结构差异。
- 文章看似赢了，但只赢了一个 straw man。

处理方式：把反方强化到最合理版本，再问你的结论是否仍成立。若仍成立，写出真正不可替代的结构关系。

差：`age-skill 可能没有被加载，所以 AGE 不能做成 skill。`

好：`即使 age-skill 被 always-on / global 加载，它仍然是从外部施加到仓库上的能力包；AGE 要成为仓库自身的底层拓扑。`

差：`这种测试不够真实。`

好：`这个测试只检查调试 JSON，没有断言用户可见内容；即使它稳定通过，也不能证明用户路径有效。`

### 压缩压掉了承重句

去 AI 味不是把文章越改越短。AI 文稿的问题通常是重复、顺滑、没有抓手；但有些句子是论证承重墙，删掉后文章反而变空。

坏症状：

- 为了精炼，把因果链的起点删掉，只留下结论。
- 删除看起来重复的句子后，概念边界变软。
- 把具体机制压成抽象判断。
- 把强前提改成温和背景。

处理方式：先找论证脊柱，再删枝叶。承重句必须保留，重复句只有在不新增事实、边界、机制、反驳或证据时才删。

差：`AI 扰动很快，所以仓库要成为真相源。`

好：`AI 在状态空间中引入的扩张力超过少数人类的信息处理能力；除非强行降速，否则系统真相源就不可能再是人的脑内记忆。`

### 过度安全，每个判断都包退路

平庸文本的典型症状是：没有一句错，也没有一句值得记住。

过度安全的表现：

- 先批评，再补一句"这并不是说它没有价值"。
- 先下判断，再补一句"具体还要看场景"。
- 把明确错误改成"存在一定风险"。
- 把"不成立"改成"还可以进一步完善"。

如果文本已经给出证据，就不要再用礼貌句撤回判断。

差：`这个测试存在一定不足，后续可以进一步完善。`

好：`这个测试没有点击保存按钮，不能证明保存功能有效。`

## 如何改

### 1. 先过语感初筛

标黄所有读起来不像中国人正常说话的句子。这一步不分类、不讲道理、不跳过。

### 2. 扫元话语

标出这些词：`这里`、`下面`、`说到这里`、`换句话说`、`更准确地说`、`值得注意的是`、`需要指出的是`、`某种意义上`。

逐个判断：它是在承载内容，还是在模拟转场？后者删。

### 3. 每段必须有一个主推进

不要机械要求"每段只有一个判断"。一个段落有时需要两个判断形成冲突、转折或推进。

真正要问的是：这一段有没有推进读者判断？

允许两个判断互相咬住。禁止三句并列正确话轮流占位。

### 4. 把抽象判断压到具体对象上

优先补：

- 文件路径
- 真实例子
- 具体误读
- 失败案例
- 明确机制
- 当前基线

不要用"更深层""系统性""工程能力"这类词替代机制。

### 5. 反驳最强反方

不要只问“我的论证顺不顺”。要问：如果反方采用最强、最合理、最工程化的版本，我的结论还成立吗？

把反方强化之后再写反驳：

- `可能不会调用` -> `即使 always-on 也不够，因为...`
- `测试不真实` -> `即使测试稳定通过也不够，因为它没有证明...`
- `文档会过时` -> `即使文档存在也不够，因为它没有 owner / proof / freshness 关系...`

### 6. 保留文章的硬度

不要把所有句子都改成长句。不要把判断都改成"可能、通常、在一定程度上"。该硬的地方要硬。

### 7. 处理翻译腔：打散重写，不要近义词替换

翻译腔的根子通常是整句结构来自英文，不是某个词选错了。改的时候把句子拆开，用中文的节奏重新组织。改完读一遍，像人话就算过。

## 快速问题单

交稿前问十四个问题：

1. 有没有句子只是在描述写作动作？
2. 有没有一整段删掉承接词后意思不变？
3. 有没有三句换词重复同一个判断？
4. 有没有大判断后面没有机制、例子或证据？
5. 有没有"不是 A，而是 B"只是在制造气势？
6. 有没有核心概念在定义前就被大量使用？
7. 有没有同一个 hierarchy 在不同章节反复展开？
8. 有没有把讨论过程当成正文？
9. 有没有为了稳妥把锋利判断磨钝？
10. 有没有翻译腔、硬翻译、半中半英导致中文不顺？
11. 有没有一个段落没有任何具体名词仍然成立？
12. 有没有哪个句子读起来不像中国人会说的话？
13. 有没有只反驳了对方最弱版本，没有处理 always-on / 强制执行 / 仓库感知后的版本？
14. 有没有为了精炼删掉了承重前提、真实证据或防误读边界？

## 审稿输出格式

审稿不能只贴标签。每条意见必须带证据。

每条至少包含：

- 原句或原段。
- 问题类型。
- 为什么它没有信息增量，或为什么它磨平了判断。
- 建议改法。

三类结果只用于排序，不用于偷懒。

### 必删

- 纯写作动作。
- 纯模板废话。
- 纯重复总结。
- 依赖聊天上下文的台词。

### 应改

- 抽象判断缺少抓手。
- 对比句空转。
- 假有力动词没有机制。
- 概念定义滞后。
- 翻译腔和生硬表达。
- 过度安全的退路句。

### 必留

- 理论钉子句。
- 防误读句。
- 真实例子。
- repo 映射。
- 概念第一次定义。

## 非工程文章也一样查

这份规则不只用于工程方法论。对外文章、产品稿、品牌稿同样会有 AI 味。

差：`这不仅是一次产品升级，更是品牌理念的全面进化。`

好：`这次升级只改了三个入口：注册、搜索和账单。品牌理念没有变化。`

差：`我们希望为用户打造更加智能、高效、便捷的全新体验。`

好：`用户现在可以在账单页直接补开发票，不需要再进入客服工单。`
