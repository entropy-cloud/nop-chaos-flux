# 下一代低代码平台新吸引子探索（独立实验 v1）

> 约束：本实验刻意不参考本项目既有文档与代码，只基于通用低代码领域问题、主流产品形态与架构演化规律进行探索。

## 当前问题域

面向下一代低代码平台，重新定义以下核心问题：

- 如何让“页面结构、业务数据、可执行行为、宿主能力、多人协作变更”不再被塞进同一套 schema 解释器。
- 如何让低代码从“描述一个会跑的页面”升级为“声明一组可审计、可推演、可组合、可协作的业务状态空间变换”。
- 如何让 AI 生成、人工维护、在线运行、版本演化四件事使用同一套核心语义，而不是四套彼此泄漏的抽象。

## 当前系统基线

这里的“当前系统”不是指某个具体项目，而是主流前端低代码平台的平均基线：

- 以页面/表单 schema 为中心，组件树承担主要组织职责。
- 数据、动作、校验、联动、权限、可见性通常挂在节点属性或事件配置上。
- 运行时以解释执行为主，编译只是做轻量规范化。
- 宿主能力被包装为 API、service、action、plugin 或 expression context 注入。
- 协作、审计、回放、AI 生成通常是外围能力，不是内核第一性对象。

## 当前不满意的症状

- 结构中心是组件树，导致业务语义只能寄生在 UI 节点上。
- 数据与能力混杂，任何地方都可能“顺手拿到”不该拿到的上下文。
- 动作语义不纯，既像事件处理器，又像流程节点，还像副作用脚本入口。
- 页面能跑，但无法稳定回答“为什么在这个状态下做出了这个结果”。
- AI 能生成初稿，但难以生成可长期维护、可审计、可迁移的系统。
- 多人协作通常停留在 schema merge，而不是语义级协作。

## 1. 当前吸引子识别

当前主流低代码方案依赖的核心结构假设：

1. UI 树天然就是系统主索引，其他语义都应附着其上。
2. “数据访问方便”比“边界纯净”更重要，因此上下文默认宽暴露。
3. 动作本质上是事件回调的结构化变体，不需要独立语义层。
4. 运行时解释器是核心，编译期只负责把输入变得更好跑。
5. 协作、审计、AI 生成是附加能力，不必进入最小语义闭包。

明显来自主流框架平均路径的假设：

- 假设 1 来自 schema-driven UI 的共同惯性。
- 假设 2 来自表单框架、页面搭建器、表达式上下文模型的平均做法。
- 假设 3 来自事件驱动前端与 BPMN-lite 的混合传统。
- 假设 4 来自“配置驱动渲染器”的工具链习惯。
- 假设 5 来自主流产品把平台内核定义得过窄。

## 2. 混合概念清单

1. 组件结构 vs 业务结构
   结构性代价：业务边界只能借 UI 层级表达，UI 重排会污染业务语义。
2. 数据 owner vs 数据视图
   结构性代价：谁拥有状态、谁消费状态、谁可变更状态经常无法区分。
3. 用户意图 vs 可执行动作
   结构性代价：同一个 action 既承担交互入口，又承担流程编排，又承担副作用落地。
4. 宿主能力 vs 业务权限
   结构性代价：能调用某个 API 不等于语义上被允许，但系统往往把它们混成一层。
5. 本地编辑变更 vs 运行时状态变更
   结构性代价：设计期、发布期、运行期缺少统一的变更语义，难以审计与回放。
6. 校验规则 vs 业务约束
   结构性代价：字段校验和跨实体约束被放进同一套机制，导致能力下沉不足或过度上浮。

## 3. 替代切分矩阵

| 方案                  | 概念中心           | 保留什么                                         | 放弃什么                       | 复杂度迁移到哪里                   |
| --------------------- | ------------------ | ------------------------------------------------ | ------------------------------ | ---------------------------------- |
| A. 纯词法作用域页面机 | 页面节点与词法环境 | 保留 schema 页面开发习惯                         | 放弃大部分动态上下文注入       | 迁移到作用域声明与静态分析         |
| B. 以动作图为核心     | 业务动作与依赖图   | 保留流程编排可视化                               | 放弃 UI 作为主组织轴           | 迁移到动作图调度与数据映射         |
| C. 以实体状态机为核心 | 领域对象生命周期   | 保留表单与列表作为投影                           | 放弃页面中心化建模             | 迁移到领域建模与状态约束           |
| D. 以提交单元为核心   | 可审计变更包       | 保留页面、表单、动作，但都降为生成提交单元的界面 | 放弃“直接操作运行时对象”的便利 | 迁移到提交协议、约束求解与冲突处理 |
| E. 以能力租约为核心   | capability lease   | 保留插件与宿主扩展                               | 放弃任意表达式随处拿能力       | 迁移到能力声明、续租与失效治理     |

五套切分的进一步说明：

1. A 把核心问题理解为“可见性污染”。它会显著改善局部可推理性，但仍然默认页面是主语。
2. B 把核心问题理解为“动作失控”。它更适合流程密集系统，但容易回流到 BPMN-like 编排器。
3. C 把核心问题理解为“领域对象缺位”。它对复杂业务更强，但对通用搭建场景门槛较高。
4. D 把核心问题理解为“系统缺少第一性变更语义”。它让 UI、AI、协作、审计共享同一执行核。
5. E 把核心问题理解为“能力边界污染”。它很关键，但单独成立时更像安全层升级，不足以定义整个平台。

## 4. 主流盆地回流检测

1. A 会部分回流到表单框架盆地。
   原因：它主要解决上下文污染，却没有改变“页面 schema 驱动 + 事件回调”的主结构。
2. B 会高度回流到 BPMN/action graph 盆地。
   原因：一旦动作图成为主轴，系统很容易重新变成可视化流程编排器。
3. C 会部分回流到 DDD + state machine 平台盆地。
   原因：它强化领域边界，但未必重写低代码平台对 UI、协作、AI 的统一语义。
4. D 不明显回流到现有主流盆地。
   原因：它不以页面、动作图、表单、流程为主轴，而以“提交单元”作为唯一一等执行对象。
5. E 会回流到 capability-based security 与 plugin sandbox 盆地。
   原因：它更像关键子系统，而不是新平台总吸引子。

结论：

- A 是必要升级，但不是新吸引子。
- B 是旧盆地的强化版。
- C 有潜力，但仍不够普适。
- D 是最值得保留的候选。
- E 应作为 D 的配套边界，而不是独立总方案。

## 5. 新吸引子候选

### 候选 1：Commit-Oriented Low-Code Kernel（COLK，提交导向低代码内核）

核心定义：

- 平台的最小执行对象不是页面节点，不是 action，不是流程图节点，而是 `Commit Unit`。
- 任意交互、自动化、AI 代理、外部事件都只能提议一个提交单元。
- 提交单元声明“谁要改什么、依据什么约束、需要什么能力、对哪些投影产生影响”。
- 运行时不直接执行脚本式动作，而是对提交单元做 admission、constraint solving、effect scheduling、journal append。

为什么它不是旧方案换词重说：

- 它把“动作”降级为提交生成器，把“页面”降级为提交入口，把“流程”降级为提交链路，而不是保留它们为一等对象。
- 它把协作、审计、AI 生成直接拉进内核，因为它们都围绕提交单元展开。
- 它的评价函数从“页面是否能跑”改为“提交是否可解释、可审计、可合并、可重放”。

它重新定义的边界：

- UI 不再拥有业务语义，只拥有提交提议权。
- 动作不再直接拥有副作用执行权，只拥有提交构造权。
- 宿主能力不再是上下文可读对象，而是提交执行时申请的租约资源。
- 协作不再基于 schema diff，而基于提交日志与约束冲突。

### 候选 2：Projection-Native Business Fabric（PNBF，投影原生业务织体）

核心定义：

- 系统主语不是页面，也不是实体，而是 `Projection Contract`。
- 页面、表单、报表、机器人、外部 API 都只是不同投影端点。
- 业务变化先写入一个共享业务织体，再由不同投影订阅与派生。

为什么它不是旧方案换词重说：

- 它把“渲染结果”从终点变成投影副产品。
- 它让同一业务语义天然面向多端、多角色、多代理，而不是先建页面再做适配。

它重新定义的边界：

- 前端低代码不再是页面搭建平台，而是业务投影平台。
- 组件库从核心层退出，投影协议上升为核心层。

保留判定：

- 候选 2 有启发，但容易与 event sourcing、CQRS、projection system 重叠。
- 候选 1 更强，因为它同时覆盖运行、审计、协作、AI 生成和宿主执行边界。

## 6. 反例与失败模式

### 候选 1：COLK

1. 高频细粒度表单输入场景。
   风险：每次键入若都形成提交，将导致系统噪声与性能灾难。
   判断：实现层与概念层共同作用。需要区分草稿态变更与业务提交，不然概念本身不完整。
2. 强实时协同画布场景。
   风险：大量临时意图、光标移动、拖拽预览并不适合走完整提交协议。
   判断：概念层需要承认“ephemeral intent lane”存在，否则会把所有交互都过度制度化。
3. 纯展示型页面场景。
   风险：如果没有业务提交需求，COLK 可能显得过重。
   判断：概念层问题较小，但需要允许只读投影模式，否则适用面被错误压缩。
4. 极端脚本自由度诉求场景。
   风险：开发者会觉得系统不够“顺手”，因为不能随处直接改上下文。
   判断：这是旧吸引子被废弃后的摩擦，不是缺陷本身，但 adoption 成本真实存在。

### 候选 2：PNBF

1. 简单单页表单场景。
   风险：投影织体抽象过重，难以解释为何不直接做页面 schema。
   判断：概念层过宽。
2. 复杂宿主集成场景。
   风险：若没有独立提交语义，投影更新与副作用执行仍会重新耦合。
   判断：概念层不足。
3. AI 自动生成场景。
   风险：AI 容易重新生成“页面中心”的伪投影模型，导致回流主流盆地。
   判断：概念与 adoption 双重风险。

## 7. 最小可执行语言

如果候选 1 成立，至少需要以下新术语与不变量：

1. `Commit Unit`
   定义：一次业务意图提交的最小封装。
2. `Intent Producer`
   定义：页面、机器人、定时器、外部 webhook 等提交生成端。
3. `Admission Contract`
   定义：一个提交进入执行核前必须满足的可见性、权限、上下文、依赖约束。
4. `Capability Lease`
   定义：提交执行期间临时授予的宿主能力句柄，不可被任意传播。
5. `Constraint Set`
   定义：对状态转换成立条件的声明式约束集合。
6. `Effect Schedule`
   定义：提交通过后允许发生的副作用计划，而非内联脚本立即执行。
7. `Journal`
   定义：系统唯一可信变更历史，不区分人工、AI、自动化来源。
8. `Projection`
   定义：从 journal 与当前状态派生出来的任意界面、报表、视图、提醒或外部接口输出。

必须成立的不变量：

- 任意可持久业务变化都必须来源于 `Commit Unit`。
- 任意宿主能力调用都必须绑定在 `Capability Lease` 上。
- 任意副作用都必须可追溯到某个提交及其 admission 结果。
- UI 可以失败重渲染，但 journal 语义不可歧义。

## 8. 收敛验证计划

1. 最小切片一：单表单提交核
   目标：验证表单不再直接改状态，而是生成提交单元。
   观察指标：字段联动是否可由提交前约束和提交后投影替代；调试信息是否比传统 action 更可解释。
2. 最小切片二：多人并发编辑同一业务对象
   目标：验证冲突处理是否可在提交协议层完成，而不是 UI patch merge。
   观察指标：冲突是否能精确归因到意图与约束，而不是“某个字段被覆盖了”。
3. 最小切片三：AI 代理代填与人工复核
   目标：验证 AI 是否能只生成提交提议，而不越权直接操纵运行时。
   观察指标：AI 输出是否更稳定、可审计、可回滚。
4. 最小切片四：外部系统回调触发业务更新
   目标：验证 webhook 与前端交互是否能共用同一提交核。
   观察指标：跨入口一致性是否显著高于“页面 action 一套、后端流程一套”的传统模式。

收敛判据：

- 若四个切片最终都需要重新引入“页面节点直接改状态”“action 直接拿全局上下文”“宿主能力随处可读”，则候选失败。
- 若四个切片可以共享同一提交语言、同一 admission 机制、同一 journal 语义，则候选成立概率显著上升。

## 9. 结论

当前最值得验证的新吸引子是 `Commit-Oriented Low-Code Kernel`。

为什么它明显优于其他方案：

1. 它统一了 UI 交互、自动化、AI 代理、外部事件四类入口，其他方案通常只能统一其中两类。
2. 它把审计、协作、回放从外围补丁提升为内核能力，其他方案大多仍依赖事后日志。
3. 它让宿主能力使用变成显式租约，而不是模糊上下文，语义纯度和安全边界明显更高。
4. 它把“能跑”替换为“可解释、可合并、可重放”，更适合未来 AI 大规模参与构建与运维。
5. 它保留页面、表单、流程这些熟悉表层，但剥夺它们作为内核主语的地位，因此比主流方案更容易形成真正的新收敛方向。

它最可能失败在：

- 把所有交互都错误地提交化，导致系统过重。
- 无法设计出足够简单的草稿态与正式提交分层。
- 团队在 adoption 时偷偷把旧 action 语义塞回提交对象，导致概念反污染。

必须明确废弃的一条旧吸引子：

- “页面节点及其事件处理器天然就是低代码平台的一等执行中心。”

如果这条旧吸引子不被废弃，任何新方案最终都会退化成更复杂的页面 schema 解释器，而不会成为下一代低代码平台。
