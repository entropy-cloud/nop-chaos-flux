### 2026-05-11

- completed jsdom-to-happy-dom migration (plan 240) by introducing a global `test-setup-happy-dom.ts` polyfill that fixes `@testing-library/dom`'s `getByLabelText` and `window.confirm` compatibility with happy-dom.
- root cause: happy-dom implements `element.labels` on labelable element prototypes, but its `HTMLLabelElementUtility.getAssociatedLabelElements` has subtle differences from browser behavior. `@testing-library/dom`'s `getRealLabels` uses `element.labels !== undefined` as feature detection — happy-dom's implementation causes the label-based query path to misbehave, preventing `aria-label` elements from being found. The fix deletes `labels` from `HTMLButtonElement`, `HTMLInputElement`, `HTMLSelectElement`, and `HTMLTextAreaElement` prototypes, forcing testing-library to use its own polyfill (`querySelectorAll('label')` + `label.control`) which works correctly in happy-dom.
- also polyfills `window.confirm` (happy-dom does not provide it).
- reverted all per-test-file workarounds (controller-inspect-form `getByText` → `getByLabelText`, word-editor `window.confirm` guard) since the global polyfill handles them.
- remaining happy-dom-specific test adjustments kept: spreadsheet `getComputedStyle` → `tagName` check (happy-dom doesn't return UA default styles), flux-renderers-basic cascade threshold (microtask timing difference).
- updated `vitest.shared.ts` to include `setupFiles` for happy-dom environment packages.
- all 7 happy-dom packages pass individually: nop-debugger (132), flux-renderers-form (180), flux-renderers-form-advanced (571), flux-renderers-basic (205), flow-designer-renderers (115), report-designer-renderers (136), spreadsheet-renderers (84), word-editor-renderers (90) — 1713 tests total.
