# 65 CRUD Query Form ID Mismatch Between Authoring Transform and Runtime Compiler Fix

## Problem

- CRUD query workflow (search/reset) silently failed: clicking "search" never found the query form component, causing `handleQuerySubmit` to fall through to `submitQueryValues(queryState, ...)` where `queryState` was always `{}`.
- The query form was EXISTING in the precompiled region (`queryFormRegion`), but the CRUD renderer could not locate it in the component registry.
- 5 e2e tests in `crud-query-and-ownership.spec.ts` failed: query filtering had no effect, fetchOnFilter was broken, refresh showed stale data.

## Diagnostic Method

- **Diagnosis difficulty**: Moderate. The symptom (query not working) was clear, but tracing why the form component was never found required comparing two parallel code paths.
- Inspected the CRUD renderer's `handleQuerySubmit` flow: it calls `componentRegistry.get(queryFormId)` where `queryFormId` is generated from `props.id` (the compiled node ID).
- The compiled node ID is generated by `createNodeId(path, schema)` in the runtime compiler. For a CRUD with no explicit `id`, it generates an ID from the path (e.g., `$.body_0_`).
- But the queryForm region was created during authoring-time transform in `data-schema-validation.ts` using a different function: `createCrudQueryFormId` → which used `schema.id` (the authoring ID, which was `undefined` when no id was set).
- The mismatch: runtime used `$.body_0_/queryForm`, authoring transform used `undefined/queryForm`. The component registry could never match them.
- Verified using Playwright `page.evaluate()` to check scope data and confirm the query form handle was not registered.

## Root Cause

Two different code paths generated the query form component ID using different ID sources:

1. **Authoring transform** (`data-schema-validation.ts:createCrudQueryFormId`): Used `schema.id` from the authoring schema. When no explicit `id` was set on the CRUD schema, `schema.id` was `undefined`, producing IDs like `undefined/queryForm`.

2. **Runtime compiler** (`node-compiler.ts:createNodeId`): Generated the node ID from the schema path when `schema.id` is not set, using `path.replace(/[^a-zA-Z0-9_.\-:]/g, '_')`. Produced IDs like `$.body_0_`.

The transform used the raw authoring path while the compiler used the compiled path — both must use the same ID function.

## Fix

Changed `createCrudQueryFormId` in `data-schema-validation.ts` to use `createNodeId(path, schema)` — the same function the runtime compiler uses — instead of raw `schema.id`. This guarantees the authoring transform and runtime compiler produce identical component IDs.

- `packages/flux-core/src/utils/schema.ts` — `createNodeId` function

## Notes For Future Refactors

- Any authoring-time transform that generates component IDs must use `createNodeId(path, schema)` to stay consistent with the runtime compiler's ID generation.
- The `createNodeId` function falls back to `path.replace(/[^a-zA-Z0-9_.\-:]/g, '_')` — ensure any custom path manipulation produces the same sanitized format.
- When adding new region IDs to a schema transform, always verify the generated ID matches what the runtime expects by checking the compiled template node's `id` field.
