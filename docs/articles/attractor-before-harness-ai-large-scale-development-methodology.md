# Attractor Before Harness: AI 大规模开发的方法论

如果让我用一句话概括这篇文章的观点，那就是：

**很多人以为 AI 大规模开发的核心问题是怎么搭 harness，怎么加 guardrail，怎么做 verification。其实这些都不是第一性问题。第一性问题是：这个系统到底应当向哪里长期收敛。**

这就是标题里 `Attractor Before Harness` 的意思。

不是说 harness 不重要。

恰恰相反，在 AI 参与的大型开发里，harness 非常重要，甚至比传统开发更重要。但在谈 harness 之前，必须先谈 attractor。因为 harness 解决的是“怎么管轨迹”，而 attractor 解决的是“轨迹应该被管向哪里”。

如果这个顺序搞反了，再强的 harness，也可能只是在把系统更稳定地推向错误方向。

## 一、为什么这个问题在 AI 时代会突然变得尖锐

传统开发也会有 bug、误解、返工、架构漂移。但 AI 把这些问题放大了，原因不是 AI 更笨，而是它把系统展开状态空间的速度提高了太多。

人类工程师写一个功能，通常会经历一个比较慢的过程：

- 想一想
- 写一点
- 看一眼
- 再改
- 再跑
- 再解释

AI 不是这样。AI 往往会在极短时间内同时给你：

- 一段代码
- 一套类型
- 几个测试
- 一段说明
- 一句“已经完成”的总结

最危险的地方不在于它会胡说八道。

最危险的地方在于：**它给出的东西往往局部都很像对的。**

接口有了，类型有了，测试也补了，文档也更新了，解释还特别流畅。你如果只看这些局部信号，会非常容易产生一种错觉：系统正在稳步前进。

但真实情况可能是，系统整体已经开始偏航了。

这就是 AI 时代最典型的失稳模式：

**不是某一步明显做错，而是每一步都像合理，整体却持续漂移。**

一旦你看见这种现象，就会发现，传统那套“哪一步违规了”“哪条规则没过”“哪个 PR 有问题”的语言，已经不够用了。因为问题不只是点上的错误，而是轨迹本身错了。

这时候，更合适的语言不是流程治理语言，而是动力系统语言：

- AI 提供扩张力
- 工程机制提供收敛力
- 仓库历史体现的是轨迹
- 真正的问题是，这条轨迹是不是一直贴着正确的稳定结构在走

## 二、为什么只谈 Guardrail，不够

现在很多人讲 AI engineering，最熟悉的话语是这些：

- `guardrail`
- `feedback loop`
- `verification`
- `human-in-the-loop`
- `harness`

这些词都没错，但它们有一个共同的隐前提：

**系统正确的方向已经知道了，现在的问题只是怎么防错，怎么纠偏，怎么把执行过程组织好。**

这在很多小任务里是成立的。

比如做一个脚本，修一个页面 bug，补一个 CRUD 功能，你大概率已经知道“正确结果长什么样”，剩下的确实只是执行和验证问题。

但大型系统不是这样。

大型系统真正困难的地方，往往不是“如何防止越线”，而是“到底哪条路才是长期正确的结构”。

这两者看起来只差一点，实际上不是一个层级。

`guardrail` 回答的是：

- 哪些事不能做
- 哪些情况必须失败
- 哪些偏离要立刻阻断

它很重要，但它只回答“哪里不能去”，不回答“系统应该往哪里去”。

如果长期方向没有先定义出来，那么 guardrail 再多，也只是局部阻断，不是真正意义上的收敛。

更糟的是，如果方向本身定义错了，那么控制越强，只会让错误状态更稳定。

这就是为什么我说，很多人一上来就谈 harness、guardrail、verification，其实已经在默认一个前提了：**方向问题已经解决。**

而在 AI 大规模开发里，这恰恰往往是最先没解决的问题。

## 三、什么是 Attractor

所以，必须先引入另一个词：`attractor`。

在动力系统里，attractor 指的是状态空间里的稳定区域。轨迹只要从足够近的地方出发，最终就会被它持续拉回，并长期停留在它附近。

把这个概念放到软件工程里，attractor 指的不是某个具体功能，也不是某个单独模块的完成态，而是：

**系统长期应该收敛到什么结构。**

注意，这里最容易犯的第一个错误，就是把 attractor 说成 boundary。

- `boundary` 回答的是：什么不能做
- `attractor` 回答的是：系统应向哪里长期收敛

边界是局部约束语言，吸引子是系统演化语言。

边界定义禁区，吸引子定义稳定区。

这不是同义词，也不是“更强边界”和“更弱边界”的关系。

## 四、Attractor 不是一张画好的终局蓝图

说到这里，很多人会马上进入第二个误解：

“那是不是先把终局蓝图画出来，再让 AI 照着执行？”

也不是。

这还是把问题想得太静态了。

在大型系统里，长期正确的状态通常没法被逐点枚举。你不可能提前写出所有合法终态，也不可能把每一步演化路径都计划好。

更真实的情况是：

- 人不是全局主宰
- AI 也不是全局主宰
- 双方都只是系统中的局部参与者
- 双方能做的，主要是施加局部信号

所以 attractor 更准确的理解，不是一个固定靶点，而是一个**由少量高阶规范隐式定义出来的稳定流形**。

这个说法听上去有点抽象，但意思其实很朴素：

- 不是枚举所有合法点
- 不是直接规定每一步该怎么走
- 而是给出一组真正抓住系统一阶结构的条件

这些条件会决定：

- 哪些状态彼此相容
- 哪些变化不会破坏整体结构
- 哪些局部修补会导致长期漂移
- 什么才算这个系统的稳定正确形态

如果借一个数学类比，这更像“代数方程定义代数流形”，而不是“枚举所有合法点”。

方程并不列出流形上的每一个点，但它定义了什么样的点可以长期存在。工程里的 attractor 也是这样。

## 五、那 Harness 到底是什么

如果 attractor 解决的是“方向是什么”，那么 harness 解决的就是：

**如何通过局部信号，持续地测量、纠偏、更新系统轨迹。**

这里说的 harness，不是狭义的测试 harness，而是更广义的执行支架。

它通常包括这些东西：

- 上下文路由
- 实现与验收分离
- 计划与关闭条件
- 验证机制
- 审计机制
- 诊断工具
- 外部化记忆

换句话说，harness engineering 关心的是：

- AI 该读什么，不该读什么
- 哪些结论能进入当前基线，哪些只能停留在探索层
- 什么算完成，谁来判定完成
- 漂移如何被尽早发现
- 同一个错误如何不在新 session 中重复出现

所以 harness 是什么？

它不是“几条测试规则”，也不是“再加点 lint”。

它是一整套让局部信号持续作用于系统轨迹的工程支架。

## 六、为什么必须是 Attractor Before Harness

到这里，标题其实就已经可以严格解释了。

`Attractor Before Harness` 不是一个修辞口号，而是在规定一个顺序：

1. 先定义系统应向哪里长期收敛
2. 再设计 harness，让局部实现和局部验证持续被拉回这个方向附近
3. 再把其中稳定、高频、可自动化的部分固化成 guardrail

这个顺序不能反。

因为 harness 能做的事再多，也默认“正确方向”已经先被定义出来了。

它可以：

- 让明显违规更早失败
- 让假完成更难通过
- 让审计独立于实现视角
- 让跨 session 的记忆得以保留

但它不能自动发明正确的 attractor。

如果 attractor 没有先被定义出来，那么 harness 只是在提高执行稳定性，而不是提高方向正确性。

如果 attractor 定义错了，那么 harness 越强，错误状态反而越稳定。

这就是为什么顺序必须是 `Attractor Before Harness`。

## 七、Guardrail 在这个框架里处于什么位置

一旦把 attractor 和 harness 区分开，guardrail 的位置也就清楚了。

它不是 attractor 的替代品，也不是 harness 的全部。它只是 harness 的一个子集。

更准确地说：

- attractor 定义长期正确结构
- harness 定义如何通过局部信号持续影响轨迹
- guardrail 定义哪些明显偏离必须立刻失败

所以 guardrail 更像什么？

- 测量器
- 势垒
- 早期失败条件

它回答的是“这一步有没有越线”，而不是“系统应向哪里收敛”。

因此，不能把更多 guardrail 误认为更强 attractor，也不能把 guardrail engineering 误认为 harness engineering 的全部。

## 八、Attractor 在工程里通常由什么定义

如果 attractor 不是一句口号，那它就必须有明确承载物。

在严肃的软件系统里，最接近 attractor 的通常不是测试，不是 lint 规则，也不是某个实现技巧，而是这些东西：

### 1. Owner doc

系统必须有明确的 owner doc，定义：

- 当前基线是什么
- 哪些不变量不能被局部便利覆盖
- 哪些边界属于顶层契约
- narrower 规则在冲突时服从谁

没有这层文档，系统方向就只能依赖隐性默契，而 AI 并不擅长维持隐性默契。

### 2. 架构不变量

真正构成 attractor 的，不是所有规范，而是那些抓住一阶结构的规范。比如：

- 哪些层可以依赖哪些层
- 什么属于 compile-time，什么属于 runtime
- 什么是作者可见的 effect path
- 哪些 contract 只能在 owner 层定义

这类不变量决定的不是某个局部功能怎么实现，而是系统长期能否维持清晰结构。

### 3. Precedence 结构

大系统不能只有一堆平铺的规范，还必须明确哪些文档拥有 precedence。

因为 attractor 不是“所有规范并列一起定义”的结果，而是需要一条清晰的层级链：

- 方向层
- 顶层规范层
- 局部 contract 层

没有 precedence，系统就会退化成多种规则并列共治，方向问题重新变得模糊。

## 九、Harness 在工程里通常由什么组成

如果 attractor 定义的是长期结构，那么 harness 解决的就是如何让实际轨迹持续贴着这个结构演化。

一个成熟的 harness，通常至少包括这几层。

### 1. 路由层

系统必须明确：

- 碰到哪类问题先读哪类材料
- 哪些材料属于 current baseline
- 哪些材料只是分析、探索、计划或历史记录

这本质上是在控制进入问题空间的入口。

### 2. Plan 层

系统必须明确：

- 当前这轮扩张的边界是什么
- 目标是什么，非目标是什么
- 哪些证据算完成
- 哪些 leftover work 必须显式外移

这决定的是局部轨迹如何合法闭合。

### 3. Verification 层

系统必须尽可能把高频、明确、可自动化的偏离检测下沉为机器检查，例如：

- typecheck
- build
- lint
- test
- check scripts

这一层的任务不是定义方向，而是尽早发现明显偏离。

### 4. Audit 层

并非所有偏离都能被自动化规则抓住。更高层的语义漂移、结构偏差、假完成、局部自洽但整体失真的问题，仍然需要独立审计。

因此，成熟的 harness 必须让评估视角尽量独立于实现视角，而不是让同一个上下文既写代码又裁决自己写得对不对。

### 5. Memory 层

AI 最容易重复犯错的地方，不是不会写，而是会在新的上下文里忘掉哪些路径已经被证伪。

所以系统必须把这些信息显式写进仓库：

- 为什么某种解释是错的
- 哪条路径已经证明会发散
- 哪个“已完成”判断后来被推翻
- 哪些术语必须避免被重新误译

这层不是 attractor 本身，但它决定 attractor 是否能在跨 session 里被稳定继承。

## 十、放到真实仓库里会是什么样

如果一个仓库采用文档驱动、owner-doc 驱动的方法，通常会出现这样的对应关系：

- `architecture` 文档承担 attractor 的主要定义
- `index`/router 文档承担阅读路由 harness
- `plans` 承担局部收口 harness
- `logs`、`bugs`、`discussions` 承担 memory harness
- `lint / check / test / audit / closure` 承担 verification 和 audit harness

这里最关键的一点是：

**不要把这些层写成并列机制。**

`architecture` 文档和 `lint` 规则都重要，但它们不在同一个层级上。

- 前者首先定义方向
- 后者根据这个方向构造失败条件

如果把它们拉平成“大家共同定义流形”，那么 attractor 的第一性位置就会被稀释掉。

## 十一、最常见的三种误解

### 1. 把 attractor 说成 boundary

这会把“长期收敛方向”降格成“局部禁区”。

### 2. 把 harness 说成 guardrail

这会把更广义的执行支架缩小成若干检查规则，导致人们误以为只要不断加规则就够了。

### 3. 把所有层都写成并列机制

例如说“类型、architecture、plan、verification、logs 一起定义流形”。

这种说法比直接说 boundary 更接近正确，但它仍然会模糊“谁先定义方向、谁再负责纠偏”这一关键顺序。

## 结语

AI 大规模开发真正困难的地方，不是让 AI 多写一点代码，而是让系统在高速扩张中仍然向正确结构收敛。

这就是为什么方法论必须从 attractor 开始，而不是从 harness 开始。

因为 harness 只能放大局部信号的作用，不能替代方向定义；guardrail 只能让明显偏离更早失败，不能先天地给出长期收敛目标。

所以真正需要被固定的，不只是更多检查，而是这条顺序本身：

1. 先定义 attractor
2. 再设计 harness
3. 再让 guardrail 服务于这套 harness

如果顺序被打乱，那么再强的工程支架，也可能只是在把系统更稳定地推向错误方向。
