# 代码注释策略

## 目的

本文解释 `nop-chaos-flux` 为什么默认几乎不写代码注释，以及这种选择在当前 AI 辅助、多 agent 协作开发模式下会产生什么效果。

这不是一篇泛泛而谈的代码风格文章，而是对仓库当前运行基线的说明。

## 当前基线

- 代码里默认不写注释。
- 优先依靠代码结构、命名、类型、测试、lint 规则、运行时保护和带外文档承载信息。
- 只有在反复调试、审计或回归修复后，确认某个局部约束很容易被误读或误改时，才补一条短注释。

这是一种有意选择，不是单纯遗漏。

## 为什么默认少注释

### 1. 大多数注释比代码更容易漂移

在当前工作流里，代码修改频繁、切片细、且经常由 AI 协助完成。一次编辑中最容易漏掉的，往往就是旁边那句解释性注释。

一旦代码和注释分叉，注释就会变成高置信度的错误信号。实践里，这通常比完全没有注释更糟。

### 2. AI 会把注释当作证据消费

当前 AI 编辑工作流并不会稳定地把注释视为“弱于代码”的信息。它往往会把注释、代码、测试和文档一起读入，再综合成一个自洽叙事。

如果注释已经过期，模型很可能会：

- 因为注释还在描述旧实现，而把过时逻辑继续保留下来
- 在冲突信号之间做错误折中
- 把过期解释继续复制进新代码、测试或文档
- 因为局部注释更近、更显眼，而错过真正的 owner doc 基线

在 AI 辅助开发里，过期注释不是被动噪声，而是会主动污染后续修改的输入。

### 3. Agent 的读取往往是局部和片段化的

很多 agent 任务只会读取一个文件片段、一条 grep 命中，或者少数几个相关文件。在这种工作方式下，局部注释很容易被过度放大，因为更大的架构上下文并没有同时加载进来。

这会让低质量注释变得尤其危险：

- agent 看到了注释，但没看到对应的 architecture doc
- agent 看到了注释，但没看到后来推翻它的回归测试
- agent 改了函数体，却没意识到旁边注释已经失真

减少注释数量，本质上是在减少这种局部上下文失真。

### 4. 宏观信息本来就不该主要写在代码里

这个仓库里很多重要事实并不是局部代码事实，而是跨文件、跨包、跨阶段的决策，例如：

- 为什么某个责任边界应该落在 `compiler` 而不是 `runtime`
- 为什么一个 plan 现在还不能关闭
- 为什么某个 host contract 必须保持窄合同
- 为什么一个 workaround 只是过渡接缝，而不是新基线

这类信息更适合由下面这些材料承载：

- `docs/architecture/`
- `docs/plans/`
- `docs/logs/`
- `docs/bugs/`
- `docs/discussions/`

把这些信息写进局部代码注释，本质上是在制造一份更弱、也更容易失真的第二现实。

## 为什么这种策略更适合当前 AI 生产方式

当前仓库工作流已经有比普通解释性注释更强的替代机制。

### 1. 代码结构本身就应承载局部语义

命名、模块边界、包 ownership、测试命名，本来就应该解释这段局部代码是什么、扮演什么角色。

如果一个局部行为必须靠一大段说明才能理解，通常应优先怀疑代码结构是否还不够清晰。

### 2. 测试比注释更能保护行为

如果一个约束可以被表达为行为，focused test 通常比注释更好。

因为测试会失败，注释不会。

对这个仓库来说，测试更适合固化：

- 并发、stale-run、supersede 语义
- 订阅式读取和快照式读取的边界
- dialog / drawer 打开态的时序行为
- owner boundary 和 handle resolution 的行为
- 历史回归修复不应再次静默出现

### 3. 机制化约束比自然语言更强

如果一条规则可以通过类型、schema 形状、lint 规则、运行时断言或结构化元数据表达，那就比自然语言注释更可靠。

优先级大致应是：

1. 可执行约束
2. 结构化元数据
3. focused test
4. 代码注释

### 4. 带外文档更适合跨 session 保留宏观上下文

AI session 是不连续的。真正能稳定留下来的，是仓库里的证据，而不是某个连续作者记忆。

这个仓库已经用带外文档来保留：

- 规范性基线
- 执行范围和 closure criteria
- 决策历史和下一步计划
- bug 的诊断路径和回归模式

这比把大段理由塞进代码注释，更适合当前 AI 工作模式。

## 什么样的注释仍然值得加

注释不是绝对禁止，而是只保留给那一小类即使有代码、测试和文档，局部陷阱仍然很容易反复出现的地方。

一条代码注释通常只有在以下条件同时满足时才值得加：

1. 问题是局部代码路径上的问题
2. 这个约束无法仅靠命名或结构自然表达清楚
3. 这条规则又不适合只靠类型、lint、运行时保护或测试单独表达
4. 反复调试、审计或回归修复已经证明同类误解出现过不止一次

典型例子：

- 一个 stale-run 分支看起来能被“简化”，但其实会吞掉真实失败 telemetry
- 一个同步/异步路径看起来能改成更“普通”的写法，但会重新引入时序回归
- 一个边界保护性 workaround 看起来多余，但删掉就会重新泄露 owner boundary 或 host coupling

在这些情况下，注释应该保持短、小、局部化，解释约束，不解释显而易见的机制。

## 什么样的注释通常是负资产

应避免那种只是把代码重新说一遍的注释。

典型低价值模式包括：

- 描述直白的赋值、条件或循环
- 用自然语言复述组件 JSX 结构
- 重复函数签名或类型已经表达的事实
- 把本应进入 architecture / plan / bug doc 的宏观理由塞进代码里
- 留下没有 owner plan、bug 或后续证据的 TODO 式备注

这类注释只增加文本，不增加可信信息。

## 实际判断规则

在写代码或 review 代码时，可以按这个顺序判断：

1. 这个约束能不能先通过结构、命名或 ownership 表达？优先这样做。
2. 能不能通过类型、lint、运行时保护或测试机械化固化？优先这样做。
3. 这个原因是不是本来就属于 architecture、plan、log、bug、discussion？那就放到带外文档。
4. 只有当它仍然是一个会反复出现的局部陷阱时，才补一条短注释。

压缩成一句话：

- 默认不写注释
- 把过期注释视为负资产
- 只有在反复证明确有局部误读风险时，才加一条短注释
