### 2026-04-13 (session 39)

- Fixed `detail-view` (and by extension any renderer using `parentForm.setValue`) not updating the viewer after the second confirm.
- Root cause (deeper than session 38's fix): `FormRuntime.setValue` and `setValues` in `packages/flux-runtime/src/form-runtime.ts` called `store.batchUpdate` without first calling `setLastChange`. `NodeRenderer` subscribes to `props.scope.store` and uses `store.getLastChange()` as its `getSnapshot`. When `setValue('summary', v1)` was called the first time, `lastChange` was a new object (different reference from the initial change), so `useSyncExternalStoreWithSelector` detected a snapshot change and re-rendered. On the second `setValue('summary', v2)`, `lastChange` was still the same object from the first call — same reference — so `useSyncExternalStoreWithSelector` saw no snapshot change and skipped the re-render.
- Fix: added `setLastChange({ paths: [name || '*'], sourceScopeId: formId, kind: 'update' })` before `store.batchUpdate` in both `setValue` (`form-runtime.ts:578`) and `setValues` (`form-runtime.ts:641`). This ensures every field write produces a fresh `lastChange` object, so `NodeRenderer`'s snapshot always changes and expression re-evaluation is triggered correctly.
- Also: `readOwn()` on the draft form scope returns `$form` meta (injected by `createReadonlyScopeBinding`). To avoid polluting the parent form's field value, `applyCommitResult` should strip the `$form` key before writing back. Added `$form` stripping to both `detail-view.tsx` and `detail-field.tsx`.

### 2026-04-13 (session 38)

- Fixed `detail-view` and `detail-field` confirm not updating the viewer on second (and subsequent) edits.
- Root cause: both renderers called `draftForm.scope.read()` to collect the values to write back. `draftForm` is created with `parentScope` as its parent, so `draftForm.scope.read()` returns `{ ...parentScopeData, ...draftOwnValues }` — a merge of all parent scope keys plus the draft form's own values. On the first confirm, `setValue('summary', mergedData)` accidentally stored a polluted object (containing the parent's `summary` key plus all draft fields at the top level). The viewer still showed correct data because `mergedData.title` happened to be the edited value. On the second confirm, the polluted `summary` object was read back as `scopeProjectedValue`, spread into `initialValues`, and merged again — causing the write-back to be identical to the previous confirm result, so `Object.is` equality held and the viewer never updated.
- Fix: replace `draftForm.scope.read()` with `draftForm.scope.readOwn()` in both `detail-view.tsx:147` and `detail-field.tsx:96`. `readOwn()` returns only the draft form's own store values, which is exactly what was seeded from the field's current value at open time — clean, with no parent scope contamination.
