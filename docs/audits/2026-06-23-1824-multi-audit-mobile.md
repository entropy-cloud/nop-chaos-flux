### [MM-07] notice-bar: `currentIndex` becomes stale when `text` shrinks to a single item → persistent blank bar

- **Files**: `packages/flux-renderers-mobile/src/notice-bar.tsx:61, 90-101, 131`
- **Dimension**: 06 / 14
- **Evidence**:
  ```ts
  // :61 — only writers are the initializer and the carousel timer
  const [currentIndex, setCurrentIndex] = React.useState<number>(0);
  // :90-101 — early-returns at length<=1; updater is monotonic modulo
  React.useEffect(() => {
    if (textList.length <= 1) return;
    if (!loop && currentIndex >= textList.length - 1) return;
    const id = setTimeout(() => {
      setCurrentIndex((idx) => {
        const next = idx + 1;
        if (loop) return next % textList.length;
        return next >= textList.length ? idx : next;
      });
    }, CAROUSEL_INTERVAL_MS);
    return () => clearTimeout(id);
  }, [textList.length, currentIndex, loop]);
  // :131 — out-of-bounds index silently falls back to ''
  const activeText = textList[currentIndex] ?? '';
  ```
- **Severity**: P3 (borderline P2)
- **现状**: When the host changes `text` from `['a','b','c']` (carousel already at index 2) to `['x']`, `textList` rebuilds to length 1, the carousel effect early-returns, but **no effect resets `currentIndex`** — it stays at 2. Render computes `textList[2] = undefined ?? '' = ''`; the `length === 0` guard (`:127-129`) does not fire (length is 1), so the DOM stays mounted showing a persistent blank bar.
- **风险**: Persistent blank (but layout-occupying) notice bar in a dynamic-data scenario (e.g. `text: ${activeNotices}` bound to a filtered array). No test covers the shrink path.
- **建议**: Clamp on textList change: `React.useEffect(() => { setCurrentIndex((idx) => (idx < textList.length ? idx : 0)); }, [textList.length]);` + a `['a','b','c'] → ['x']` regression test.
- **复核状态**: 维度复核 + 子项复核通过 — 保留 (P3, borderline P2). Highest-priority follow-up of the known set.

### [MM-14] countdown `paused` is ignored in `targetTime` mode — resume makes the display jump forward by the paused duration

- **Files**: `packages/flux-renderers-mobile/src/countdown.tsx:91-116` (tick body `:100-112`); contract `docs/components/countdown/design.md:97` ("`paused` 变为 `true` 时暂停计时器，`false` 时恢复")
- **Dimension**: 05 / 07
- **Evidence**:
  ```ts
  // :91-98 — effect clears the interval when paused (deps include paused)
  React.useEffect(() => {
    if (!started) return;
    if (paused) return;        // → effect returns, cleanup clears interval
    if (isFinished) return;
    ...
    const tick = () => {
      setRemaining((prev) => {
        if (typeof targetTime === 'number') {
          return Math.max(0, targetTime - Date.now());   // recomputes from wall clock on resume → JUMP
        }
        if (typeof time === 'number') {
          return Math.max(0, prev - interval);            // time mode unaffected (resumes from prev)
        }
        return prev;
      });
    };
    const timer = setInterval(tick, interval);
    return () => clearInterval(timer);
  }, [started, paused, isFinished, targetTime, time, interval]);
  ```
- **Severity**: **P2** (highest-priority finding of this audit)
- **现状**: While paused the interval is cleared (good), but `Date.now()` keeps advancing in real time. On resume the first tick recomputes `targetTime - Date.now()`, swallowing the entire pause window. e.g. `targetTime = now+60s`, pause 10s, resume → display jumps 60→50. The `time` branch uses `prev - interval` and correctly resumes. `design.md:97` explicitly promises pause/resume; `design.md:94` documents `剩余时间 = targetTime - Date.now()` for this branch. The existing "does not advance while paused" test (`countdown.test.tsx`) only covers `time` mode.
- **风险**: Pause/resume is a documented no-op-while-paused contract that silently fails in the more common `targetTime` (fixed-deadline) mode. Flash-sale / "pause during a dialog" patterns lose elapsed time. User-visible display jump.
- **建议**: On a `paused` transition capture `remainingAtPause`; on resume rebase `targetTime = Date.now() + remainingAtPause` (or unify both branches to elapsed-from-`Date.now()` with a `pausedAccumulatedMs` correction). Add a regression test: `targetTime + paused + advance + unpause → value unchanged`.
- **误报排除**: Not MM-08 (the `autoStart` reset effect at `:72-76`) — different code path (the tick body `:100-112`) and different observable (jump-on-resume vs reset-on-prop-change).
- **复核状态**: 维度复核 + 子项复核通过 — 保留 (P2). Reviewer: real correctness bug, documented-contract breach, affects the primary `targetTime` mode; not P1 (no crash, requires runtime `paused` toggle).

### [MM-16] infinite-scroll `immediateCheck` double-fires `onLoadMore` under React 19 StrictMode / remount

- **Files**: `packages/flux-renderers-mobile/src/infinite-scroll.tsx:59-62, 118-126`
- **Dimension**: 07 / 06
- **Evidence**:
  ```ts
  const isLoadingRef = React.useRef(false);
  React.useEffect(() => {
    // :60 — declared FIRST, runs on every setup
    isLoadingRef.current = false;
  }, [loading, error]);
  React.useEffect(() => {
    // :118 — no cleanup; deps incl fireLoadMore
    if (!immediateCheck) return;
    if (disabled || isFinished || loading === true || hasError) return;
    if (isLoadingRef.current) return; // MA-13 guard
    fireLoadMore(); // sets isLoadingRef.current = true
  }, [immediateCheck, disabled, isFinished, loading, hasError, fireLoadMore]);
  ```
- **Severity**: P3
- **现状**: Effects run in declaration order; neither has cleanup. Under React 19 StrictMode (setup → cleanup → setup) the `[loading,error]` effect re-runs on the second setup and resets `isLoadingRef.current=false` _before_ the immediateCheck effect re-runs, which then sees the guard clear and dispatches `onLoadMore` a second time. Both dispatches land before the host `loading` prop can dedupe. MA-13's own comment assumes a single mount. The infinite-scroll test wrapper is **not** wrapped in `<React.StrictMode>` (pull-refresh and swipe-cell wrappers are), so this is invisible to the 138-test suite (see MM-23).
- **风险**: Two `onLoadMore` calls per mount in dev StrictMode and on any production remount (key change / Suspense fallback rollback / list reordering) → duplicate records / double network requests for a non-idempotent host.
- **建议**: Make the guard robust to remount: track "already fired for this mount cycle" by comparing prev `loading`/`error` values (only reset the guard on actual _change_), not on every effect setup. Add a StrictMode test asserting `onLoadMore` called once on mount (will fail today).
- **误报排除**: Not the documented MA-13 behavior (which dedupes observer-vs-immediateCheck _within_ a single mount). This is a different race: between two complete effect setups the reset effect clobbers the guard.
- **复核状态**: 维度复核 + 子项复核通过 — 保留 (P3).
