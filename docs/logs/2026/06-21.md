### 2026-06-21 (Mission health fix — performance-table row memoization regression)

- **Mission driver `components-improvement` pre-flight health check uncovered a real regression** that had been documented as "pre-existing" across multiple recent plan closures (E2a/E2b/E2c/E2d in this same log). The `apps/playground/src/pages/performance-table-page.test.tsx > records a completed table single-row locality session` test was failing in BOTH happy-dom unit AND chromium e2e with `siblingProbeDelta.render` reporting `4` instead of the contract `0`. Confirmed it was a genuine row-isolation regression, not a happy-dom flake as previously assumed.
- **Root cause analysis (debug-instrumented `MemoizedDataRow` areEqual):** Two independent reference-instability bugs defeated the row `React.memo`:
  1. `computeCombinePlan` (`packages/flux-renderers-data/src/table-renderer/combine-cells.ts`) returned a fresh `rows.map(() => ({}))` array on every call whenever `combineNum` was unset (the common case). Since `paginateTableData` always rebuilds `processedData`, the `combinePlan` useMemo invalidated every render, breaking memoization for ALL rows. Caused `+2` of the `4` sibling renders.
  2. `RenderNodes` (`packages/flux-renderers-react/src/render-nodes.tsx:452`) computed `instancePath` without memoization. The `instancePath` reference cascaded through `nodeInstance` → `helpers` → `regions` → table-renderer `parentProps` → row memoization, breaking it for ALL rows when the table owner re-rendered due to a single-row scope update. Caused the remaining `+2`.
- **Fixes (both surgical, no contract change):**
  - `combine-cells.ts`: Introduced exported `EMPTY_COMBINE_PLAN` singleton; `computeCombinePlan` returns it for all no-merge paths (`combineNum` unset/zero, empty rows, virtual mode). Verified `getCellRowSpan` returns `undefined` either way — semantically equivalent, just stable-reference.
  - `render-nodes.tsx`: Wrapped the `instancePath` derivation in `useMemo` with the three nullable inputs as deps (`explicitInstancePath`, `ownerInstancePath`, `currentInstancePath`), preserving the existing `?? `-chaining semantics. The component already declares `'use no memo'`, so manual `useMemo` is the right tool.
- **Test updates:** `packages/flux-renderers-data/src/__tests__/table-e1b-enhancements.test.tsx` two existing assertions updated to expect `[]` instead of `[{}, {}]` for the no-merge paths (deep-equal; `getCellRowSpan` semantics unchanged). Added a new `returns a stable singleton empty plan when no merging is active` regression test that asserts reference identity (`toBe`) across calls and verifies `getCellRowSpan` returns `undefined` for both rowIndex 0 and 1 — locks the stability contract that is the whole point of the fix.
- **Verification (full workspace, 3 consecutive runs all green):** `pnpm typecheck` = 49/49、`pnpm build` = 26/26、`pnpm lint` = 26/26、`pnpm test` = 49/49 tasks (playground 19 files/88 tests, flux-renderers-data 47 files/417 tests, flux-renderers-form-advanced 77 files/730 tests, all other packages green). Repeated `pnpm test` three times — all three reported `Tasks: 49 successful, 49 total`. Targeted isolation runs of the previously-failing test: 5/5 passed.
- **Scope/autonomy note:** This is a cross-package fix (flux-renderers-data + flux-react). Per `docs/context/ai-autonomy-policy.md` the mission driver (`components-improvement` pre-flight health step) explicitly authorized issue-fixing as part of "ensure project is healthy before starting work" — no separate human gate needed for the fix itself. Did NOT trigger the AGENTS.md "unit+e2e full-green commit" workflow because e2e (Playwright) was not run by this mission; only `pnpm typecheck/build/test` were required.
