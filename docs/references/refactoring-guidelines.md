# Refactoring Guidelines

## Purpose

本文档总结 Flux 代码库的重构规范——何时重构、如何记录重构、以及在低代码系统中需要特别对待的设计决策。基于 2026-03-31 架构审计的经验整理。

---

## 1. 重构文档规范

每次重构必须留下可追溯的记录。以下是必需的文档模板和规范。

### 1.1 重构计划模板

在 `docs/plans/` 下创建计划文件，文件名格式 `NN-short-description-plan.md`（NN 为递增编号）。

```markdown
# [简短标题]

> **Status: 📋 PLANNED / 🔧 IN PROGRESS / ✅ DONE**

## Problem

[用 2-3 句话描述问题。必须包含具体的文件路径和行号。]

## Root Cause

[为什么会变成这样？不是"代码写得不好"，而是系统性的原因。]

## Fix Plan

**Step 1 — [动作]**
[具体描述，包含目标文件]

**Step 2 — [动作]**
[具体描述]

**Scope**: [明确写出涉及的文件/包。不在 scope 内的一律不动。]
**Effort**: [真实的时间估算]
**Verification**: `pnpm typecheck && pnpm build && pnpm test`

## 变动文件清单

| File | Change | Lines affected |
|------|--------|---------------|
| `packages/xxx/src/yyy.ts` | 提取辅助函数 | ~30 |
| `packages/xxx/src/zzz.ts` | 调用方更新 | ~5 |

## 风险与回退

[如果重构可能引发问题，说明回退策略]
```

### 1.2 必填字段说明

| 字段 | 作用 | 常见错误 |
|------|------|---------|
| **Problem** | 让未来读者理解"为什么做" | 只写"文件太大"而不解释为什么大是问题 |
| **Root Cause** | 防止治标不治本 | 写成"代码质量差"而非系统性原因 |
| **Scope** | 防止范围蔓延 | 省略 scope 导致顺手改了不相关的代码 |
| **变动文件清单** | 代码审查的快速参考 | 计划阶段不填，事后补不全 |
| **Effort** | 排期依据 | 凭感觉估算，不读代码就填数字 |

### 1.3 重构后必做

完成重构后：

1. 在 `docs/development-log.md` 添加条目，链接到计划文件
2. 如果涉及包边界或公共 API 变更，更新对应的 `docs/architecture/*.md`
3. 运行完整验证：`pnpm typecheck && pnpm build && pnpm lint && pnpm test`
4. 如果重构过程中发现了新的问题，**创建单独的计划文件**，不要顺手修

## 2. 原因分析框架

发现代码问题时，用以下框架分析根因，避免"看着不爽就重构"。

### 2.1 常见根因分类

| 根因 | 典型表现 | 正确的修复 |
|------|---------|-----------|
| **增量开发无提取** | 方法逐个添加，共享模板未提取 | 提取共用辅助函数 |
| **文档与代码脱节** | 文档描述的角色与实际代码不符 | 更新文档（不要改代码去匹配过时的文档） |
| **跨场景复制** | 两个开发者独立实现相同模式 | 统一为一个实现，保留各自的入口 |
| **历史遗留** | 早期决策不再适用但有约束 | 评估迁移成本，可能只是加注释 |
| **合理的权衡** | 看似不理想但有意为之 | 不需要修复，需要文档说明 |

### 2.2 判断是否值得重构

在做重构前，回答以下问题：

1. **当前代码是否能正常工作？** — 如果能，重构的紧迫性自动降级。
2. **修改频率高吗？** — 如果这段代码很少被触碰，重构的 ROI 低。
3. **重构会引入新风险吗？** — 如果涉及公共 API 或多个包，风险较高。
4. **是否只是审美偏好？** — "代码不好看"不是充分理由。

## 3. 重构策略

### 3.1 安全重构流程

```
分析 → 计划 → 创建新代码 → 验证新代码 → 替换旧代码 → 删除旧代码
```

**关键原则：永远不在修改旧代码的同时写新代码。**

具体步骤：

1. **分析阶段**：通读要重构的文件，列出所有函数和它们的调用关系
2. **计划阶段**：确定提取边界，写变动文件清单
3. **创建新代码**：在独立文件中写新的辅助函数/模块，保留旧代码不动
4. **验证新代码**：在新文件上运行 typecheck + test
5. **替换调用方**：逐个修改旧代码的调用方，指向新实现
6. **删除旧代码**：确认所有调用方迁移后，删除旧的实现
7. **最终验证**：运行完整的 `pnpm typecheck && pnpm build && pnpm test`

### 3.2 按复杂度选择策略

| 复杂度 | 场景 | 策略 | 示例 |
|--------|------|------|------|
| **简单** | 单文件内提取辅助函数 | 直接提取，无需备份 | `setValidating` 改用已有 `setBooleanState` |
| **中等** | 跨文件的模块拆分 | 先创建新模块，验证后替换 | `types.ts` 拆分为 `types/` 目录 |
| **复杂** | 跨包的重构或 API 变更 | 先创建 `.bak`，逐层替换 | `schema-compiler.ts` 拆分为子模块 |

### 3.3 复杂重构的 .bak 方法

对于 200+ 行文件的结构性拆分：

1. 将原文件重命名为 `.bak`（如 `FlowDesignerExample.tsx → FlowDesignerExample.tsx.bak`）
2. 创建新的拆分文件和新的 orchestrator
3. 逐步验证每个拆分出的模块
4. 全量通过后删除 `.bak`
5. 如果出问题，`.bak` 提供即时回退

## 4. 低代码系统的特殊考量

Flux 是一个**低代码渲染系统**，这决定了某些在常规应用中适用的设计原则需要调整。

### 4.1 类型系统与动态 Schema

**核心矛盾**：TypeScript 的类型系统在编译时工作，但低代码系统的数据结构在运行时由用户定义的 JSON schema 决定。

#### `Record<string, any>` 是诚实的

在低代码系统中，form values、scope data、schema props 的类型在编译时不可知。`Record<string, any>` 不是"类型安全缺陷"，而是**最诚实的类型表达**。

**可以改善的**：为高频使用的 selector/hook 提供泛型参数，让调用方在已知类型时获得类型推断。

```typescript
// ✅ 推荐：为调用方提供泛型入口
useScopeSelector<T, S = Record<string, unknown>>(selector: (data: S) => T): T

// ⛔ 不推荐：将 any 改为 unknown，只是把问题推到调用方
FormStoreState.values: Record<string, unknown>
// 结果：每个使用 values 的地方都需要 `as MyFormValues`，等于藏了一堆 any
```

**不应做的**：为 `SchemaPath`、`ComponentId` 等添加 branded types。这些值全部来自 JSON 解析，入口点就是 `string`，branded types 只会导致到处都是 `as SchemaPath` 强转。

#### 泛型参数要向后兼容

添加泛型参数时，始终提供默认值：

```typescript
// ✅ 向后兼容
interface ScopeStore<T = Record<string, any>> { ... }

// ⛔ 破坏性变更
interface ScopeStore<T> { ... }
```

### 4.2 提取 vs 内联

在低代码渲染器中，某些看似重复的代码实际上是**有意的分支**，不应盲目提取。

**应该提取的**：
- 完全相同的逻辑块（如 `validateForm` 中出现 3 次的子验证循环）
- 已有辅助函数但未使用的情况（如 `setValidating` 应该调用已有的 `setBooleanState`）
- 明确的模板模式（如 7 个数组操作方法共享的 4 步模板）

**不应提取的**：
- "结构相似"但领域语义不同的代码（如 action debounce 和 validation debounce——它们碰巧都用 setTimeout，但关注点不同）
- 只出现 2 次的"重复"（提取后的辅助函数名可能比原代码还长）

**判断标准**：如果提取后的代码比原来的调用方代码更复杂，就不要提取。

### 4.3 文件拆分的代价

大文件确实增加认知负担，但拆分本身也有代价：

| 拆分收益 | 拆分代价 |
|---------|---------|
| 单个文件更易理解 | 文件间跳转增加 |
| 可独立测试 | import 关系更复杂 |
| 职责清晰 | barrel 文件（index.ts）维护 |
| Code review 更聚焦 | 重命名/移动更困难 |

**拆分阈值建议**：
- **< 200 行**：不需要拆分
- **200-500 行**：如果职责单一，可以不拆；如果混合了 3+ 个关注点，考虑拆分
- **500-900 行**：应该拆分，但要评估是否近期还会频繁修改
- **> 900 行**：需要拆分，优先拆分出独立的子域

### 4.4 包边界

Flux 的包依赖链是单向的：

```
flux-core → flux-formula → flux-runtime → flux-react → flux-renderers-*
```

**代码放置原则**：

1. 如果一个函数被 `flux-formula` 需要，它只能放在 `flux-core`（最底层共享包）
2. `flux-core` 可以包含纯函数工具（如 `getIn`、`shallowEqual`），只要它们无副作用、无外部依赖
3. 不要因为"flux-core 应该只包含类型"而把共享工具函数移到依赖链上层的包——这会创建反向依赖

**判断方法**：如果一个函数被 2 个以上的包使用，且它是无副作用的纯函数，它可以放在 `flux-core`。

### 4.5 性能相关的重构

在低代码渲染器中，性能敏感的路径要格外小心：

1. **React Context Provider 不要合并**——多个独立 Provider 提供更细粒度的更新控制。合并为单一 Context 后，任一值变化会触发所有消费者重新渲染。
2. **不要过早优化 `Date.now()` 调用**——除非 profiling 确认有性能问题。条件判断本身也有成本。
3. **保持对象引用稳定**——这是 Flux 相比 AMIS 的核心优化之一。重构时确保不破坏对象复用。

## 5. 经验教训

### 5.1 审计时容易犯的错误

| 错误 | 表现 | 正确做法 |
|------|------|---------|
| **计数不准** | "43 处问题"实际是 47 处 | 审计时直接运行 `grep -c`，不靠目测 |
| **行号过时** | 引用的行号与实际差几行 | 审计结果要标注审计日期，读者自行验证 |
| **描述与现状不符** | "4 个重复方法"实际已有 3 个被统一 | 审计前先读代码，不要只靠历史记忆 |
| **ROI 评估缺失** | 所有发现都列为"需修复" | 区分"必须修"、"值得修"、"不需要修" |
| **优先级反转** | 高优先级项实际工作量小于低优先级 | 按工作量 × 收益排序，不只按严重程度 |

### 5.2 重构中的常见陷阱

1. **顺手重构**：修 bug 时"顺便"重构了不相关的代码。正确做法：bug fix 和 refactor 分开提交。
2. **过度抽象**：为 2 处相似代码创建泛型容器。正确做法：至少出现 3 次再考虑提取。
3. **追赶理想状态**：把 `any` 改为 `unknown` 只是把问题移动了位置。正确做法：只在调用方能真正提供类型信息时才改善类型。
4. **忽略测试**：重构后没有运行测试就标记完成。正确做法：每次逻辑变更后运行受影响的测试。

## 6. 检查清单

重构前快速过一遍：

- [ ] Problem 描述包含具体的文件路径和行号
- [ ] Root Cause 是系统性的，不是"代码不好"
- [ ] Scope 明确列出涉及的文件，不在 scope 内的不动
- [ ] 如果涉及类型系统变更，考虑了低代码的动态 schema 特性
- [ ] 如果是文件拆分，评估了拆分的收益和代价
- [ ] 如果是跨包变更，检查了依赖方向
- [ ] Effort 估算是基于代码阅读的，不是凭感觉
- [ ] 有回退计划
