## 24. Picker CRUD 模式初始显示编号，选择后才显示名称

- **所属组件**: Picker (flux-renderers-form-advanced)
- **问题描述**: standard-crud 编辑表单中，picker 初始显示部门 ID（如 `"d1-1"`）而非名称（`"研发中心"`），但弹出选择窗口并确认后正确显示名称。根因：表单值为原始 ID（`departmentId: "d1-1"`），picker 的 `selectedLabel` 解析链为 `resolvedLabelCache → optionLabelMap → resolveSelectedLabel()`。CRUD 模式下 `options` 为空且 `labelResolveAction` 未配置，`resolvedLabelCache` 在初始 mount 时为空，`resolveSelectedLabel` 找不到匹配项则回退到原始值。选择确认后 `confirmCrudSelection` 调用 `cacheLabelsForValues` 填充 `resolvedLabelCache`，故第二次显示正确。对比 amis 行为：amis `OptionsControl` 在 form init 时自动调用 `fetchOptions()`（通过 `formItem.addInitHook`）加载 source API 数据到选项缓存，因此即使未打开弹窗，label 也可在挂载后自动解析。
- **预期行为**: picker 应至少提供一种方式在初始 mount 时自动解析 label（无需打开弹窗），与 amis 一致。
- **发现方式**: 人工测试（`http://localhost:5173/#/complex-pages/standard-crud`，编辑用户查看"部门（弹出选择）"字段）
- **状态**: 已修复（picker 代码中 `labelResolveAction` useEffect 增加 `loadAction` 后备：`const resolver = labelResolveAction ?? loadAction`。CRUD 模式下未配置 `labelResolveAction` 时自动用 `loadAction` 加载数据并解析 label，等价于 amis `fetchOptions()`）
  - 补充：`normalizeFieldValues` 新增 `valueKey` 参数支持对象值提取（如 `{id: "d1-1", name: "研发中心"}`），`selectedLabel` 优先从 `rawFieldValue[labelKey]` 直接读 label，无需远程加载）

## 34. 未配置 polling 的 CRUD 触发 false positive 告警

- **所属组件**: CRUD polling (flux-renderers-data)
- **问题描述**: 所有未显式配置 `polling` 的 CRUD（如 `advanced-query` 页面的 User CRUD）均触发控制台告警："[crud polling] polling.enabled is true but no upstream data-source was found; polling is disabled"。根因：`use-crud-polling.ts:99` 的 `schemaEnabled = enabledSpec !== false && enabledSpec !== 'false'`。当 `polling` 为 `undefined` 时 `enabledSpec` 为 `undefined`，`undefined !== false` 为 `true`，导致 `schemaEnabled` 默认启用。`userToggle` 默认为 `true`，因此 `effectiveEnabled = true`，polling effect 运行并尝试查找 data-source handle 失败后告警。
- **预期行为**: 未配置 `polling` 的 CRUD 不应触发任何 polling 相关告警。
- **发现方式**: 人工测试（浏览任意 CRUD 页面如 `http://localhost:5173/#/complex-pages/advanced-query`，观察控制台）
- **状态**: 已修复（`schemaEnabled` 改为 `polling ? enabledSpec !== false && enabledSpec !== 'false' : false`）

## 35. Picker CRUD 模式翻页失效（`pageField`/`pageSizeField` 为死配置，CRUD 不自动注入分页参数）

- **所属组件**: CRUD (flux-renderers-data) — 框架级缺陷，Picker 为受影响方
- **问题描述**: standard-crud 编辑/新增表单中弹出 picker 选择（部门/标签），分页器正确显示"1/4"（总页数计算正确），但表格展示的是全部数据，点击"第 2 页"或"下一页"无实际翻页效果。CRUD schema 支持 `pageField: 'page'` 和 `pageSizeField: 'perPage'` 配置，`createCrudEvaluationBindings` 也将 `pagination.currentPage`/`pagination.pageSize` 注入到了 AJAX dispatch 的 `evaluationBindings` 中，但 **没有任何运行时代码读取 `pageField`/`pageSizeField` 来将分页参数自动注入 AJAX 请求**。开发者必须在 `loadAction.args.params` 中手动编写 `${pagination.currentPage}` 等表达式，而这些表达式在 dialog form scope 编译时不满足可见性条件（bug #69），导致 `openDialog` 静默失败。`serverPaged` 模式在 CRUD 侧仅影响 `total` 的来源和分页器 display，没有配套的请求参数自动注入能力。
- **预期行为**: CRUD 配置 `serverPaged: true` 且设置了 `loadAction` 时，`pageField`/`pageSizeField` 应在 dispatch 时自动将当前 `pagination` 值注入 AJAX URL params（无需 schema 手动编写表达式）。Picker 使用 CRUD 模式时应即开即用。
- **发现方式**: 人工测试（`http://localhost:5173/#/complex-pages/standard-crud`，点击编辑/新增 → 部门 picker 或标签 picker）
- **状态**: 待修复（框架功能缺失：`useCrudLoadAction` 或 AJAX adapter 需根据 `pageField`/`pageSizeField` 自动拼接分页参数到请求 URL）
