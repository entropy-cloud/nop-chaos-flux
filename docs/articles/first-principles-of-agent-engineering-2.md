# 在约束中建造：AI Agent 工程的根本原则

做 Agent 的团队，几乎都会在同一个路口跌倒。

起初你觉得这很简单：让 AI 读项目文档，给它任务，让它写代码、跑测试、提 PR。头几天效果惊人。然后怪事开始发生——AI 反复提出已经被否决的方案；在不同会话里对同一件事给出矛盾判断；制造出文件齐全、测试全绿、总结流畅，但关键功能根本不 work 的 PR。你开始加规则：更多文档、更详细的提示词、更多 checklist、更多审计。系统变重了，没有变好。有时更差——审计发现了更多“问题”，修复引入新问题，系统在自己制造的噪声里膨胀。

这不是某个模块写错了。这是在和几条基本约束较劲，而这些约束不是“加更多东西”能绕开的。

要找到那几条约束，得先问一个更靠前的问题：

**到底是什么变了，让过去几十年好用的软件工程方法突然失灵？**

---

## 一、真正的变化：吞吐差改变了人的位置

答案不在 LLM 这个组件的某条属性里，而在系统的控制架构里。

人写代码，一天几十行、几百行。AI 写代码，一天可以上万行。这个吞吐差不是“快了一点”的量变，它直接改变了人在系统里的位置。

过去几十年，软件工程的方向感很大程度上靠**人脑里的隐式蓝图**控制。架构师、资深工程师、模块 owner 在 loop 之内，每一个关键改动都流经他的判断。项目“该长成什么样”这件事，不需要完全写下来，因为它活在那个 in-the-loop 的人脑子里，随时校准每一步。

低吞吐时代，人就是项目的隐式控制器。

AI 改变的不是单步能力，而是生成速度。当 AI 的生成速度远超人类的消化速度，人再也无法逐行、逐文件、逐 PR 地直接控制所有变化。你只有两个选择：

- 主动给 AI 降速，让它适配人的审查带宽；
- 或者承认人必须从低层执行 loop 里退出来，把控制粒度上移。

更准确地说，人不是彻底消失在 loop 外，而是从**微观执行 loop**迁移到**高层控制 loop**：

```text
过去：读需求 → 写代码 → 测试 → 修改 → review → 合并

现在：定义方向 → 设定约束 → 批准计划 → 检查证据 → 裁决异常 → 更新方向
```

人退出的是逐行执行，不是最终控制。

这个位置转移，是后面一切的源头。它比“LLM 会幻觉”“LLM 上下文有限”这些事实更根本。因为即使模型更强，只要吞吐差存在，人就不可能继续用旧方式控制每个细节。项目方向必须被显式外置，验证必须被工程化，经验必须被复用，修正必须形成闭环。

Agent 工程的第一原则因此不是“让 AI 更聪明”，而是：

> 在高吞吐条件下，重建人类从低层 loop 退出后留下的控制结构。

---

## 二、人退到高层之后，四件事裸露出来

人不再逐步兜底，过去由人脑隐式承担的东西就暴露成了工程问题。

LLM 有三个事实：

1. **它不会自动跨会话记住。**
2. **它每一次思考都有成本。**
3. **它的错误不是无方向随机噪声，而是有方向地向训练分布平均解漂移。**

这三个事实其实一直存在。为什么过去的软件工程不需要专门对付它们？因为过去在环内的人，就是这三件事的通用补丁：

- AI 会忘 → 人把上下文补回来；
- AI 会重复思考 → 人凭经验跳过无效路径；
- AI 会跑偏 → 人随手把方向拉回来。

低吞吐时代，人脑承担了跨会话记忆、经验缓存、方向判断和纠偏。高吞吐把人从低层 loop 推走后，这些能力不能再隐含在某个人脑子里，必须显式重建。

于是 Agent 工程可以概括成四件事：

- **维持**：让认知跨时间存在；
- **复用**：把历史计算转成未来能力；
- **验证**：判断输出是否偏离；
- **修正**：把偏离拉回正确方向。

这四件事不是装饰，而是 Agent 系统的骨架。缺了维持，系统失忆；缺了复用，系统反复付同一笔推理成本；缺了验证，系统会制造完成假象；缺了修正，系统只能看着漂移发生。

但验证和修正还预设一个东西：你得知道“正确方向”在哪里。这个东西后面会叫作**吸引子**。

---

## 三、维持：真相必须外置

LLM 不携带可靠的跨会话记忆。但比“AI 会忘”更严重的是：当人退出低层执行 loop 后，项目不再天然有一个主体持有完整蓝图。

过去仓库乱一点，文档旧一点，注释少一点，问题还不致命，因为总有资深工程师脑子里装着真图。现在 AI 以高吞吐修改项目，那个脑内蓝图跟不上变化。如果某个关系、某个边界、某个历史否决没有被外部结构承载，它就等于不存在。

所以“仓库即真相”的含义变了。

它不只是“信息放仓库里方便查”，而是：

> 仓库及其附属结构，是项目蓝图唯一可持续存在的形式。

真相必须外置。外置的真相至少有四个维度：

1. **事实本身**  
   当前代码、接口、数据结构、配置、依赖、运行行为。

2. **因果来源**  
   为什么变成这样，谁做的决策，替代了什么旧结构。

3. **置信状态**  
   哪些是确定事实，哪些是推测，哪些来源互相冲突。

4. **否定空间**  
   哪些方案被否决，哪些假设被证伪，哪些 bug 路径已排查过。

其中最容易丢的是否定空间，也最容易导致 AI 重复犯错。AI 会反复提出早被否决的方案，不是因为它“调皮”，而是因为系统没有把“这条路不能走”作为真相保存下来。

### 外置真相首先是人重夺态势感知的接口

外置真相不是只给 AI 读的。更重要的是，它是人退到高层 loop 后重夺态势感知的接口。

人不可能读完所有 AI 生成的代码，但人必须知道：

- 当前系统在往哪里走；
- 哪些边界被碰过；
- 哪些规则被破坏；
- 哪些例外是有意的；
- 哪些方向正在漂移。

所以人读的不是全部细节，而是吸引子、owner doc、ADR、审计结论、风险证据和否定空间。外置真相是人重新监督系统的观测窗。

没有这个观测窗，人虽然名义上在高层 loop，实际上已经失去控制。

---

## 四、RAG 不是记忆，向量检索不是真相

很多团队意识到“AI 会忘”之后，第一反应是上 RAG：把文档、issue、PR、代码片段切块，塞进向量库，任务来时 top-k 召回。

这有用，但远远不够。

向量检索解决的是“找语义相似文本”，不是“判断什么当前生效”。而 Agent 需要的恰恰是后者。

向量检索天然回答不了这些问题：

- 哪条规则现在有效？
- 哪份文档权威更高？
- 哪个 ADR 覆盖了哪个旧决策？
- 这段话是事实、猜测、草案，还是已废弃结论？
- 某个方案是未尝试、已否决，还是暂缓？
- 两个来源冲突时谁说了算？

这些不是相似性问题，而是**权威、版本、状态、作用域、因果**问题。

向量库最常见的失败模式有三种。

### 第一，召回相似但已失效的信息

旧设计文档往往和当前任务高度相似，所以很容易被召回。但它可能已经被新 ADR 推翻，或者只适用于旧版本。

如果检索层没有 status、version、superseded_by、scope，Agent 就会把历史遗迹当成当前真相。

最危险的是：旧文档通常写得很正式，很像权威。

### 第二，召回正例，漏掉否定空间

用户问：“能不能用 Redis 做这个缓存？”

向量库可能召回 Redis 实现、缓存文档、性能讨论，却漏掉一条关键记录：“不用 Redis，因为多租户隔离模型下 key namespace 无法满足审计要求。”

否定空间在语义上常常不像任务本身，所以不稳定依赖 top-k 召回。它必须结构化保存，例如：

```yaml
rejected_options:
  - option: 'shared Redis cache for tenant permissions'
    status: active
    rejected_by: 'ADR-021'
    reason: 'breaks tenant-local audit replay'
    scope: 'authz'
```

### 第三，向量相似性表达不了拓扑

向量空间里的“相近”，不是架构上的“可连接”。

两个概念语义相似，不代表它们应该放在一起；两个模块经常共同出现，不代表它们可以互相依赖。拓扑关系——概念边界、依赖方向、互斥关系——不能交给 embedding 隐式表达。

因此成熟的记忆系统不能是：

```text
query → vector top-k → 塞进 prompt
```

而应该是：

```text
任务识别
→ 选择信息源
→ 权威和状态过滤
→ 符号检索 + 结构检索 + 向量检索
→ 冲突裁决
→ 按预算装配上下文
```

向量库是检索器，不是真相机制。embedding 是相似性索引，不是项目记忆本身。

## 五、分离当前权威与轨迹记录

真相外置以后，新的危险不是信息太少，而是所有文本被当成同等真相。

仓库里会积累 owner doc、ADR、analysis、plan、log、discussion、audit、PR 讨论、bug note。它们都包含信息，但它们在系统里的效力完全不同。

同一句话，出现在不同地方，含义不一样。

出现在 owner doc 里，它可能是当前基线。  
出现在 active ADR 里，它可能是当前决策。  
出现在 analysis 里，它只是候选判断。  
出现在 plan 里，它是本轮收口契约。  
出现在 log 里，它是轨迹记录。  
出现在 audit 初稿里，它只是待复核信号。  
出现在旧 plan 里，它可能只是一个后来被推翻的行动假设。

如果 Agent 不能区分这些效力，它不会“综合上下文”，而是把当前规则、历史轨迹、旧假设、未复核审计和候选方案平均成一个不存在的现实。

这就是为什么外置真相不能只是文本库，而必须是带状态的权威链。

至少要区分几类效力：

```text
当前基线：owner doc、active ADR、接口契约、模块边界
当前否定空间：active rejected options、已证伪假设
局部收口契约：active plan、exit criteria、closure evidence
候选信号：analysis、audit finding、spike result
历史轨迹：completed plan、log、discussion、PR/issue 记录
可执行约束：test、lint、type、schema、runtime guard
临时推理：scratchpad、session notes
```

这些东西不是“高层文档”和“低层文档”的区别，而是效力不同。

owner doc 的本质不是“慢变量文档”，而是当前吸引子的权威投影。它回答：现在这个模块如何理解，边界在哪里，依赖方向是什么，哪些结构不再合法。

plan 的本质也不是普通历史记录。一个 active plan 是局部收敛器：它定义 current baseline、goals、non-goals、exit criteria 和 closure evidence。它在本轮任务内有约束力。但 plan 完成之后，就转化为轨迹证据；如果其中产生了长期约束，必须经过复核和裁决，升格为 owner doc、ADR、rejected option、test、lint 或 guard。

log 不是收尾总结。它保存轨迹，尤其保存“为什么不这么做”。没有 log，后续 Agent 会反复重走已经失败的路线。

discussion 不是聊天归档。它保存概念澄清过程：哪些表达被修正，哪些误解被排除，哪些前提被证伪。

audit 也不能直接变成 backlog。审计本身会产噪声。初审发现只是候选信号，必须经过独立复核和 re-verification，才能决定是进入 plan、更新 owner doc，还是判定为非问题。

所以历史不能直接投票。候选不能直接裁决。计划不能自动升格。审计不能未经复核就变成待办。

外置真相需要一条效力转换链：

```text
discussion / log / audit / incident / spike
        ↓ 产生候选信号
analysis
        ↓ 复核、裁决、定 scope
plan
        ↓ 实现、验证、closure audit
owner doc / ADR / rejected option / test / lint / guard
        ↓ 成为下一轮基线的一部分
```

这条链路的关键是：只有经过收敛的结论，才有资格进入当前基线。

因此，每条持久化信息至少要带上：

```yaml
id: ADR-021
type: architecture_decision
status: active
authority: current_baseline
owner: security-platform
scope:
  - services/authz
supersedes:
  - ADR-009
```

或者：

```yaml
id: PLAN-2026-04-18-authz-cache
type: execution_plan
status: completed
authority: historical_trace
outcome: redis-option-rejected
related_to:
  - ADR-021
```

前者是当前仍然有效的约束。  
后者是一次任务留下的轨迹证据。  
它可以解释约束从哪里来，但不能替代约束本身。

这也改变了上下文装配方式。成熟的 Agent 系统不应该是：

```text
query → 找相似文本 → 塞进 prompt
```

而应该是：

```text
query
→ 识别任务作用域
→ 先读取当前基线
→ 再召回相关历史证据
→ 标注每条信息的效力
→ 过滤过期和未复核材料
→ 装配上下文
```

没有状态、来源、作用域和权威等级的信息，不是真相，只是文本。

所以这里真正的问题不是“快变量放哪里，慢变量放哪里”。变化速度只是副属性。关键问题是：

```text
什么现在有效？
什么只是候选？
什么是本轮收口条件？
什么是历史轨迹？
什么已经被推翻？
什么能约束未来？
什么必须先经过裁决才能升格？
```

Agent 需要历史，因为没有历史就会重复犯错。  
Agent 更需要权威，因为没有权威就会把所有历史平均成幻觉。

外置真相的目标不是保存更多文本，而是让每条文本带着正确效力进入闭环。否则记忆越丰富，系统越容易被自己的历史残影带偏。

## 六、复用：思考有成本

任何任务内在的推理步数都有下界，没法靠“请认真思考”压成一步。

一条复杂推理路径走过，算力已经花掉了。后续系统能做的，是把这条路径的结果沉淀下来，避免每次重复支付。

这就是 Skill 的意义。

Skill 不是“提示词片段”，而是过去不可压缩计算的固化。它记录一类任务的局部最优路径：什么时候触发、该查什么、按什么顺序做、哪些坑不要踩、如何验证。

但 Skill 有结构性盲区：它本质上是列表。

列表能保存路径，保存不了地图。凡是带内在关系结构的领域知识，不能拍平成一条条 Skill。比如概念边界、依赖方向、领域对象之间的本体关系，这些是拓扑，不是经验片段。把它们塞进 Skill 列表，相当于把最贵的结构理解主动打散，让后来的 Agent 每次从散点重新反推全局。

所以：

- 局部操作路径 → 可以进 Skill；
- 项目结构不变量 → 应进 owner doc / ADR；
- 具体当前依赖图 → 应实时从代码重建；
- 临时任务判断 → 留在当前会话。

Skill 也必须可版本化、可淘汰。没有淘汰能力的 Skill 库不是资产，而是不断累积的债务。旧路径一旦失效，却继续被 Agent 召回，会比没有 Skill 更危险。

## 七、上下文是预算，不是容器

上下文窗口不是装信息的桶，而是一次推理的计算预算。

一个 50K token 的上下文不是“装了 50K 信息”，而是“50K token 共同参与一次注意力计算”。无关信息、冲突信息、过期信息都会消耗预算，甚至改变模型的判断方向。

两个后果很重要。

### 第一，冲突信息的成本远高于缺失信息

“用 A，但有时也用 B”比“用 A”贵很多。前者强迫模型每次判断“现在算不算那个有时”。如果例外没有明确条件，它就会变成持续噪声。

写给 Agent 的规则要追求确定性，不是追求百科全书式完整性。

### 第二，相关性比完备性重要

AGENTS.md 动辄上千行并全部常驻，是把上下文当仓库，而不是当预算。Agent 进入任务时，应在前几百 token 内拿到当前任务最关键的决策约束。

真正决定推理质量的，不只是 prompt，而是上下文装配器。它要判断：

- 当前任务是什么类型；
- 需要哪些 owner doc；
- 哪些 ADR 当前生效；
- 哪些代码事实必须读取；
- 哪些否定空间相关；
- 哪些信息过期；
- 哪些冲突需要先裁决；
- 哪些内容不该塞进上下文。

好的上下文不是“多”，而是“准”。

---

## 八、验证：错误有方向

AI 的输出不是无偏随机扰动。它系统性地朝训练分布里的平均解漂移：主流方案、常见写法、它见过最多次的模式。

这个漂移有方向。

所以多采样不能解决根本问题。偏置同向时，采样十次会一致地错向同一方向。尤其是让同一个模型、同一个 prompt、同一个上下文做“实现”和“验证”，很容易变成自我确认。

有效验证必须引入异质性：

- 可执行测试；
- 类型系统；
- lint；
- 静态分析；
- runtime guard；
- 独立 session；
- 不同模型；
- 人类裁决；
- 真实环境里的端到端路径。

验证者不能和生成者共享同一个偏置场。

### 完成必须靠证据，不靠声明

AI 很擅长制造“看起来完成”：

- 文件齐全；
- 类型通过；
- 测试补了；
- 文档更新了；
- summary 很流畅；
- checklist 全勾了。

但关键行为路径可能根本没走通。

所以完成判定不能靠“我已经完成”。完成必须靠证据链：

- 哪些测试跑过；
- 哪条真实路径执行过；
- 哪些边界条件覆盖了；
- 哪些风险仍未覆盖；
- 如果失败如何回滚。

Agent PR 的核心不应该是 summary，而应该是 evidence。

---

## 九、审计也会产噪声

验证和审计不是站在系统外的麦克斯韦妖。它们自己也会犯错，也会制造噪声。

如果审计 Agent 发现十个问题，其中四个是误报，那么它不是纯粹在降噪，而是在向系统注入新噪声。更多审计不必然更可靠，尤其当审计同源同偏置时。

审计必须有二阶收敛：

```text
发散发现 → 独立复核 → 归类确认 → 只保留稳定成立的问题
```

审计终态也必须清楚：

- 已解决；
- 明确无需行动；
- 升级为人类裁决；
- 沉淀为自动化约束。

最危险的终态是“待跟进”“以后再看”“可能有问题”。这种模糊状态会无限累积，最后治理机制拖垮系统。

高质量审计的目标不是发现更多问题，而是让真问题留下，让伪问题尽快死亡。

---

## 十、吸引子：先定义“往哪里去”

验证和修正都预设一个前提：正确方向在哪里。

如果没有方向，检查再多也只是许多无目标的力。它们可能互相抵消，也可能一起把系统推偏。

所以 Agent 工程里必须先定义一个对象：**吸引子**。

吸引子不是某个具体终点，而是一组高阶不变量定义出的合法状态集合。你不可能枚举所有合法代码，但你可以定义少数约束：

- 概念边界不能混；
- 依赖方向不能反；
- 某些数据不能跨租户共享；
- 某些模块不能知道基础设施细节；
- 某些行为必须保持幂等；
- 某些接口语义必须稳定。

这些约束像隐式方程。你给的是方程，不是解。具体实现有无数种，只要满足这些不变量，就在合法区域内。

这解释了为什么一页 owner doc 可以约束一座代码库：它不是列举每个实现细节，而是定义结构边界。

### 切向自由，法向回拉

在这个视角下，AI 的活动可以分成两种：

- **切向运动**：在不违反不变量的前提下，重构、展开、组合、优化；
- **法向偏离**：破坏概念边界、依赖方向、数据隔离、语义约束。

切向运动应该放手。AI 的高吞吐价值就在这里。

法向偏离必须拉回。因为它不是健康演化，而是漂移。

这给人机分工一个更精确的表述：

> AI 负责在吸引域内高速移动；人负责定义和校正吸引域本身。

AI 可以提出候选方向，可以帮助发现矛盾，可以模拟方案后果，但不能未经裁决地把自己的输出升格为项目方向。因为它天然会向训练分布平均解漂移，而项目真正需要的方向经常恰恰不是平均解。

---

## 十一、拓扑：什么该固化，什么该实时重建

“结构”这个词太宽。要判断什么该写进 owner doc，什么该交给 AI 即时分析，需要更锋利的刀。

这里用“拓扑”指：

> 在实现层日常变化、重构、优化中仍保持不变的关系。

软件里的真拓扑主要有两类：

1. **概念切分**  
   什么是 Payment，什么是 Refund，库存和价格为什么不能混，权限和用户资料为什么必须分开。

2. **依赖方向**  
   领域层能不能依赖基础设施层，核心业务能不能反向依赖 UI，授权模块能不能调用租户外状态。

这些是慢变量，应该固化进 owner doc、ADR、架构规则，甚至转成 lint/test。

但下面这些不是拓扑：

- 某个文件当前 import 了谁；
- 某个类现在调用了谁；
- 两个模块“看起来相似”；
- 当前实现中的临时聚类；
- 某个重构前后的具体连线。

它们是几何，是快变量，应该从代码实时重建，而不是写进慢文档。

一旦把快变量焊进慢载体，就会制造静默漂移。文档看起来仍然权威，实际上已经过期，Agent 还会继续相信它。

所以索引也要守边界：索引可以承载关系，但不能复制事实；owner doc 可以承载拓扑，但不应复述当前依赖图；向量库可以找相似文本，但不能替代结构关系。

---

## 十二、注释：自明代码消灭的是哪一种

“代码应当自明”这句话经常被误用。

自明代码能表达 what 和 how，但表达不了 why 和 why-not。

注释是否该留，不取决于“多不多”，而取决于它的生命周期是否和代码一致。

### 该删的是复述型注释

```js
// count 加一
count++;
```

这种注释对人和 AI 都是噪声。

### 该留的是局部因果

```js
// 不用 map：这里必须在第一个 invalid item 时提前返回，避免触发后续副作用。
for (const item of items) {
  ...
}
```

这类信息和代码同地、同生命周期。代码改时它就在旁边，过期是吵闹的。搬到远处文档反而会制造安静漂移。

### 必须保住 why-not

“为什么不用那个更显然的写法”尤其重要。因为 AI 默认会向主流写法漂移。如果你不把 why-not 留在现场，下一个 Agent 很可能“优化”回你当初特意避开的方案。

跨文件、跨模块、约束整体结构的方向，则不应塞进局部注释，而应进入 owner doc。

---

## 十三、DSL 与约束：重塑地形，而不是反复拉扯

对抗 AI 漂移有两种方式。

第一种是持续施加外力：人工审计、反复复核、靠纪律维持约定。这是租金。一停止付费，系统就滑回去。

第二种是重塑地形：类型、测试、lint、runtime guard、DSL、权限边界。它们让错误状态不可达，或者让错误一出现就被自动捕获。这是买断。

能买断的，就不要长期付租金。

例如：

- 依赖方向不要靠 review 记住，写成 dependency lint；
- 数据隔离不要靠 prompt 强调，写成测试和 runtime guard；
- 输出格式不要靠自然语言劝说，改成 schema；
- 任务权限不要靠 Agent 自觉，放进沙箱和 allowlist。

DSL 是一种强约束工具。它能缩小 AI 的输出空间，让结果可校验。但 DSL 不是越自造越好。越偏离主流语言和常见格式，AI 越不熟，越容易掉出能力盆地。

最稳的 DSL 往往是主流语言或格式的受限子集，而不是全新语法。它既能收窄空间，又不把 AI 推到陌生地形。

约束的目标不是让 AI 不犯错，而是让错误更难发生、更容易发现、更便宜修复。

---

## 十四、许愿式开发：把检索误当成构造

一句话让 AI 做完整个需求，有时真的有效。但这不证明 AI 完成了一次全新构造。

很多时候，一句话只是命中了模型内部已有结构。它像 query，召回了训练分布里已经成形的模式。

所以许愿式开发的边界是：

> 在分布内已有结构时有效；在分布外需要新构造时失效。

危险在于二者输出形态相同。命中缓存和完成构造，看起来都是“说一句话，出一个结果”。用户用几次检索式成功建立信任，然后把这种信任误用到真正需要构造的新问题上，系统就会突然翻车。

出路不是不用自然语言，而是给组合空间剪枝。

绝大多数需求不是全新发明，而是组合式需求。AI 的优势在语义组合：它能把形式不同但语义相邻的东西接起来。但组合空间很大，需要结构化输入缩小搜索范围：

- 目标是什么；
- 哪些约束不可破；
- 哪些模块可改；
- 哪些路径已否决；
- 哪些测试必须通过；
- 哪些地方需要人类裁决。

一句话失败，不只是信息太少，而是没有给搜索空间足够约束。

---

## 十五、Agent 究竟是什么

把前面的线索收拢，Agent 可以这样定义：

> Agent 是一个在目标约束下，跨时间维持、复用、验证、修正可行动认知结构的系统。

这一定义里有几个关键词。

**跨时间**：单次调用不是 Agent，连续演化才是 Agent 工程的问题。

**可行动**：只会总结知识，不会改变世界，只是知识库，不是 Agent。

**维持**：保存事实、因果、置信状态、否定空间。

**复用**：把历史推理沉淀成 Skill、测试、规则、索引。

**验证**：用异质机制判断是否偏离。

**修正**：把系统拉回吸引子。

**目标约束**：没有吸引子，验证和修正都失去方向。

Agent 自己不跨时间，每次调用都会重置。真正跨时间演化的是项目。Agent 只是作用在项目状态上的算子：读当前状态，生成变更，接受验证，留下痕迹，再进入下一轮。

---

## 十六、人和 AI 的分工

AI 的强项是切向运动：在已经定义好的约束内高速展开。

它适合：

- 生成候选实现；
- 补测试；
- 写迁移脚本；
- 整理文档；
- 查找相似模式；
- 提出方案对比；
- 执行局部重构；
- 根据失败反馈迭代修复。

但几件事不能让 AI 独自决定：

- 定义项目吸引子；
- 重定义概念边界；
- 裁决冲突来源；
- 判断哪些约束升格为方向；
- 决定哪些例外长期存在；
- 校准“完成”的标准；
- 选择与训练分布平均解不同的非主流方向。

AI 可以参与这些事情，但不能拥有最终裁决权。因为这些动作是在闭环之外定义参考信号。闭环系统能把误差压小，但不能自己决定“零点在哪里”。

人类的价值因此不是消失，而是上移。

架构师的价值不再主要是亲手构造所有细节，而是：

- 定义流形；
- 维护不变量；
- 裁决边界；
- 设计验证机制；
- 把反复出现的问题买断成约束；
- 让项目在高吞吐下仍然收敛。

会写代码仍然重要。不会写代码的人很难定义有效约束，也很难识别 AI 的完成假象。但写代码从主要劳动，变成了定义方向和判断结果的基础能力。

---

## 十七、这些原则会互相打架

这些原则不是 checklist。它们彼此有张力。

### 维护成本 vs 阅读成本

事实只存一处可以降低维护成本，但相关信息分散会提高阅读成本。解决办法不是复制内容，而是用索引承载关系。索引只放关系，不复制事实。

### 外置真相 vs 文档漂移

真相必须外置，但外置文本会过期。解决办法是分层、标状态、设 owner，并尽量把稳定约束沉淀成测试、lint、类型和 guard。

### Skill 复用 vs Skill 僵化

Skill 能复用历史计算，也会固化旧路径。必须可版本化、可淘汰。

### 验证增强 vs 治理膨胀

检查越多不一定越可靠。审计会自产噪声。判据不是发现多少问题，而是净降噪多少，以及能否沉淀为自动化约束。

### DSL 约束增益 vs 能力盆地

DSL 越严越可校验，但越陌生越可能让 AI 不流利。最好选择主流形式的受限子集。

真正的工程判断，就发生在这些张力里。

---

## 十八、结尾：先定义流形，再施加力

Agent 不是管道拼接，不是更长的提示词，不是更多流程节点。它是作用在项目上的控制系统。

这一切的起点，是吞吐差改变了人的位置。人从低层执行 loop 退到高层控制 loop，过去由他隐式承担的维持、复用、验证、修正，必须显式重建。

重建出来的系统有两个层次，顺序不能颠倒。

**第一层是吸引子。**  
由少数高阶不变量定义项目应该收敛到哪里。它不枚举所有合法实现，而是定义边界、方向和约束。

**第二层是力。**  
plan、context builder、lint、test、audit、runtime guard、closure，把项目沿法向拉回合法区域。

AI 的价值在切向自由：在合法区域内高速移动、组合、展开。人的价值在定义和校正合法区域本身。

贯穿两层的是一笔账：

- 能买断的别租；
- 能自动化的别长期人工审计；
- 会自产噪声的验证要证明净降噪；
- 向量检索只能找相似，不能替代权威真相；
- 文档必须有状态、来源、作用域和 owner；
- 否定空间必须保存，否则时间一致性会崩。

团队跌倒的路口，通常不是因为少了一条 prompt，而是因为没有先回答：

**项目应该往哪里去？谁说了算？这个方向以什么形式存在？偏离时靠什么拉回？**

吸引子逻辑上先于验证。  
“向哪里”必须先于“如何约束”。

一旦接受这一点，Agent 工程就不再是让 AI 无限自治，而是在约束中建造：让 AI 在正确的地形里高速运动，让系统把错误状态变得更难到达，让人从逐行控制转为方向控制。

这不是一套最佳实践清单，而是一种工程立场：

> 先定义流形，再施加力；  
> 先外置真相，再谈记忆；  
> 先建立权威，再做检索；  
> 先买断约束，再堆审计；  
> 先回答“往哪里”，再让 AI 跑起来。
