# Development Log — 2026-04-15

### 2026-04-15

- Added `docs/analysis/2026-04-15-scope-state-and-materialization-optimization-report.md`, a focused scope-performance analysis covering current `ScopeRef` state shape, `read()` / `readOwn()` role separation, formula `materialize()` pressure, row-scope implications, benchmark evidence, and a recommended single-state + lazy-visible-view refactor direction.
- Key decision from the analysis: if compatibility is not a constraint, the optimal scope direction is not “flatten less” in isolation but “keep only own snapshot as real state, treat visible scope as a lazy view, and move plain-object flatten to explicit enumerable boundaries”.
- The analysis also concludes that `readOwn()` and `read()` should share one real underlying state model but remain distinct semantic APIs: owner-local snapshot vs visible lexical view.
