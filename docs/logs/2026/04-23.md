### V11 Design Architectural Revision

- **Identified fundamental flaw**: V11's execution strategies (Resumability, Islands, ExecutionStrategy per node) incorrectly assumed DSL can control framework-level execution
- **Root cause**: DSL layer cannot override React's hydration model, Resumability requires Qwik's architecture
- **Corrected design**: 
  - DSL declares **intent** via hints, not execution directives
  - Framework adapter layer decides how (or whether) to honor hints
  - Execution control (hydration, server/client split) stays with framework layer

### Follow-up Correction: Static Analysis Design

User pointed out issues with static analysis approach:

1. **Renderer should declare `staticCapable`** - e.g., `input` is never static even with static props
2. **Bottom-up computation** - post-order traversal, compute children first, avoid redundant recursion
3. **Region handling** - all children in all regions must be static for parent to be static
4. **name binding** - any `name` in schema means data binding, not static

Corrected static analysis rules:
| Condition | Result |
|-----------|--------|
| Renderer `staticCapable: false` | Not static |
| Props contain expressions | Not static |
| Schema has `name` binding | Not static |
| Has event handlers | Not static |
| Creates scope (data/sources) | Not static |
| Any child not static | Not static |
| All pass | **Static** |

### Key Lessons

```
DSL Layer:
- Schema compilation (including static analysis)
- Expression evaluation, Scope, Actions, Validation

Renderer Definition:
- Declares staticCapable (renderer knows its own nature)

Compiler:
- Bottom-up static analysis (no hints needed)
- Post-order traversal for efficiency

Framework Adapter Layer:
- Uses staticAnalysis for optimization decisions
- DSL doesn't dictate how to use it
```
