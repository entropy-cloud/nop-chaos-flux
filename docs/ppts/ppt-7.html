nop-chaos-flux 是什么

  • 全称：NOP Chaos Flux

    一个 Schema 驱动的前端运行时和渲染框架

  • 核心理念："Schema that executes, not just renders"

    Schema 不仅描述 UI，还参与执行

  • 定位：下一代前端运行时基础设施

    不是应用框架，不是 UI 组件库

  • 围绕七个原语构建

    Template、ScopeRef、Value、Resource、Reaction、Capability、Host Projection

高速生成下系统没有解体：效率只是副产品

真正的证据

  • 包边界越来越硬

    最终收敛为 compiler / action-core / runtime 三层分离

  • 结构越来越纯

    CompiledSchemaNode 等过渡结构被清除

  • 审计越来越苛刻

    经过 meta-review 和 re-verification 继续收敛

效率数据

实际投入：1 名开发者 + AI，6 周
保守估计传统工作量：12-24 人月
效率倍数：约 8-16 倍
真正罕见的不是快，而是在这种快里系统没有解体。

核心问题：AI 生成是扩张力，没有收敛机制就会持续漂移

  • 高带宽生成

    AI 能在极短时间内产出大量代码、测试、文档

  • 乐观偏差

    AI 倾向于流畅地叙述"已经完成"，即使行为闭环并未真正建立

  • 上下文漂移

    旧前提在长上下文里不断被复用，整组文件一起偏离真实基线

最危险的地方不是"写错一段代码"，而是局部都像对的，但整体在持续偏离。

局部都像对的，往往正是系统正在整体跑偏的时候。

答案：把开发组织成一个受控收敛的动力系统

  • 吸引子 (Attractor)

    一组能持续把实现拉回正确方向的稳定约束。由人定义、当前可执行的正确方向。

  • 扩张力 (Divergence)

    AI 生成代码、展开方案、补齐文档，把系统推向更大的状态空间。

  • 收敛力 (Convergence)

    工程机制（文档路由、Plan 收口、验证体系、审计闭环）把扩张结果压回吸引子附近。
定义吸引子
→
AI 扩张
→
收敛验证
→
基线更新
↺
没有吸引子，AI 不是在开发系统，而是在高效采样错误解空间。

什么是动力系统，什么是吸引子

动力系统

状态随时间演化、且下一步依赖当前状态的系统。我们关心的不是某个静止点，而是轨迹。

吸引子

状态空间中的一个稳定区域或结构——从足够近的地方出发的轨迹，最终会被它吸引并长期停留在它附近。

类比映射

动力学概念 软件工程对应
状态空间 所有合法实现状态的集合
吸引子 架构基线允许的结构空间
轨迹 每一轮 AI 生成 + 收敛的演化路径
混沌不等于随机；局部不可预测，不等于整体失控。

仓库真相带来的工程后果：生成和验收被真正拆开

传统开发

仓库 = 认知的外化

作者认知持有系统模型，仓库主要是表达媒介

AI 深度参与后

仓库 = 系统真相的载体

没有连续作者认知，系统真相主要落在仓库与外部证据中

对应关系

  • test / lint / audit = 测量

  • owner doc / plan / closure = 约束

  • logs / history / bugs = 轨迹记录

这个模型和传统开发的关键区别

瀑布 敏捷 AI 深度参与
强蓝图 / 长流水线 弱蓝图 / 短流水线 动力系统 / 轨迹控制
先设计，再实现，再测试 迭代设计，快速反馈 生成与验收被拆开
默认人类可同步把关 默认团队仍能持续跟上 重点变成系统轨迹控制

旧工程隐喻是"蓝图和流水线"，新隐喻是"动力系统和轨迹控制"。

再用流水线思维管理 AI 高速生成，只会把混乱组织得更整齐。

吸引子的工程实现 1：架构基线与文档权威链

核心约束

  • 七个原语闭合集

  • 编译优先原则

  • 数据/能力正交

  • 统一渲染器合约

  • 严格包边界

文档权威链

docs/index.md → 导航入口
docs/architecture/ → 定义现行基线
docs/analysis/ → 提供候选发现
docs/plans/ → 负责执行收口
docs/logs/ → 保存开发记忆
架构的价值不在于描述一切，而在于让错误结构无法合法存在。

吸引子的工程实现 2：保留澄清过程，而不只保留结论

  • 专门保留 docs/discussions/

    讨论文档不只写最终结论，也保留澄清过程

  • 澄清过程包括

    原始表达、AI 复述与提问、用户补充、纠正记录、最终总结

  • 后续 AI 不用凭空猜"为什么这样设计"

    也能看到"为什么不能那样设计"

关键洞察：对复杂系统来说，早期理解偏差往往比编码错误更致命。如果只保留最后结论，后来的 AI 看不到哪些前提曾经被证伪过——吸引子就会弱化。

扩张力的角色定义：AI 负责展开，不负责最终裁决

AI 主要承担

  • 代码生成

  • 文档整理

  • 方案展开

  • 审计候选生成

不交给 AI 自主决定

  • 架构裁决

  • 概念发明

  • 边界重定义

  • 方案淘汰

人机分工：人定义边界和收口标准，AI 高速扩张，多 agent 交叉验证把结果拉回基线，收敛规则由人工抽样校准。核心是不让同一个上下文既做实现又做收敛裁判。

收敛力 1：Plan 从"路线图"变成"收口机制"

  • Plan 不是路线图，而是局部收敛器

    核心文档：docs/plans/00-plan-authoring-and-execution-guide.md

每个 Plan 至少要写清

📋 Current Baseline
🎯 Goals / Non-Goals
🚪 Exit Criteria
✅ Validation Checklist
🔍 Closure Audit Evidence

硬规则：写计划前先核对 live repo，不沿用旧计划和旧 completion note。

Plan 作为收敛力的实际效果

历史上反复出现的漂移模式

一个计划过宽，没人知道什么时候算完成
只记录最后 landing 的内容，不回看整份 plan
接口和类型出现了，就误判语义已经落地
实现者顺手把 plan 标成 completed，没有独立 closure audit

Plan 机制的解决方案

  • scope 太宽 → 拆 successor plan

    每个 plan 只负责一个可关闭的结果面

  • 接口存在 ≠ 行为落地

    closure audit 必须区分 interface presence 和 behavior landing

  • completed 必须来自独立 closure audit

    不允许实现者自我宣布完成

收敛力 2：开发日志作为外部化工作记忆

日志最少记录四件事

  • 改了什么

  • 为什么这么改

  • 相关代码/文档/计划路径

  • 下一步是什么

日志的隐藏价值

保留"为什么不这么做"：

某个收敛尝试回滚了，因为暴露了另一个 runtime regression
某个架构方向先只收敛文档语义，不强推 runtime 重写
某个能力先作为 docs-only alignment 落地，不把 scope 扩大
日志不是收尾材料，而是系统记忆。记忆缺失会让吸引子变弱。

验证体系：自动化引力场与验证顺序

  • 根级质量门

    check:react19 / check:src-artifacts / check:oversized-code-files / check:i18n-keys / lint

  • 高信号规则直接升到 error

    react-hooks/exhaustive-deps / react-compiler/react-compiler / i18n 硬编码字符串

  • 验证顺序也是收敛逻辑

    先 focused verification → 修当前切片 → workspace 级 typecheck/build/lint/test
能靠机器提前失败的，就不要留给 review，更不要留给 AI 继续扩散。

审计闭环：从多 Agent 发散到共识收敛

发散
18 个 subagent 并发
→
过滤
meta-review
→
确认
re-verification

  • 多 agent 并发，不等于独立认知

    真正有价值的是认知独立性和最后的 reconciliation

  • 只有经过收敛的结论，才允许进入 owner doc、plan 或 backlog

高质量审计，不是堆问题，而是尽快排除假问题。

长期收敛力：Debugger、Bug 记忆与完成判定

Bug 记录

保存的不是 diff，而是排查路径：

为什么难定位
先看了什么
排除了哪些假设
最后依据什么确认了问题

nop-debugger 设计原则

按"AI 可消费"设计：

以 live cid 为中心
暴露 automation-facing API
AI 和 E2E 优先读调试 API，不读截图

"完成"定义：live code path 真经过 + focused/full verification 收口 + 文档同步 + leftover work 归属明确 + 独立 closure audit

闭环如何实际运转：一个完整周期

  1. 定义吸引子：owner doc 固定当前基线
  2. AI 扩张：实现代码、展开方案、补齐文档
  3. 实时收敛：lint/check/typecheck 持续拉回
  4. 切片收敛：focused verification 确认当前切片
  5. 记忆沉淀：开发日志记录决策理由
  1. 全局收敛：full repo verification 确认无回归
  2. 审计收敛：多源审查 → meta-review
  3. 关闭判定：独立 closure audit 确认到达吸引子
  4. 吸引子更新：更新 owner doc
  5. 下一轮：在新基线上继续扩张
这不是流程图，是系统的呼吸方式。循环的意义不是避免错误，而是让错误无法长期滞留。

吸引子为什么能起作用——它解决了 AI 的三个核心偏差

AI 偏差 表现 对应收敛机制
乐观偏差 "看起来做完了" Exit Criteria + Closure Audit 验证行为闭环
上下文漂移 "旧前提不断被复用" 开发日志 + 文档权威链恢复正确基线
局部正确全局漂移 "在错误的层做了正确的修复" 架构基线 + owner doc 限制错误方向

吸引子模型的核心价值：不依赖 AI 自己发现问题，而是构造外部的力场，让漂移自动被修正。

重点不是让 AI 不犯错，而是让错误不能长期存活。

这套方法的本质：Audit-and-Closure Driven Development

  • 审计-收口驱动开发

    把实现和完成判定明确拆开：人先定义吸引子，再让扩张和收敛交替发生

  • 持续纠偏闭环

    文档、discussion、plan、AI 实现、audit、closure、logs 组织成一个闭环

  • 承认 AI 的边界

    当前主流 AI 擅长在既有知识边界内高效执行，但很难自己长出真正新的架构观念
不要指望 AI 自己找到正确设计，要先定义一个不会把系统带偏的吸引子。

可复制结构与核心结论

五个必须回答的问题

🎯 谁定义方向？
⚖️ 冲突时信谁？
🔍 怎么发现漂移？
💾 怎么保留记忆？
✅ 怎么判定完成？

这五个问题都有答案，你就有了闭环；没有答案，系统就很容易在高速推进中偏掉。

可复制的不是表面做法，而是在当前主流 AI 条件下，人先定义新吸引子，AI 再围绕它高效收敛。

真正边界：当前主流 AI 擅长收敛执行，不擅长原创架构

AI 最适合承担

  • 实现

  • 展开

  • 补全

  • 验证

  • 整理

最不适合单独承担

  • 核心概念发明

  • 边界重定义

  • 第一性原理层面的架构突破

例子：ActionScope 与 Data Scope 分离并采用词法作用域，这类方案不是让 AI 自由采样平均解能自然长出来的。

当前主流 AI 最擅长把已有知识执行到极致，但很难自然长出真正新的架构观念。
