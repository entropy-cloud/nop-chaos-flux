# AGE：当软件工程从“完成任务”转向“控制轨迹”

AI 进入软件开发后，团队最先想到的往往是约束它：加 guardrail，加 review，加测试，加 audit，加 spec，加 checklist，加 skill。这并不奇怪。过去的软件工程长期围绕一个问题展开：**当前这次变更是否正确？**

AGE（Attractor-Guided Engineering，吸引子引导工程）要改变的不是某个流程，而是问题本身。

它要问的不是：

> 这次 AI 有没有把任务做完？

而是：

> 在 AI 连续高频扰动之后，整个仓库是否仍然沿着正确的长期结构收敛？

这是一种从“状态工程”到“轨迹工程”的转向。

如果不理解这一点，AGE 很容易被误读成更复杂的文档治理、更严格的审计流程、更完整的 spec-driven development，或者一个全局 `age-skill`。这些理解都抓到了一部分表象，但错过了 AGE 最有价值的地方：**它把 AI 软件工程的核心风险，从单次错误重新定义为长期漂移。**

传统软件工程并非从未关心长期演化。变化在于：过去经常作为隐含背景、事后诊断或资深工程师直觉存在的轨迹判断，在 AI 时代必须升格为软件工程的第一性对象。轨迹一旦成为第一性对象，就会反过来重排其他概念的位置。测试、文档、plan、audit、skill、review 不再只是各自独立的好实践，而要在“仓库轨迹是否收敛”这个问题下重新获得定义。

---

## 一、传统软件工程默认有人在脑子里维持方向

传统软件工程并非没有轨迹意识。技术债、架构腐化、测试脆弱、文档漂移、设计退化，这些词早就存在。

但这些词大多用于事后诊断。它们通常在系统已经出现问题之后才被拿出来使用：

> 这里技术债太重了。  
> 这里架构已经腐化了。  
> 这里测试太脆了。  
> 这里文档和代码漂移了。

这说明传统软件工程确实看见了轨迹问题，但并没有总是把“轨迹”作为第一等的工程对象。

把轨迹变成第一等对象不是术语替换。工程构造顺序变了：不再默认系统方向由人脑持续补足，而是要求仓库结构本身承载方向、暴露偏离、提供证明并保存纠偏记忆。

为什么过去这样也能工作？因为人类工程师本身承担了大量隐式轨迹整合工作。

一个核心工程师会记得：

- 某个模块为什么不能这么扩展；
- 某个接口虽然能复用，但不应该复用；
- 某个测试虽然通过，但保护的是旧实现细节；
- 某个文档虽然还在仓库里，但已经不是当前事实；
- 某个设计虽然代码能跑，但方向不对。

这些判断很多时候没有完整写进仓库，而是留在人脑、review 习惯、团队共识和历史经验里。

传统工程的基本结构是：

```text
状态检查 + 人类隐性方向感
```

测试、review、CI、文档、架构讨论负责检查当前状态；人的长期记忆负责把这些状态放回演化方向里理解。

AI 深度参与后，这个结构不再可靠。

AI Agent 不是更快的人类工程师。它是另一种扰动源。它可以在很短时间内跨模块修改代码，可以同时生成实现、测试、文档和总结，可以让局部材料彼此自洽。但它没有稳定的跨 session 方向感。每一次新的会话，都需要重新从仓库中恢复系统现实。

这意味着，过去藏在人脑里的轨迹判断，必须被外化到仓库结构中。

## 二、AI 时代最危险的不是“写错”，而是“每一步都看起来对”

如果 AI 只是偶尔写错代码，传统工程手段大体还能处理：测试失败、review 发现、lint 报错、类型检查不过。

更棘手的是另一种情况：

```text
代码能跑；
测试能过；
文档也更新了；
plan 也关闭了；
summary 写得很完整；
但系统整体越来越偏。
```

这种局部自洽的漂移，是 AI 大规模开发的核心风险。

比如一次局部 bug 修复中，AI 为了快速解决问题绕过了某个边界。测试通过了。下一次 AI 看到这个绕过已经存在，就把它当成当前事实。再下一次，它补了一些测试，测试开始保护这个绕过。之后文档也被更新，把这个临时结构描述成合法结构。

每一步都很合理。

但系统已经被带入了错误轨迹。

传统状态视角会问：

> 这次改动有没有问题？

AGE 会问：

> 这次改动把系统推向了哪个长期结构？

AGE 不满足于局部正确。它要判断局部正确的累积方向。

---

## 三、吸引子不是目标状态，而是系统继续成为自己的结构

AGE 的核心顺序是：

```text
状态空间 → 吸引子 → 轨迹 → 控制
```

这组顺序重新组织了软件工程的基本对象。

仓库不是静态文件夹，而是一个不断演化的状态空间。代码、测试、文档、plan、log、bug note、schema、CI 配置、audit evidence，都是这个状态的一部分。

每一次 AI 修改、测试补充、文档更新、审计结论、bug 复盘，都会把仓库推到一个新状态。

这些状态不是孤立点。它们连起来就是轨迹。

AGE 所说的吸引子，就是系统在长期演化中应该反复回到的稳定结构。

常见误读是把吸引子理解成 desired state，比如把它类比成 GitOps 里的声明式目标状态。这种类比有帮助，但会把 AGE 讲浅。

GitOps 里的 desired state 往往是一个明确状态：

```text
replicas = 3
image = v1.2.3
```

AGE 里的 attractor 不是这种明确状态。它不是一个固定快照，不是最终代码形态，也不是功能清单。

它更像一组结构关系：

```text
哪些领域概念不能混；
哪些依赖方向不能反；
哪些旧结构已经不再合法；
哪些语义承诺必须有 owner；
哪些行为必须有 proof；
哪些历史上被证伪的路径不能复活；
哪些文档有当前基线权威，哪些只是历史记录。
```

**吸引子不是“系统最后长什么样”，而是“系统在变化中如何继续成为它自己”**。

AI 开发中的很多错误，并不是直接破坏功能，而是改变系统成为它自己的方式。它让概念边界变松，让测试保护错误对象，让文档抢错权威，让历史错误重新出现，让临时绕路变成新的事实。

AGE 要保护的是这个层面的结构。

## 四、控制不是越多越好，控制必须知道往哪里拉

AI 工程讨论常从 harness 开始：如何加测试，如何审计，如何限制 Agent，如何让 workflow 更严格。

AGE 的判断是：

> 如果不知道系统应该收敛到哪里，控制机制越强，越可能把错误基线固化得越牢。

测试不是天然正确的控制。测试可能保护语义承诺，也可能保护旧实现细节。

审计不是天然正确的控制。审计可能重新观察 live repo，也可能制造一堆新的叙事和 backlog。

文档不是天然正确的控制。文档可能承载长期结构，也可能把过期事实伪装成当前基线。

skill 不是天然正确的控制。skill 可能帮助执行，也可能让 Agent 更高效地执行错误抽象。

AGE 不说“多加 harness”。AGE 说：

```text
Attractor before Harness.
```

`Attractor before Harness` 不是要求先写完所有架构文档再开发。它说的是逻辑顺序：先有“长期结构”的定义，控制才有方向。

没有吸引子，就没有偏离。

没有偏离判断，就没有有效纠偏。

没有纠偏方向，harness 只是局部质量检查。

这也是 AGE 不能被降级为 skill 的原因。skill 的结构是：

```text
任务意图 → 能力包
```

而 AGE 的结构是：

```text
领域概念 → 语义承诺 → owner → 实现 → proof → memory → 后续义务
```

`owner` 首先是一种语义权威功能，不是“某个人必须永远裁决一切”。系统需要某种机制定义、维护、裁定并演化这条语义承诺。

当前实践中，这个功能通常由人类 owner 与仓库结构共同承担，因为主流 AI 仍容易回到训练分布中的平均方案，也缺少稳定跨 session 责任。但这不是理论上的永久限制。只要某种 AI、团队机制或仓库自动化能够可靠承担语义权威和长期责任，它也可以成为 owner function 的一部分。

skill 可以帮助 Agent 执行任务，但不能替代仓库自身的语义结构。AGE 必须内化在仓库里，而不是只存在于某个被调用的 procedure bundle 里。

## 五、AGE 改变了“完成”的含义

传统任务视角里，完成通常意味着：

```text
代码改了；
测试过了；
文档更新了；
任务勾选了；
PR 合并了。
```

AI 开发中，这些信号尤其不可靠。AI 容易制造完成感：同时生成实现、测试、文档和总结，让所有材料看起来彼此支持。

AGE 对“完成”的定义更严格：

> 完成不是 checklist 被勾掉，而是变化义务被转化为仓库中的稳定事实。

用 q/p 语言说：

```text
p → q
```

`p` 是变化义务。它表示系统为什么需要动、往哪里动。

`q` 是稳定状态。它表示系统现在以什么形式承载这条承诺。

例如，一个需求说：

> 隐藏字段不应该参与表单提交。

实现前，这是一种变化压力。它可能存在于用户请求、requirement 或 plan 中，要求系统改变。

任务完成后，这条承诺不应该只出现在 summary 里，也不应该只被写进 log。它应该变成稳定事实：

- 代码实现了对应行为；
- 测试证明的是“隐藏字段不参与提交”，而不是仅仅证明“隐藏字段 UI 不显示”；
- 如果它是长期规则，owner doc 吸收这条规则；
- audit 能从 live repo 重新验证；
- 未来 session 能恢复这个判断；
- bug note 或 log 保存必要的历史语境。

如果这些没有发生，任务只是叙事完成，不是结构完成。

AGE 对“完成”的重定义，能揭穿 AI 开发中大量“看起来已经完成”的假闭合。

## 六、PHS 语言的价值：不是炫技，而是让微妙差异可见

AGE 后来借用了 PHS、Hamiltonian、J-flow、R-flow、q/p、Park transform 这些数学物理语言。很多读者会本能反感：软件工程为什么要说这些？

这种反感有道理。如果这些词被当作严格数学证明，那确实过度。

它们的价值不在证明，而在**提高概念分辨率**。

普通工程语言太容易把不同动作混在一起：

```text
整理需求；
更新文档；
同步信息；
记录问题；
关闭任务；
做审计。
```

这些词看起来清楚，分辨率却不足。

比如，把一个模糊需求从 chat 写进 plan，再从 plan 写进 log，普通语言会说“信息已经沉淀了”。但 AGE 会追问：

> 你只是移动了不确定性，还是消除了不确定性？

J-flow 是语义承诺在正确拓扑中移动。它可能是必要的，但不等于收敛。

R-flow 是未澄清、未归属、未验证的不确定性减少。比如 scope 被裁定，owner 被明确，测试证明行为，owner doc 吸收稳定规则，closure audit 确认 live repo。

没有这组语言，团队很容易把“信息搬运”误认为“系统收敛”。

普通语言说：

> 当前不支持 CSV 导出。

这句话至少有四种含义：

```text
稳定地不支持，因为 out of scope；
暂时不支持，但 active requirement 要求实现；
当前支持，但应该被删除；
当前支持，并且是稳定能力。
```

只看当前状态不够。必须区分当前事实和变化义务。

Park transform 同样只是类比。它不是说软件仓库真像电机坐标变换，而是强调：把 chat 内容写进 repo 不是复制，而是重坐标化。

chat 里的信息混合了当前事实、未来计划、临时猜测、历史判断、未决问题、已证伪路径。写入仓库时，必须把它们放到不同责任坐标里：

- 长期结构进入 owner docs；
- 本轮收口进入 plans；
- 历史轨迹进入 logs；
- 非显然根因进入 bugs；
- 探索判断进入 analysis；
- 当前实现进入 code/tests。

这不是文档整理，而是工程现实的坐标变换。

PHS 语言不该让 AGE 显得高深。它的作用是避免关键差异被普通工程话语吞掉。

## 七、AGE 不是文档更多，而是仓库承担更多现实

把 AGE 理解成文档治理，会错过它的核心。

AGE 不是说文档越多越好。文档越多，漂移面也越大。

AGE 要求的是：

> 仓库必须承载 AI 下次进入时能够恢复的工程现实。

传统开发里，工程现实的一部分留在人的脑中。AI 没有这种持续记忆。对下一个 session 来说，未进入仓库的判断几乎不存在。

AGE 中的文档、测试、plan、log、bug note、audit evidence，不是辅助材料，而是仓库现实的组成部分。

代码仍然重要。代码回答当前实现是什么。

但代码不能单独回答：

- 系统应该向哪里收敛；
- 某个设计为什么成立；
- 哪个旧路径已经被证明会发散；
- 某个测试到底证明哪条承诺；
- 某条文档是不是 current baseline；
- 哪个未完成问题是阻塞项，哪个只是非阻塞后续。

AGE 不再把仓库理解成“代码为主，文档为辅”。

> 仓库是代码、测试、owner docs、plans、logs、bugs、audit evidence 共同组成的可恢复工程现实。

不同材料回答不同问题。冲突时不能简单问“代码权威还是文档权威”，而要问：

```text
你在问当前行为，还是长期方向？
你在问证明证据，还是历史原因？
你在问本轮 closure，还是最终基线？
```

AGE 的语义权威结构建立在这些问题区分上。

## 八、AGE 和传统实践的关系：不是否定，而是重新定位

AGE 并不否定传统实践。

测试、review、CI、ADR、spec、traceability、GitOps、DDD、fitness functions，这些都很有价值。

AGE 改变的是它们的位置。它们不是 AGE 的本体，而是 AGE 图像中的局部投影。

当测试保护语义承诺时，它是 proof relation。

当测试锁死旧实现细节时，它是漂移源。

当 spec 管理行为变更时，它是有用的 harness。

当 spec 试图吞掉架构、历史、bug、analysis、owner precedence 时，它会压平语义责任空间。

当 ADR 记录关键决策时，它是 memory harness。

当 ADR 只是沉睡文档时，它不能约束未来轨迹。

当 skill 帮助 Agent 执行审计或定位 bug 时，它是方法节点。

当 skill 试图承载 AGE 本身时，它把仓库内生结构降级成外部调用能力。

这些工具要回到同一个判断标准下：

> 它们是否帮助仓库轨迹围绕吸引子收敛？

## 九、AGE 最锋利的一句话

AGE 可以被压缩成一句话：

> AI 时代的软件工程，不能只管理任务完成，必须管理仓库轨迹收敛。

AI 改变了软件工程的风险结构。

过去，很多长期方向可以由人脑补足。

现在，AI 高频生成会让局部正确变得廉价，却让长期一致性变得稀缺。

稀缺的不是更多会写代码的 Agent，而是能够承担 owner function 的语义权威结构：定义吸引子，维护轨迹，识别假闭合。

在当前阶段，这通常表现为人类 owner 和仓库结构的协同。但 AGE 关心的是这种功能是否存在并可靠工作，而不是它必须永久由人类逐步手工执行。

AGE 不只是长期复杂仓库的治理术。长期复杂仓库最早、最清楚地暴露轨迹漂移，所以最需要完整 AGE 结构。但 AGE 的对象是 AI 时代的软件工程本身。小项目、原型和业务应用同样有状态空间、吸引子、轨迹和控制，只是它们所需的 attractor 与 harness 可以更轻。

AGE 还改变人与 AI 的分工方式。传统控制依赖 human in the loop：人盯住每一步输出、逐个 diff、逐条 review 来强行控制。AI 高频开发下，这种方式很快不可扩展。

AGE 更接近 human out of loop guidance：人或 owner function 设计状态空间、吸引子、权威结构、验证面和记忆机制，让 AI 在其中自主展开和收敛。人不必停留在逐步控制上，而是在更高层引导轨迹边界和收敛方向。

AGE 的价值不是让项目多写几份文档，也不是让流程更繁琐。它的价值在于逼迫团队面对一个过去经常被人脑兜底的问题：

```text
系统在持续变化中，如何仍然成为同一个正确系统？
```

如果一个实践不能回答这个问题，它就不是 AGE 的核心。

如果一个实践能帮助回答这个问题，哪怕它只是一个轻量 plan、一条 bug note、一个 focused test、一次 fresh session audit，它也是 AGE 的一部分。

---

## 结语：AGE 带来的认知

AGE 带来的不是一个新模板，而是一种新的工程直觉：

```text
不要只看当前点，要看轨迹。
不要只看任务完成，要看变化义务是否转化为稳定事实。
不要只看文档是否更新，要看语义承诺是否进入权威位置。
不要只看测试是否通过，要看测试是否证明正确承诺。
不要只看 audit 是否列出问题，要看它是否降低不确定性。
不要只看 AI 是否很能干，要看仓库是否能让下一个 AI 恢复同一个系统现实。
```

这就是 AGE 的第一性原理。

它不是传统工程实践的简单组合，而是对 AI 软件工程对象的重画：从任务到轨迹，从状态到收敛，从执行能力到结构保持。

AGE 的革新不在于声称传统实践从未存在。它把过去散落在技术债、ADR、fitness function、GitOps、review 文化和架构直觉中的长期演化问题，提升为 AI 软件工程的基础构造原则。第一性对象改变之后，其他实践不再按原来的方式并列存在，而是在轨迹收敛这个中心问题下被重新组织。

把 AGE 视为更严格的流程，它会显得繁琐。把 AGE 视为轨迹工程，它的锋利之处才出现：

> 在 AI 高频扰动下，软件系统最危险的失败方式，不是某一步明显错了，而是每一步都看起来对，最后系统不再是它本来应该成为的系统。
