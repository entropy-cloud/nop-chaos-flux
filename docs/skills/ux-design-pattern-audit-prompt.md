# nop-chaos-flux UI/UX 设计合规性审查提示词

> **定位**: 本文档是一份面向 AI 的多轮递归式 UI/UX 设计合规性审查提示词。目标是发现渲染器组件中不符合行业常见设计模式、视觉不一致、交互缺陷、可访问性不足等问题。
> **执行方式**: 多轮递归发现，每轮在上一轮基础上扩展搜索范围，直到收敛。
> **输出格式**: 每个发现按统一格式输出（见附录 A）。

---

## 审查定位

本审查不同于 `deep-audit-prompts.md` 的维度 09（渲染器契约）和维度 10（样式系统合规性）。那些维度关注的是**代码架构契约**（RendererComponentProps 遵循、marker class 纯标记、BEM 残留等）。

本审查关注的是**用户可见的视觉和交互质量**——用户看到什么、点击什么、感受到什么。即使代码在架构层面完全合规，也可能在视觉和交互层面存在设计缺陷。

### 本审查覆盖的范围

| 类别           | 示例                                                                       |
| -------------- | -------------------------------------------------------------------------- |
| 图标语义       | 同一操作使用不同图标；图标选择不符合行业惯例（如用 chevron 代替漏斗）      |
| 按钮一致性     | 同一语义操作在不同组件中使用不同 variant/size/颜色                         |
| 状态指示       | 排序方向无区分、选中状态不明显、禁用状态不可感知                           |
| 表单交互       | 标签不可点击、缺少 indeterminate 状态、radio 用 checkbox 模拟              |
| 视觉反馈       | 可聚焦元素无 focus ring、loading 无 spinner、空状态无内容                  |
| 对话框/弹出层  | 缺少关闭按钮、trigger 文案误导、不可见原生控件、缺少焦点陷阱和 Escape 处理 |
| 颜色使用       | 硬编码颜色而非设计令牌、颜色语义不当（红色用于非破坏性操作）               |
| 间距对齐       | 删除按钮浮动而非对齐、装饰元素重叠、未使用 ml-auto 靠右                    |
| 焦点管理       | 对话框未限制 Tab 循环、关闭后焦点未回到 trigger、Escape 未关闭弹出层       |
| 破坏性操作确认 | 行级删除无提示直接执行、批量删除缺少确认对话框                             |
| 滚动容器       | 嵌套双滚动条、粘性头部失效、虚拟化与普通组件滚动行为不一致                 |
| 触摸友好性     | icon-only 按钮过小（<44px）、可点击元素间距过挤                            |
| 无障碍可见性   | role 语义错误、缺少 aria-label、图表使用 role="button"                     |
| 跨组件一致性   | 分页 UI 不统一、同功能不同交互模式                                         |
| 版式层次       | 同一含义文本在不同组件中使用不同字号/字重、长文本溢出无处理                |

### 本审查不覆盖的范围

以下由 deep-audit 已有维度覆盖，不要重复报告：

- 渲染器是否遵循 RendererComponentProps 契约（维度 09）
- marker class 是否带视觉样式、BEM 残留（维度 10）
- 是否用了原生 HTML 而非 @nop-chaos/ui（维度 11）
- field metadata / value-or-region 建模（维度 12）
- 可访问性的全面 WCAG 合规（维度 20）

**边界规则**：如果一个发现同时涉及 UX 交互质量（本审查）和 WCAG 合规（维度 20），优先归入**维度 20** 以避免重复，除非该问题的**主要影响**是视觉/交互缺陷而非可访问性合规。以下给出具体边界示例：

| 属于本审查（UX 交互质量）                     | 属于 deep-audit 维度 20（WCAG）              |
| --------------------------------------------- | -------------------------------------------- |
| 使用了错误的排序图标（功能正常但约定错误）    | 缺少 `aria-label` 导致屏幕阅读器无法识别操作 |
| 按钮颜色不一致（同一操作的 variant 不统一）   | 颜色对比度不足导致低视力用户无法辨认         |
| 缺少 loading 状态指示器（用户不知操作进行中） | `role="button"` 用于非交互元素               |
| 对话框无关闭按钮（用户找不到退出方式）        | 焦点未限制在对话框内导致键盘用户迷失         |
| 空状态无提示内容（用户不知为何无数据）        | 表单验证错误消息未关联到输入控件             |
| 行间距不均、字体大小不一致                    | 缺少 `lang` 属性或地标角色                   |

不确定时标记为 `[scope-conflict]` 在汇总中说明，由主 agent 判断归属。

## 行业设计基准

审查时以以下行业基准为参照。不需要在发现中引用具体来源，但判断"是否符合常规"时应以这些系统的实际做法为准：

| 交互场景       | 行业标准做法                                                                                           |
| -------------- | ------------------------------------------------------------------------------------------------------ |
| 可排序列头     | asc = `ArrowUp`（高亮），desc = `ArrowDown`（高亮），未排序 = `ArrowUpDown`（极淡），不可排序 = 无图标 |
| 删除操作       | 行级删除 = ghost + Trash2Icon + hover:text-destructive；批量/危险删除 = destructive variant + 确认     |
| 新增操作       | ghost + PlusIcon 或 outline 按钮，统一风格                                                             |
| 表格全选       | 部分选中 = indeterminate 状态（短划线）                                                                |
| 单选行         | 使用 radio 语义，非圆形 checkbox                                                                       |
| 表格筛选       | 漏斗图标（ListFilterIcon），非 chevron                                                                 |
| Loading 状态   | Spinner 组件 + 可选文本，非纯文本                                                                      |
| 空状态         | 有意义的提示内容（"暂无数据" / 图标），非空白                                                          |
| 对话框         | 右上角可见关闭按钮（X），不隐藏；打开时焦点陷阱 + Escape 关闭                                          |
| 表单标签       | 点击标签切换控件状态（Label + htmlFor）                                                                |
| 可聚焦交互元素 | focus-visible ring，非仅 hover 变色                                                                    |
| 颜色语义       | 硬编码色值应改为 CSS 自定义属性（设计令牌）                                                            |
| 分页           | 同一 CRUD 系统内分页 UI 保持一致                                                                       |
| 图表           | role="img"，非 role="button"                                                                           |
| 按钮文案       | 无 label 时 fallback 为有意义的文本或 aria-label，非字面 "Button"                                      |
| 删除确认       | 批量/危险删除必须弹确认对话框；行级删除推荐确认（至少 toast 撤销）                                     |
| 触摸目标       | icon-only 按钮最小 32x32px（建议 44x44px）                                                             |

> **惯例冲突处理**：当 Ant Design、shadcn/ui、MUI 之间对同一模式有不同做法时：
>
> 1. 优先以 **shadcn/ui** 为准（因为本项目直接使用 shadcn/ui 组件库）。
> 2. 若 shadcn/ui 未定义该模式，以 **Ant Design** 为参考（因为目标用户是 AMIS 低代码用户）。
> 3. 若三者均不同，选择本项目已有组件中一致性最高的做法，并在发现中注明"存在多个行业惯例，本项目应采用其中一种并统一"。
> 4. 如果本项目不同组件已采用不同做法，无论是否符合行业惯例一律报告为"跨组件不一致"。

## 执行前验证（Pre-flight）

在开始第 1 轮之前，主 agent 必须执行以下验证：

```
1. 确认审查范围内的包目录存在且非空
   - ls packages/flux-renderers-form/src/renderers/
   - ls packages/flux-renderers-data/src/
   - ls packages/flux-renderers-basic/src/
   - ls packages/flux-renderers-form-advanced/src/
2. 确认 docs/analysis/ 下是否存在同日期或近期 UX 审查报告
   - 如存在，阅读 1 分钟了解范围，避免重复
   - 如不存在，继续正常流程
3. 确认 pnpm build/typecheck/lint 已通过（代码状态可靠）
4. 向用户输出验证结果："环境就绪，开始第 1 轮审查"
```

如果目录为空或代码无法构建，输出"环境检查失败"并终止。

## 多轮递归执行模型

### 执行方式

本审查采用**开放式多轮递归发现**模型：

#### 阶段一：迭代发现（多轮）

1. **第 1 轮（初扫）**：按"检查视角"（见下文）全面扫描所有渲染器组件，产出的发现列表保存到 `docs/analysis/{日期}-ux-audit/round-01.md`。
2. **第 2 轮（递归扩展）**：读取第 1 轮全部发现，基于已有发现暴露的模式、组件、关联路径，继续深挖：
   - 已有发现涉及的组件是否有同类型问题未被检出？
   - 已有发现的模式是否在其他组件中也存在？
   - 第 1 轮扫描中是否有尚未覆盖的组件？
   - 是否有跨组件交互场景（如"表格 + 筛选 + 排序组合使用时"）未被检查？
   - **去重检查**：将你的新发现与已有发现列表比对根因。如果根因相同（即使表象不同），不要重复报告。
3. **第 N 轮（继续递归）**：重复第 2 轮逻辑，以第 N-1 轮发现为输入继续扩展。
4. **终止条件**：
   - 某轮输出"未发现新的问题"（需按零发现报告格式）或新发现数量为 0
   - 或新发现仅为低价值、机械重复、缺少明确用户影响的问题
   - 上限保护：每轮审查最多 **10 轮**
5. **价值收敛条件**：递归不是为了凑数量。如果已有发现已经覆盖了所有主要交互模式，继续搜索只能得到极低价值的问题，应输出"未发现新的高价值问题。审查结束。"

#### 阶段二：复核

深挖阶段结束后，进入复核阶段：

1. **维度复核**：派发一个独立子 agent，读取全部轮次的完整发现文件，重新对照 live code 逐条核对，输出"保留 / 降级 / 驳回"的逐条判断。
2. **子项复核（分层）**：对 HIGH 发现和存在争议的发现项逐项复核；MEDIUM/LOW 风险项可按组件或按模式批量复核。

#### 核心要求

- 复核 agent 不能复用发现 agent 的结论当作事实。**具体做法**：对每条发现，先 grep 定位到文件位置，不看发现 agent 的结论自行判断问题是否存在，然后再与发现 agent 的结论对比。
- 最终汇总报告只能使用"已通过独立复核"的结论。
- **发现条目完整性**：所有子 agent 的输出都必须遵守"发现条目完整性守则"（见附录 A）。
- **质量门槛**：发现条目的证据片段必须确实支持结论。如果证据与结论之间逻辑断裂（如证据代码中没有渲染上下文但结论断言"缺少空状态"），视为不合格，要求重新生成。

### 调度模式

```
主 Agent（协调者）
  │
  │  ── 阶段一：迭代发现 ──
  │
  ├── Task → Round 01（初扫全部组件） → 保存到 round-01.md
  ├── 去重校验（主 agent 兜底纠正：即使子 agent 已做自检，主 agent 再扫一次是否有与已有发现根因重复的条目）
  ├── Task → Round 02（基于 R01 递归扩展） → 保存到 round-02.md
  ├── 去重校验（主 agent 检查 Round 02 与 R01 无重复）
  ├── ...（直到无新发现或达到 10 轮上限）
  │
  │  ── 阶段二：复核 ──
  │
  ├── Task → 全量维度复核（读取所有轮次，逐条保留/降级/驳回）
  ├── Task → HIGH 逐项复核
  ├── Task → MEDIUM/LOW 批量复核
  │
  └── 汇总已复核通过的结果 → summary.md
```

### 递归扩展提示词模板

从第 2 轮起，主 agent 在基础提示词后追加以下内容：

```text
---

## 递归扩展指令（第 N 轮）

以下是本审查前 N-1 轮已保存的全部发现：

[粘贴前 N-1 轮的完整发现文本]

你的任务：

1. 读取上述已有发现，理解已经覆盖了哪些组件、模式和交互路径。
2. 基于已有发现暴露的组件、模式和关联路径，**继续深挖**尚未覆盖的盲区：
   - 已有发现涉及的组件是否有同类型问题未被检出？
   - 已有发现的模式是否在其他组件中也存在？
   - 第 1 轮扫描中是否有尚未覆盖的组件或交互场景？
   - 是否有跨组件交互场景（如"表格 + 筛选 + 排序 + 分页组合使用时"）未被检查？
   - 是否有第 1 轮未检查到的边缘场景（如 disabled 状态、空数据、加载中、错误状态）？
   - 是否有响应式/移动端场景的视觉问题？
3. **只输出新发现的条目**。不要重复已有发现中已经报告的内容。
4. **去重自检**：在输出前，将你的新发现与已有发现列表按**根因**比对。根因相同的合并或丢弃。
5. 如果经过仔细检查后确实没有发现新的问题，按"零发现报告格式"输出。
6. 如果仍能找到零散细节，但这些细节只是低价值、机械重复、缺少明确用户影响，不要为了凑数量继续报告；输出："未发现新的高价值问题。审查结束。"
7. 每个新发现仍需遵守统一的发现格式（见附录 A）。
```

### 零发现报告格式

当某轮次未发现新问题时，输出必须包含以下内容以便验证：

```text
## 零发现报告 — 第 N 轮

### 本轮检查范围
- 读取的文件列表：[文件路径列表]
- 检查的组件/模式：[组件名列表]
- 对照的发现类别：[已对照的前 N-1 轮发现编号]

### 本轮检查方法
- 对每个文件，检查了：___（如：图标语义、按钮 variant、空状态、加载态）
- 特别关注了前轮未覆盖的场景：___（如：disabled 状态的按钮颜色、空数据的筛选器）

### 结论
经过对上述文件的逐项检查，未发现新的 UX 设计合规性问题。
```

零发现报告必须包含"检查范围"和"检查方法"——仅"未发现新的问题"五个字不被接受。

### 保存与追加规则

1. **每轮结果独立文件**：`round-01.md`, `round-02.md`, ... 不覆盖、不合并。
2. **复核结论**：`review.md` — 引用轮次和发现编号，不重写发现内容。
3. **汇总报告**：`summary.md` — 仅包含已通过独立复核的结论。
4. **去重记录**：如果主 agent 在轮次间发现重复，在 `review.md` 中标注合并/驳回记录。

### 上下文窗口管理

从第 4 轮起，如果累积发现文本超过 2000 行，使用以下策略管理：

```
- 在递归扩展指令中，将前 N-1 轮发现**压缩摘要**（每条发现 1 行：编号 + 组件 + 根因）
- 仅保留第 1 轮和上一轮的**完整文本**用于引用
- 在摘要末尾注明："如需核对某条发现的完整代码证据，请参考原轮次文件"
```

## 共享提示词前缀

主 agent 在派发任一轮次前，应先附加以下共享前缀：

```text
你正在审查 nop-chaos-flux 项目的 UI/UX 设计合规性。这是一个 React 19 + Zustand + TypeScript 的低代码渲染引擎 monorepo（pnpm workspace）。

执行前先阅读：
1. docs/index.md
2. AGENTS.md
3. docs/architecture/styling-system.md
4. docs/architecture/renderer-markers-and-selectors.md
5. docs/architecture/theme-compatibility.md
6. packages/ui/src/index.ts（可用组件清单）

审查范围：
- packages/flux-renderers-form/src/
- packages/flux-renderers-form-advanced/src/
- packages/flux-renderers-data/src/
- packages/flux-renderers-basic/src/
- packages/ui/src/components/（作为"正确用法"参照）
- apps/playground/src/（实际使用场景）

审查标准：
- 以用户可见的视觉和交互质量为准，不是代码架构合规性
- 以行业常见设计模式为参照（Ant Design、shadcn/ui、MUI、AG Grid 等）
- 行业惯例冲突时的优先级：shadcn/ui > Ant Design > MUI
- 同一语义操作在不同组件中的视觉表现应一致
- 交互状态（hover、focus、active、disabled、loading、empty）应有恰当的视觉指示
- 不要求追求"最完美"的设计，但要求没有明显违背行业惯例的问题

通用口径：
1. 以当前代码为准，不与历史版本对比。
2. 不重复报告架构契约问题（由 deep-audit 维度 09-12 覆盖）。
3. 不把"不够美观"当问题，必须有明确的用户交互障碍或行业惯例偏离。
4. 每个发现必须可定位：文件路径 + 行号范围 + 3-10 行证据片段。
5. 区分"功能缺陷"和"设计不佳"：功能缺陷（如点击标签无法切换 checkbox）优先级高于纯视觉问题。
6. 区分"真发现"和"建议"：
   - **发现** = 当前实现违背行业惯例或交互模式有缺陷
   - **建议** = 功能正常但可以进一步优化
   只报告"发现"。"建议"可以在汇总报告末尾的"可暂缓项"中列出，不计入发现统计。
7. 所有结论都必须经过独立复核。
8. **真实用户影响检验**：每个发现回答一个问题——"非设计师用户不被告知的情况下，是否会在正常使用时注意到这个问题？"如果答案是否定的，降级或驳回。

严重程度判级：
- HIGH: 构成用户交互障碍或功能缺陷。用户无法完成一个操作、数据丢失风险、或操作有不可逆的破坏性后果。例如：点击标签不切换 checkbox、删除无确认对话框、loading 状态无指示器（操作耗时>2s时）。
- MEDIUM: 明显的不一致性、缺失的视觉反馈、可改进的交互模式。用户能完成操作但体验不一致或不符合预期。例如：同语义操作在不同组件中使用不同图标、缺少 hover/focus 态、空状态无文字提示。
- LOW: 视觉细节优化、边缘场景微调。用户可能注意到但不会影响任务完成。例如：2px 间距偏差、次要操作的 color shade 不一致、非关键路径上的图标语义可选歧义。

升级规则：如果 MEDIUM 问题出现在高频交互路径（如 CRUD 列表页的删除操作），应升级为 HIGH。
降级规则：如果 HIGH 问题仅在极低频率的操作路径中出现（如每页一次的可关闭提示条），可降级为 MEDIUM。
```

## 已知常见误报模式（False Positive 对照表）

以下模式在之前的审查中被误报为问题，实际是正确设计。子 agent 发现类似模式时不要报告：

| 误报模式                                                                | 正确解释                                                                                                                               |
| ----------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------- |
| `opacity-0` 的 native `<select>` 用作日期选择器或自定义选择框的 trigger | 这是 shadcn/ui 的标准模式，通过不可见原生控件保持表单语义和键盘导航                                                                    |
| `ml-auto` 对齐方式显得"hacky"                                           | 这是 Tailwind flex 布局的标准右对齐方式，不是问题                                                                                      |
| 按钮使用 `variant="ghost"` 无边框                                       | shadcn/ui 规范中 ghost variant 是标准用法，适用于行级操作和次要操作                                                                    |
| Icon-only 按钮使用 `size="icon-xs"` 或 `size="icon-sm"`（小于 36px）    | shadcn/ui 规范定义了 icon-xs (24px)、icon-sm (32px)、icon-md (36px) 等尺寸，用于行内操作。在数据密集的表格行中使用小图标按钮是行业惯例 |
| 包裹内容溢出使用了 `text-ellipsis overflow-hidden` 等截断               | 对于表格列、标签等固定空间内容，截断是标准做法。只有当截断导致信息丢失且没有 Tooltip 提示时才应报告                                    |
| 使用 `div` 而非 `button` 元素但添加了 `role="button"` 和 `tabIndex={0}` | 在自定义组件中这是可接受的 ARIA 模式，不应仅因此报告                                                                                   |
| 表单验证错误使用 `variant="destructive"` 按钮                           | destrucitve 用于按钮颜色语义，当验证错误导致需要重置或危险操作时是合理的                                                               |
| 使用了 `transition-all` 而非逐个声明的 `transition-colors`              | 在当前性能预算内可接受。只有当有明显的 layout 动画导致的性能问题时才应报告                                                             |

````
### 完整示例

```markdown
### [视角1-01] Table 排序列头缺少方向区分图标

- **文件**: `packages/flux-renderers-data/src/table-renderer/table-header-row.tsx:158-168`
- **证据片段**:
  ```tsx
  {isSortable && (
    <ArrowUpDownIcon
      className={cn(
        'inline ml-1 size-3',
        currentSort ? 'text-primary' : 'text-muted-foreground',
      )}
    />
  )}
````

- **严重程度**: MEDIUM
- **现状**: 可排序列无论 asc/desc/unsorted 都渲染同一个 `ArrowUpDownIcon` ↕，仅靠颜色（`text-primary` vs `text-muted-foreground`）区分是否激活。用户无法直观看出当前排序方向。
- **行业惯例**: shadcn/ui Table、Ant Design Table、AG Grid 均使用三态排序图标：asc = `ArrowUpIcon` 高亮、desc = `ArrowDownIcon` 高亮、unsorted = `ArrowUpDownIcon` 极淡。用户一眼即可识别排序方向和激活状态。
- **用户影响**: 用户需要点击表头后仔细观察颜色变化才能确认当前排序方向，在大表格中易混淆。键盘用户更困难。
- **建议**: 改为三态渲染：`currentSort === 'asc'` 时渲染 `ArrowUpIcon + text-primary`；`currentSort === 'desc'` 时渲染 `ArrowDownIcon + text-primary`；无排序时渲染 `ArrowUpDownIcon + text-muted-foreground/40`。
- **复核状态**: 未复核

```

5. **增量审查**: 可以只聚焦特定包（如只审查 flux-renderers-form），不必每次全量。
6. **参考已有分析**: 如果 `docs/analysis/` 中已有 UI/UX 审查报告，先快速浏览以避免重复，但不以此为限制。
7. **多轮递归不是线性**: 第 N 轮可以跳到任何上一轮发现中涉及的文件或模式，不限于第 N-1 轮。
8. **每轮上限**: 最多 10 轮。达到上限后强制进入复核阶段。
9. **错误恢复**: 如果子 agent 超时或返回畸形输出，主 agent 应重试一次，更换 seed/prompt 表述方式。如果第二次仍然失败，跳过该轮并记录"第 N 轮因执行异常跳过"。
10. **环境验证**: 开始前必须执行"执行前验证"步骤，确认扫描目录存在且代码可构建。
```
