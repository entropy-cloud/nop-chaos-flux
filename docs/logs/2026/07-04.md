- **修复 dialog 级 `actions` 不渲染（surface-runtime）**：用单元测试验证了 `openDialog.args.actions` 确实不渲染（此前 CRUD demo 因此改用 form 级 actions）。
  - **根因**：`surface-runtime.open` 构造 `SurfaceEntry` 时，`body`/`title` 都有 `options?.x ?? surface.x` 回退，唯独 `actions: options?.actions` 无回退；而 `openDialog` adapter（`action-adapter.ts:202-213`）调用 `surfaceRuntime.open` 时只传 `ownerScope/actionScope/componentRegistry/validationPlan/ownerNodeInstance`，不传 `options.actions`。结果 `entry.actions` 恒为 undefined，`DialogHost`（`dialog-host.tsx:219`）的 `actionsNode` 为 null。
  - **修复**：`surface-runtime.ts` 将 `actions: options?.actions` 改为 `actions: options?.actions ?? (surface.actions || undefined)`，与 `body` 对齐。`renderSurfaceNode`（`dialog-host-surface.tsx:89`）本就接受 raw schema 并经 `RenderNodes` 编译，故无需额外改动。
  - **回归测试**：新增 `packages/flux-react/src/__tests__/dialog-actions.test.tsx`（body 对照 + actions 渲染两个用例）。修复前 actions 用例 fail、修复后全绿。

- 关键记录：
  - RendererDefinition 的 `fields` 只包含该 renderer 特有的字段，不含继承自 schema type 的字段（如 data-source 的 `action`/`args`/`name`）。`validate-json.mjs` 因此对继承字段报 Unknown key，`validate-with-compiler.mjs` 同样报 Unknown property。这是定义模式本身的特性，非 bug。
  - `packages/flux-core/src/constants.ts` 的 `BUILT_IN_ACTION_REGISTRY` 不含 `confirm`/`alert` ——它们有 ActionSchema 类型和 runtime dispatch，但不在 compiler 的 builtin action 注册表中。
