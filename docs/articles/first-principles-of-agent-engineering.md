按确定的方案处理：新增"吸引子"一节聚拢方向、命名为一等对象并含分层/优先级/共演化；给"自我验证陷阱"命名；同时撤销我上一版对第三节的拆解，让它回到干净的"错误有方向 → 验证四条"。范畴论全部去掉。

# 在约束中建造：AI Agent 工程的根本原则

做 Agent 的团队，几乎都在同一个路口跌倒。

起初你觉得这很简单：让 AI 读项目文档，给它任务，让它写代码提 PR。头几天效果惊人。然后怪事开始发生——AI 反复提出已经被否决的方案；在不同会话里对同一件事给出矛盾判断；制造出文件齐全、测试全绿、总结流畅但功能根本不 work 的 PR。你开始加规则：更多文档、更详细的提示词、更多 checklist、更多审计。系统变重了，没有变好。有时更差——审计发现了更多"问题"，修复引入新问题，系统在自己制造的噪声里膨胀。

这不是某个模块写错了。这是在和几条基本约束较劲，而这些约束不是"加更多东西"能绕开的。

大语言模型有三个绕不过去的事实：它不会自动记住、它的每一次思考都有成本、它的错误有方向。认真对待这三件事，Agent 的设计就会从"最佳实践"变成不得不如此的结果。

---

## 一、模型会忘记，真相必须外置

LLM 不携带跨会话记忆。你今天在一个窗口里建立的全部理解，明天换个窗口重新进入时不会延续。当 AI 的生成速度远超人类消化速度，人也不再持有完整蓝图。模型记不住，人也记不全，"项目现在是什么样"这个问题就只剩一个回答者——仓库自己。

真相必须存在于模型权重之外的权威载体里。这不是最佳实践，是唯一选择。

但"真相"不只是代码当前长什么样。一个能支撑长期协作的真相，至少有四个维度。

**事实本身**——代码、接口、数据结构、当前架构。最直观的一层。

**因果来源**——为什么变成这样、谁做的决策、基于什么约束、替代了什么旧结构。缺这一层，下一个 Agent 看到一段代码就无法判断它是深思熟虑的设计，还是一次临时将就。缺失因果，项目认知会在时间中漂移。

**置信状态**——知道什么、不知道什么、哪些存疑、哪些来源在冲突。LLM 是概率系统，如果它只输出"最可能答案"而隐藏不确定性，它就隐藏了真相的一个维度。置信状态不是附加说明，是真相本身。

**否定空间**——哪些路已被排除。哪些方案被否决、哪些假设被证伪、哪些重构被证明代价过高、哪些 bug 路径排查过但判为误报。这一维最容易丢，丢了之后后续 Agent 会反复重提早被否决的方案。表面像在探索新思路，实质是在破坏时间一致性。

这四维有一个共同点：它们记录的都是"现在是什么"和"曾经发生过什么"。还有一种东西不在这四维里——"系统应该收敛到什么"。那是方向，不是事实，需要单独处理。这一点后面专门有一节谈。

### 真相不能平铺，必须分层

事实搬出大脑之后，新问题是事实本身会变。领域规则、时效事实、私有约定——这些"快变量"如果烧进模型权重，模型一更新事实就跳变，多个 Agent 跑在不同版本上事实就分裂。

凡是会变的，必须可被替换；凡是必须可被替换的，就不能烧进权重。

权重只承载慢变量：语言、推理、常识。快变量必须在权重之外。外置的知识也不能简单堆在一起。当不同来源冲突时，如果没有明确的"谁说了算"，AI 会把它们平均成一个错误现实。架构原则高于分析文档，分析文档高于历史记录，历史记录高于临时讨论。

这条原则推到底就是尺度分离：信息应存放在与它的变化速度、作用域和权限相匹配的载体里。

| 信息               | 变化速度 | 该放哪             |
| ------------------ | -------- | ------------------ |
| 语言、推理、常识   | 极慢     | 模型权重           |
| 架构原则           | 慢       | owner doc          |
| 领域知识、Skill    | 中       | 可版本化的外置文档 |
| 项目状态、生产数据 | 快       | 项目状态源         |
| 当前任务上下文     | 很快     | 当前会话           |
| 临时草稿           | 瞬时     | scratchpad         |

大多数错误源于放错层：快变量烧进权重、临时假设写进长期记忆、一次性 workaround 固化成 Skill、过期注释当真、当下实现细节误升为架构原则。

尺度分离推到底还有一个反直觉的结论：AI 深度参与的项目里，注释应该克制，而不是越多越好。注释和代码绑在一起，描述的往往是会变的东西——当时的取舍、临时的理由——是放错层的快变量。代码改了注释没改，它就成了过期的错误信号。AI 读到要么被误导，要么得花算力判断它还可不可信。宏观信息进 owner doc、log、discussion；局部约束靠类型、测试、lint、runtime guard 这类可执行机制固化。不预防性写注释，只在反复调试或审计后确认某个局部约束极易被误读时，才补一条短注释。

## 二、思考有成本，而且不能打折

Wolfram 的"计算不可压缩性"在 LLM 上有一个直接体现：任何任务内在的推理步数都有下界，没法用一句"请认真思考"压成一步。

### 拆任务看判断维度，不看长度

很多人知道复杂任务要拆，但说不清"复杂"的标准。标准不是字数，不是文件数，是任务内在的逻辑判断点数量。

一个修 bug 的任务，如果同时要求"修 bug + 遵守代码风格 + 写测试 + 更新文档 + 保持向后兼容 + 考虑性能"，看起来是 6 个要求，实际是 6 个独立判断维度。每个维度在每个需要决策的代码位置都要 AI 做出选择。维度数 × 决策点数 = 总判断量。超过单次推理算力时，AI 不是做不好，是做不到——它会在某些维度上完全失守，通常以"看起来很周到"的方式失守。

正确的拆法是按维度拆：先只修 bug 并验证修好，再调风格，再补测试，再更新文档。每步把算力集中在一个维度上。

小模型不是"差的大模型"，是单次计算容量更小的执行器。给大模型一段含糊描述它能跑动，给小模型必须明确分步。这不是它笨，是它单次能动用的算力本就更少。用小模型时，prompt 必须从描述式切换到指令式——把人类该做的拆解工作提前做完。

OODA 循环（观察-判断-决策-行动）常被当成一种好的设计模式。它其实是算力约束下的强制形态：单次推理装不下完整任务，所以必须分步；分步必须循环；循环必须有反馈。任何宣称"一步搞定"的 Agent 设计都在违反这一条。

计算不可压缩，不等于信息不可组织。推理过程省不掉，但可以用更好的表示来消除无效计算——用索引降低检索成本、用 schema 消除歧义、用测试降低验证成本、用类型收缩状态空间、用工具外包精确计算、用子任务隔离上下文污染。推理不可任意压缩，但任务空间可以通过表示、工具和分解来重构。

### 上下文是预算，不是容器

这是最常被搞错的一条。

大多数人把上下文当成装信息的桶——能装多少装多少，AGENTS.md 越写越长，prompt 越塞越满，理由是"反正模型读得到"。

上下文的本质是计算预算。每个 token 都在消耗注意力分配，每条信息都在和其他信息争夺处理能力。一个 50K token 的上下文不是"装了 50K 信息的桶"，是"50K 个 token 互相干扰、共同决定一次推理质量的计算场"。

这有两个直接后果。

第一，冲突信息的成本远高于缺失信息。"用 A 模式，但有时也用 B"比"用 A 模式"贵一个数量级——前者强制 AI 在每次决策时分一份算力去判断"现在算不算那个'有时'"。写指令要追求确定性，不是完整性。宁可说"统一用 A"再在极少数例外处单独标注，也不要在通用规则里堆条件分支。

第二，相关性比完备性重要。把所有规则塞进 system prompt 是反模式。核心约束和高频偏好进常驻上下文，其余按需加载——SKILL 触发式、owner doc 引用式、历史日志检索式。

一个 Agent 系统设计水平的简单信号：看它如何组织知识。AGENTS.md 动辄上千行且全部常驻的，基本可以判定团队拿信息总量当设计，而不是拿加载策略当设计。真正的设计问题是知识加载策略，不是知识总量。AI 进入任何任务时，应该能在前几百个 token 内拿到决策所需的最小相关集。

## 三、错误有方向，验证必须层层防漂移

AI 的输出不是无偏的随机扰动。它系统性地朝训练分布的平均解漂移——朝主流方案漂、朝看起来合理的常见写法漂、朝它见过最多次的模式漂。这个漂移有方向。

要纠正一个有方向的漂移，前提是先知道正确的方向在哪——这件事本身后面有一节专门讲。这里先假定方向已经给定，看验证这套力必须满足什么条件。每一条都来自"错误有方向"。

### 多采样救不了，验证必须异质

人们处理 AI 错误常用统计直觉——"它有 X% 概率出错，多采样几次取一致"。这套对付随机噪声有效，对付方向性偏置无效。偏置方向相同，采样十次会一致地错向同一个方向。多采样能消随机抖动，消不掉系统偏置。

系统偏置只能靠外部锚点：方向定义、可执行的约束、与训练分布不一致的明确指令。

这立刻推翻一种常见设计：验证 Agent 和实现 Agent 用同一个模型、同一种 prompt、共享同一份上下文。它们带着同向偏置——实现错了，验证也认为对。反馈和误差同向，纠偏就退化成放大。有效验证必须引入异质性：换模型、用独立 session（不共享上下文）、用根本不受 LLM 偏置约束的可执行检查（lint / test）、或者用人。

### 验证器自己也带偏，验证需要二阶收敛

异质验证还不够。验证器本身是 AI，它同样带偏置、同样会错、同样制造噪声。

普通 Agent 设计想到"LLM 会错，加个 verifier"就停了。可 verifier 是 AI，它一样会错。验证本身需要二阶收敛——审计的产出不能直接当结论，它自己得先收敛一遍。

审计结果不能自动变成 backlog。把审计发现照单全收地塞进待办，等于把系统拖回噪声里。审计要走"发散 → 过滤 → 确认"的闭环：多视角发散提候选，再用独立视角逐项回到现场重新核对，只保留跨视角仍然稳定成立的。在一个真实项目里，初审提出的判定经独立复核后约四成被重新归类为非问题——不加二阶收敛的审计，制造的噪声可能比发现的问题还多。

还有一个停止判据专治治理机制自我膨胀：审计终态只允许"已解决"或"明确无需行动"，禁止"延后""待跟进""挪到后续"这类模糊终态。否则 backlog 无限增长，治理机制最终把系统自己拖垮。

高质量审计的目标，不是发现更多问题，而是让真正的问题留下来，让伪问题尽快死亡。

### 单步全合格，整体也会跑偏

前面的验证都是单步——这一步对不对、这个发现成不成立。但 AI 漂移的本质是每步偏一点点，而每步又都看起来合理，所以每步都能通过单步验证。局部稳定不蕴含全局稳定：每个点附近都收敛的系统，整条轨迹仍可能缓慢发散。

最危险的失败由此而来：每一步都看似合理，每个改动都有理由，每个测试都通过，但整体方向已经偏离了最初目标。这是"局部正确，全局漂移"。逐步置信度标记抓不到它，因为每一步的置信度都很高。

它只能靠轨迹层机制来抓：定期全局复核、与架构基线对照、与最初意图对照、与否定空间对照（看是不是又走回了早被排除的老路）、用独立 session 复核。核心是回答一个单步视角永远回答不了的问题：这个系统是不是正在悄悄变成另一个东西？

第一节埋下的否定空间，在这里成了轨迹层复核的关键参照——如果你没保存什么路被排除过，你就无法判断当前的漂移是不是在重蹈覆辙。

### 漂移最隐蔽的产物：完成的假象

AI 朝"看起来合理"漂移，它就极度擅长制造"看起来完成"——文件有了、类型有了、测试有了、文档有了、checklist 勾了、summary 写得流畅全面。一切齐备，但关键的行为路径根本没走到。

接口存在不等于行为落地。"完成判定"因此是一个一级问题，不是流程末端的小事。

这里有一个值得单独命名的风险——**自我验证陷阱**：让同一个上下文既生成又判断完成，它会倾向于证明自己完成了。这不是效率问题，是认识论问题——生成者没有资格做自己的验收者。所以完成判定必须满足两点：completed 靠证据不靠声明，要有真实走通的代码路径、通过的验证、同步到位的文档；完成判定必须由独立上下文回到现场重新确认，而不是实现者顺手宣布。

### 置信度必须显式输出

真相包含置信状态这一维，AI 的输出又是带方向的概率分布，那么一个只报告"最可能答案"、不报告自己有多确定的 Agent，就是在隐藏真相的一维。

每个推理步骤应附带置信或边界标记；遇到知识冲突应显式暂停、请求裁决，而不是自行把冲突和稀泥消解掉；"未知"必须是一等公民的答案——Agent 要能说"这个问题我无法在当前仓库状态下确定"。这不是工程礼貌，是概率系统保持真相完整的硬性要求。

## 四、Skill：把不可压缩的计算存下来

计算不可压缩——一条推理路径走过去，那份算力就花掉了，省不回来。但走过的路径，结果可以复用。而真实任务不是完全随机的，存在反复出现的诊断路径、被一再重新发明的解法。

Skill 是过去不可压缩计算所探索出的局部最优路径的固化。它的意义不是组织代码，是把不可压缩的计算成本一次性付清，避免每次推理重复支付。Skill 之于 Agent，就像缓存之于 CPU、肌肉记忆之于人。没有 Skill 机制的 Agent，等于每次都让 CPU 不带缓存跑。

Skill 也是快变量，同样服从"仓库即真相"：环境变了旧路径会失效，所以它必须外置、可版本化、可被淘汰。它的来源不止人工编写——那些经过二阶收敛、确认成立的高信号审计发现，那些反复出现的诊断套路，都该被识别出来、下沉成 Skill。这是把扩张过程中偶发的智慧，固化为系统长期能力的标准路径。

---

## 五、接口可以柔，但柔不等于免费

LLM 能在不同形式之间识别相同语义，这是它和传统软件差别最大的地方。

传统软件接口刚性，因为它只能逐字节匹配。对 Agent 来说，一段能力描述写成 SKILL 格式、写在 AGENTS.md 里的自然语言、还是 README 的一节，在语义层面是等价的。接口承载语义，不被某种形式锁死。

柔性不等于免费。语义可以等价，处理成本不等价。结构化形式更省 token、触发更稳定；自然语言形式更灵活、更易演化。形式选择不是在"自由"和"死板"之间选，是在预算约束下对保真度、成本和演化灵活性做权衡。形式语义等价，形式代价不等。

---

## 六、吸引子：把方向变成一等对象

前面几节反复出现一个词，却一直没有正面定义它——"方向""架构基线""系统该收敛到哪里"。验证要对照它，漂移是相对它而言，人机分工的核心也是它。它出现得这么频繁、承担这么多重量，却散落在 lint 规则、测试断言、某份文档和某个人的脑子里。这种东西必须被拎出来，命名成一个独立对象。

它叫吸引子：系统应当长期收敛到的结构。

### 吸引子不是事实，也不是规则的副产品

第一节那个伏笔在这里收回。四维真相记录的是"现在是什么"和"曾经怎样"，吸引子记录的是"应该收敛到什么"。它们是不同的事实源，必须分开存放。混在一起会出一种很难查的错：当代码现状和吸引子冲突，你分不清是代码错了（该改代码），还是吸引子过时了（该改吸引子）。

吸引子也不是 lint 规则、测试断言的总和。把方向散在检查机制里，能工作，直到你问它两个回答不了的问题。

第一个：两条规则冲突时听谁的？散落的方向没有仲裁能力，AI 会把冲突来源平均成一个谁都不满意的折中。

第二个：一个结构没违反任何规则、通过所有测试、不报任何错，但它就是不对——怎么办？检查机制对它完全沉默，因为它没踩任何线。

这两个问题逼出吸引子的独立性：它定义的是约束，不是合法状态的清单。你不可能枚举所有合法的代码，但可以用很少的高阶不变量框定"什么样的结构是合法的"。这是吸引子能用少数文档承载、却约束海量代码的原因——它说的是边界，不是逐点列举。

### 吸引子能驱动没有错误信号的动作

把方向独立出来，解锁了一种检查机制永远做不到的动作：主动排除。

一段代码可以完全正常工作、测试全绿、lint 不报错，却因为违反吸引子的不变量而被删除。触发删除的不是任何错误信号，是它和系统该收敛的方向不一致。一个只有检查、没有独立吸引子的系统永远不会主动做这件事；它只会等到某个结构真的出错才响应，而那时债已经累积很久了。能否主动清除"还能用但不该留"的东西，是有没有真正的吸引子的试金石。

### 吸引子的分层与优先级

吸引子不是一块铁板，它分三层。

**结构层**——最高阶的方向不变量，规定系统的基本切分方式：哪些东西必须分离、依赖只能朝哪个方向流、什么概念不允许混在一起。它最稳定，改动代价最大。

**承载层**——把结构层落成可版本化、可审计、可继承的载体，也就是 owner doc、架构文档。它是结构层的书面形态，让方向可以被引用、被传承、被审查。文档不是吸引子本身，是吸引子的载体。

**实现层**——具体代码如何体现这些方向。它变化最快，是被前两层约束的对象，不是约束的来源。

这套分层直接给出了那个最难的日常判断的判据——"这个约定到底算架构基线，还是临时 workaround"。判据不是凭感觉，而是先问"你在问什么问题"：问"现在的行为是什么"，看实现层；问"应该收敛到哪"，看结构层和承载层；问"为什么变成这样"，看历史记录。冲突出现时，显式归类成三种之一——实现漂移（代码偏离方向，改代码）、文档漂移（方向过时，改吸引子）、有意保留的遗留行为（两者都对，记录例外）。归类清楚，该改谁就清楚。

### 吸引子逻辑上先于验证

这是整套体系里顺序不能颠倒的一处。验证、审计、lint 都是作用在系统上的力，而力必须有目标。没有吸引子，"纠偏"这个词没有意义——纠向哪里？

所以绝大多数团队的发力点其实搞反了。他们把精力全压在第二层——加更多检查、更严的门禁、更细的审计——却没有人先把"系统该往哪去"定义成一个独立的、说了算的东西。检查再多，也只是一堆没有目标的力，彼此抵消，或者一起把系统推得更偏。先有吸引子，验证才谈得上。

### 吸引子本身也会被校正

吸引子不是一次定下就永久不变。实践会暴露出方向定义里的不精确，所以真正的循环是闭环，不是单向执行：

> 定义吸引子 → 让 AI 围绕它高速扩张 → 检测并纠偏 → 用实践反馈校正吸引子 → 再扩张

反复出现的 bug 模式蒸馏成新的结构约束，审计中被反复确认的边界沉淀成更精确的方向定义——这些不是在收紧验证，是在精炼"系统该收敛到哪里"本身。一次次纠偏喂回来的，最终改写的是吸引子。

但这里有个不能松的约束：吸引子的更新不能是 AI 在迭代中自己漂出来的。AI 默认朝训练分布的平均解漂移，让它自由演化方向，等于让它把你的系统慢慢拉回最常见的样子。方向的每一次更新，必须经过明确的裁决并沉淀——人来定，或经独立审计确认，而不是默默累积。校正吸引子是有意的工程动作，不是迭代的副产品。

## 七、这些原则之间会打架

把这些原则放在一起，它们内部存在真实的张力。用得对不对，恰恰体现在如何处理这些张力。

单一权威真相对上全局漂移。仓库是真相的载体，但仓库里每一样东西都可能随时间漂移——注释过期、文档滞后、Skill 僵化。尺度分离能调和大半，剩下那部分靠吸引子的分层和优先级：把方向和事实分开存，冲突时先问"你在问什么问题"，再归类成实现漂移、文档漂移或有意遗留。

Skill 复用对上 Skill 僵化。固化下来的局部最优，叠加上 AI 朝训练分布漂移的倾向，会加重路径依赖。缓存能提速，也能缓存错误。"可淘汰"不是口号，是必须真正建立的机制——没有淘汰能力的 Skill 库不是资产，是不断累积的债务。

回拉力对上治理膨胀。验证本身也会引入噪声。"检查越多越可靠"并不成立——尤其当这些检查同源、同偏置时，它们只是在重复同一种错误。审计终态只允许"已解决/无需行动"、禁止模糊终态，就是为这条张力设的刹车。验证的质量不看数量，看它是否异质、是否独立、是否能杀死伪问题而不是永久养活它们。

这些张力不是体系的缺陷，正是判断力真正介入的地方。原则负责指方向，权衡留给用的人。

---

## 八、Agent 究竟是什么

把这些约束和推导收拢，Agent 的定义自然浮现：

> Agent 是一个在目标约束下，跨时间维持、验证、修正、复用可行动认知结构的系统。

四个动词各自回应一个必须解决的问题：

维持——让认知跨时间存在。针对"模型会忘记"。不做就会失忆、时间漂移。

验证——判断认知是否仍然接地、整体是否跑偏。针对"错误有方向"。不做就会自洽幻觉、假完成、全局漂移。

修正——偏离后回到方向上。同样针对"错误有方向"。不做就会僵化、错误长期滞留。

复用——把历史计算转化为未来能力。针对"思考有成本"。不做就会重复付费、永远从头开始。

还有隐含的第五点：行动。缺了它，上面这一切退化为一个知识库，不是 Agent。

这四个动作里，验证和修正都预设了"知道正确的方向在哪"。所以真正排在最前面的，是第六节那个对象——定义吸引子。方向先于约束，这个顺序贯穿整套体系。

---

## 九、人和 AI 的分工随之确定

AI 的强项是扩张：高带宽地生成代码、补齐文档、展开方案、发散候选——在已定义好的方向上高速推进。

但有几件事不能交给 AI 自主决定：定义吸引子、发明概念、重定义边界、裁决冲突、校准收敛标准、判断哪些是真问题、决定哪些新约束该升格为方向。这些恰恰要求"与训练分布不一致"，而 AI 默认朝训练分布的平均解漂移。你让它设计新架构，它给的是最常见架构的重组；你让它解决非主流问题，它把问题扭成它见过的主流问题。

人类的稀缺价值不在写代码，在定义代码应该长成什么样。AI 适合在已定义的方向上高速展开，不适合定义方向本身。架构师的价值不是被削弱了，是从"亲自构造所有细节"转为定义系统该收敛到哪里、以及让它收敛的机制。

---

## 十、结尾

Agent 不是管道拼接，不是更长的提示词或更多的流程节点。它是那个带着认知结构、跨时间行动的执行体。当它作用于项目，整个项目的演化就构成了一个受控动力系统——在有限计算下，面对重复但变化的任务分布，通过外部状态、工具调用、验证反馈和计算摊销来最小化长期认知成本，在持续扩张中维持向正确方向的收敛。

吸引子定义“向哪里”。Harness 负责测量和纠偏。当这些力方向独立、作用点正确，系统才能对抗 AI 概率性带来的天然漂移，在时间中持续维持真实的、可行动的项目认知。

吸引子逻辑上先于 harness。“向哪里？”必须在“如何约束？”变得有意义之前被回答。

这不是一套可以勾选的最佳实践清单。它是一旦接受那几条公理之后，你必然要走到的地方。
