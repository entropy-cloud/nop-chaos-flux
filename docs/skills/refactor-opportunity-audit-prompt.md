## 项目背景

这是一个 React 19 + Zustand + TypeScript 5.9 + pnpm workspace 的低代码渲染引擎 monorepo。主要分层如下：

- `flux-core`：基础类型、常量、纯工具
- `flux-formula`：表达式编译/求值
- `flux-runtime`：编译结果落地、action、request、validation、store、runtime 生命周期
- `flux-react`：React 集成层、hooks、NodeRenderer、SchemaRenderer
- `flux-renderers-*`：渲染器层
- `*-core` / `*-renderers`：复杂设计器或编辑器的 core/render 分层
- `ui`：共享 UI 组件
- `docs/`：架构、计划、日志、参考资料

核心依赖方向应保持为：

`flux-core -> flux-formula -> flux-runtime -> flux-react -> flux-renderers-*`

## 这个项目历史上高频出现的真实重构触发器

你必须优先按这些模式找问题，因为这些模式已经在本仓库中多次被证明是真问题：

1. 超大文件混合多种职责
2. orchestrator 之外的文件同时承担 UI、状态、运行时编排、工具函数
3. React effect 中承载了本应属于 runtime 的生命周期、轮询、缓存、去重、取消、清理逻辑
4. 本地 state 与 store / scope / form values 维护同一份事实来源
5. 外部 store 的 selector 或 snapshot 过宽，导致不必要重渲染或 React 19 兼容问题
6. 同一概念有双词汇、双字段、双实现，长期靠兼容层维持
7. 复杂模块的核心入口已经做过一轮拆分，但残留 orchestrator 仍继续堆积细节
8. domain core / renderer / runtime / docs owner 边界漂移
9. 测试文件过大、跨领域、setup 过重，导致验证脆弱或难维护
10. 目录顶层文件过多，相关实现没有按子域归组
11. 为兼容旧写法保留的过渡路径已经开始妨碍主边界收敛
12. 文档与代码当前 owner 不一致，导致后续重构方向继续漂移

## 审计时必须遵守的判断原则

1. 不要把“代码难看”当作重构理由，必须给出结构性原因。
2. 不要把低代码系统中诚实存在的动态类型边界误判为缺陷，例如某些 `Record<string, any>` 可能是合理的。
3. 不要因为文件大就机械建议拆分；如果它已经是边界清晰的 orchestrator，可以保留。
4. 不要把“再抽一层”当作默认答案；优先寻找更小、更稳定的收口点。
5. 不要只报现象，必须说明根因类别：职责混杂、边界错误、重复实现、兼容债务、订阅过宽、结构漂移、测试拖累、文档 owner 漂移等。
6. 对每个建议都要判断 ROI：为什么值得现在做，而不是以后再说。

## 优先扫描维度

### 维度 1：超大文件与职责混杂

重点寻找：

- 超过 300 行的源码文件
- 超过 500 行且混合 3 种以上关注点的文件
- 已经存在“入口文件 + 一些子模块”，但入口文件仍不断吸入细节的情况

对每个候选文件，请回答：

- 文件路径与大致行数
- 文件内部有哪些职责簇
- 哪部分是合理 orchestrator
- 哪部分已经应该提取
- 建议提取到什么边界，而不是只说“拆一下”

### 维度 2：React 层承载 runtime 语义

重点寻找：

- `useEffect` 中做轮询、缓存、AbortController、订阅编排、request dedup、source/reaction 清理
- renderer 或 React hook 直接拥有 runtime 生命周期
- cleanup 里做本应由 runtime owner 负责的资源释放

这类问题在本仓库里历史上多次被修正，优先级很高。

### 维度 3：双状态 / 双事实来源

重点寻找：

- `useState` 和 form/store/scope 同时维护同一数据
- effect 把 props/store 值同步到本地 state
- 一个组件既从 selector 读，又在本地缓存镜像数据
- 两个字段名表达同一业务概念

### 维度 4：订阅过宽与 React 19 external-store 风险

重点寻找：

- selector 直接订阅整个 `values`、整个 `scope`、整个大对象
- selector 返回新的对象/数组引用导致无意义重渲染
- `getSnapshot()` 每次返回新对象
- 组件仅依赖少量字段，却被大范围状态变动频繁触发

### 维度 5：重复实现与契约未收敛

重点寻找：

- 同一概念的双词汇、双字段、双 API
- 对称路径长期复制，例如 dialog/drawer、node/provider、action/compiler、renderer/shell 两条路径重复增长
- 非必要兼容层仍保留且开始污染主路径
- 不同包各自维护相似 helper、类型、常量、流程

### 维度 6：包边界与模块归属错误

重点寻找：

- runtime 逻辑遗留在 `flux-react` 或 renderer
- 本应纯函数的共享逻辑散落在上层包
- core/render 边界不清晰
- 包间依赖方向虽未形式上反转，但语义 owner 已经错位

### 维度 7：目录结构与文件归组问题

重点寻找：

- `src/` 顶层文件过多
- 明显属于同一子域的文件未归组
- 已经拆出一些子模块，但仍有大量同类文件散在顶层
- 子目录切分方式与真实职责边界不一致

### 维度 8：测试结构拖累重构

重点寻找：

- 单个测试文件过大且覆盖多个子域
- setup/helper 大量重复
- 测试名称、场景、夹具已经暴露出被测模块边界不清
- 验证困难的原因其实来自结构问题，而非单个 bug

### 维度 9：文档 owner 漂移造成的重构障碍

重点寻找：

- 文档 owner 与真实代码 owner 不一致
- 历史迁移说明仍占据当前架构文档主体
- 读文档会把后续重构者引向错误边界

这类问题不是第一优先级，但如果它直接妨碍代码边界判断，也要报出来。
