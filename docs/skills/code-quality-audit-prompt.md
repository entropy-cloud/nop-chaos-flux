# 代码质量审计提示词

审计当前项目代码质量，重点关注三类结构性问题：冗余代码、重复实现、过度抽象。

## 审计口径

1. 以当前代码为准，不以历史日志或已关闭计划为准。
2. 每个发现必须有结构性原因和当前 ROI，不要把"不够优雅"当理由。
3. 区分"真实缺陷"与"合理的 orchestrator / 动态边界 / 过渡方案"。对低代码框架中的动态类型和 orchestrator 模式保持克制。
4. 同一个问题可能同时属于多个类别（如同一文件既过大又包含重复代码），如实报告。

## 三类核心问题

### 1. 冗余

不该存在的代码。死了的、没人用的、绕了一圈什么都没做的。

- 无消费者的导出、类型、函数
- 仅为兼容已消亡的旧路径而保留的中间层，且主路径已稳定
- 过度包装的 pass-through 函数（调 A 只是 return A）
- 文件只剩下 2-3 行重导出，且无独立 owner 理由

### 2. 重复

同一件事做了多遍。不同文件、不同名字，但干的是同一件事。

- 相同或高度相似的代码块出现在两个或更多文件中（复制粘贴后轻微修改）
- 同一概念有两个字段、两个函数、两个类型在表达
- 同一状态管理模式在多个组件/hook 中各自手写一遍，而非提取为共享 hook
- 同一工具逻辑（错误处理、列表比较、key 构建、数据转换）在多处内联实现
- 多个 hook 的订阅/创建样板代码高度雷同，仅 selector 不同
- 多个组件的生命周期编排（注册、订阅、清理）模式雷同

### 3. 过度抽象

拆了不该拆的，包了不该包的，层次多到改一处要动一堆文件。

- 一个子系统被拆成 10+ 个碎片文件，单个文件只有 1-2 个导出，但互相紧密耦合（改一个行为要同时改 3+ 个同子系统文件）
- 抽象层超过 3 层，中间层只做转发
- 为了"行数最小化"而拆分，导致文件间交叉引用密度不降反升
- 接口/类型层级过深（A extends B extends C），但每层只加了一个字段

## 执行方式

1. 先统计全仓库源文件的行数分布，找出超过 500 行的文件和顶层文件超过 20 个的目录，作为导航起点。
2. 对每个包，逐文件扫描上述三类问题。不要求覆盖全部文件，但要求覆盖全部包。
3. 发现问题时，给出：文件路径、行号范围、问题类别（冗余/重复/过度抽象）、严重程度（P0-P3）、一句话现状、修复方向。
4. 最后汇总：按文件分组的问题清单、出现最多问题的文件（跨类别热点）、可暂缓项、误报排除。
