nop-chaos-flux 是什么

  • 全称：NOP Chaos Flux，一个 Schema 驱动的前端运行时和渲染框架

    核心理念："Schema that executes, not just renders"

  • 定位：为 NOP 低代码体系提供下一代前端运行时基础设施

    不是应用框架，不是 UI 组件库

  • 围绕七个原语构建

    Template、ScopeRef、Value、Resource、Reaction、Capability、Host Projection

  • 表单、表格、对话框、设计器、验证等都是这七个原语之上的派生系统

高速生成下系统没有解体：效率只是副产品

稳定性的真正证据

  • 包边界越来越硬

    最终收敛为 compiler / action-core / runtime 三层分离

  • 结构越来越纯

    CompiledSchemaNode 等过渡结构被清除

  • 审计越来越苛刻

    经过 meta-review 和 re-verification 继续收敛

效率估算

实际投入：1 名开发者 + AI，6 周（1.5 人月）

传统估计：4-6 名高级前端工程师，3-4 个月 = 12-24 人月

效率倍数：约 8-16 倍

真正罕见的不是快，而是在这种快里系统没有解体。

核心问题：AI 生成是扩张力，没有收敛机制就会持续漂移

  • 高带宽生成：AI 能在极短时间内产出大量代码、测试、文档

  • 乐观偏差：AI 倾向于流畅地叙述"已经完成"，即使行为闭环并未真正建立

  • 上下文漂移：旧前提在长上下文里不断被复用，整组文件一起偏离真实基线

最危险的地方不是"写错一段代码"，而是局部都像对的，但整体在持续偏离。

局部都像对的，往往正是系统正在整体跑偏的时候。

我们的答案：把开发组织成一个受控收敛的动力系统

吸引子
架构基线 / owner doc
扩张力
AI 生成代码/文档/方案
收敛力
Plan / lint / audit / logs
基线更新
升格为下一轮吸引子
循环

关键前提：在当前主流模型的训练分布和偏置下，新吸引子通常不会由 AI 自己自然长出来。真正新的概念切分、边界定义和结构语言，仍然需要人先给出。

没有吸引子，AI 不是在开发系统，而是在高效采样错误解空间。

什么是动力系统，什么是吸引子

动力系统

状态随时间演化、且下一步依赖当前状态的系统。我们关心的不是某个静止点，而是轨迹。

吸引子 (Attractor)

状态空间中的一个稳定区域或结构——从足够近的地方出发的轨迹，最终会被它吸引并长期停留在它附近。

类比映射

动力学概念 软件工程对应
状态空间 所有合法实现状态的集合
吸引子 架构基线允许的结构空间
轨迹 每一轮 AI 生成 + 收敛的演化路径
混沌不等于随机；局部不可预测，不等于整体失控。

仓库真相带来的工程后果：生成和验收被真正拆开

传统开发

仓库更多是认知的外化；作者认知持有系统模型。

AI 深度参与后

仓库开始承担系统真相的载体。下一个 session 能重新加载的，不是作者脑中的完整模型，而是代码、diff、日志、测试和文档。

对应关系变化

机制 作用
test / lint / audit 测量
owner doc / plan / closure 约束
logs / history / bugs 轨迹记录
当作者不再持有完整模型，生成和验收就必须依赖仓库里的外部证据重新接起来。

这个模型和传统开发的关键区别

瀑布 敏捷 AI 深度参与
强蓝图 / 长流水线 弱蓝图 / 短流水线 动力系统 / 轨迹控制
先设计，再实现，再测试 迭代设计，快速反馈 生成速度超出人脑自然收敛速度
默认人类可同步把关 默认团队能跟上系统演化 生成与验收被拆开
按图施工 迭代施工 控制系统轨迹
再用流水线思维管理 AI 高速生成，只会把混乱组织得更整齐。

吸引子的工程实现 1：架构基线与文档权威链

核心约束

  • 七个原语闭合集

  • 编译优先

  • 数据/能力正交

  • 统一渲染器合约

  • 严格包边界

文档权威链

docs/index.md → 导航入口

docs/architecture/ → 定义现行基线

docs/analysis/ → 提供候选发现

docs/plans/ → 负责执行收口

docs/logs/ → 保存开发记忆

架构的价值不在于描述一切，而在于让错误结构无法合法存在。

吸引子的工程实现 2：保留澄清过程，而不只保留结论

  • 专门保留了 docs/discussions/

    讨论文档不只写最终结论，也保留澄清过程

  • 记录：原始表达 → AI 复述与提问 → 用户补充 → 纠正记录 → 最终总结

  • 后续 AI 不用凭空猜"为什么这样设计"，也能看到"为什么不能那样设计"

核心价值：对复杂系统来说，早期理解偏差往往比编码错误更致命。

扩张力的角色定义：AI 负责展开，不负责最终裁决

AI 主要承担

  • 代码生成

  • 文档整理

  • 方案展开

  • 审计候选生成

没有交给 AI 的

  • 架构裁决

  • 概念发明

  • 边界重定义

  • 方案淘汰

人机分工：人定义边界和收口标准，AI 高速扩张，多 agent 交叉验证把结果拉回基线，收敛规则由人工抽样校准。

收敛力 1：Plan 从"路线图"变成"收口机制"

  • Plan 在这里不是路线图，而是局部收敛器

    核心文档：docs/plans/00-plan-authoring-and-execution-guide.md

每个 Plan 至少要写清

Current Baseline
Goals / Non-Goals
Exit Criteria
Validation Checklist
Closure Audit Evidence
Plan 不是路线图，它是局部收敛器。

Plan 作为收敛力的实际效果

漂移模式 → Plan 机制

  • 计划过宽 → 拆 successor plan

    每个 plan 只负责一个可关闭的结果面

  • 接口存在 ≠ 行为落地

    closure audit 必须区分 interface presence 和 behavior landing

  • 自我宣布完成 → 独立 closure audit

    completed 必须来自独立审计

真实案例

Plan 145：closure/audit 后，将新确认的 domain-host projection / vocabulary convergence 跟进面分离为 Plan 146。把不同结果面硬塞在一起，是在制造假关闭。

Plan 143：closure assumption 被独立审计连续推翻，直到 2026-04-27 独立复核确认 live repo 全部过线，才真正允许关闭。

收敛力 2：开发日志作为外部化工作记忆

日志最少记录四件事

改了什么
为什么这么改
相关路径
下一步是什么

日志的隐藏价值——保留"为什么不这么做"：

• 某个收敛尝试回滚了，因为暴露了另一个 runtime regression
• 某个架构方向先只收敛文档语义，不强推 runtime 重写
• 某个能力先作为 docs-only alignment 落地，不把 scope 扩大

日志不是收尾材料，而是系统记忆。

验证体系：自动化引力场与验证顺序

根级质量门

  • check:react19

  • check:src-artifacts

  • check:oversized-code-files

  • check:i18n-keys

  • lint (eslint)

验证顺序

1. focused verification 确认当前切片

2. 修当前切片

3. workspace 级 typecheck/build/lint/test

高信号规则升到 error：react-hooks/exhaustive-deps、react-compiler、i18n 硬编码字符串

能靠机器提前失败的，就不要留给人类事后理解。

审计闭环：从多 Agent 发散到共识收敛

发散
18 subagent 并发
过滤
分类 / 证据检查
Meta-Review
重叠 / 冲突分析
共识确认
re-verification

关键规则：只有经过收敛的结论，才允许进入 owner doc、plan 或 backlog。

高质量审计，不是堆问题，而是尽快排除假问题。

长期收敛力：Debugger、Bug 记忆与完成判定

Bug 记录保存的不是 diff

而是排查路径：为什么难定位、先看了什么、排除了哪些假设、最后依据什么确认了问题。

nop-debugger 设计原则

按"AI 可消费"设计：以 live cid 为中心，暴露 automation-facing API，AI 和 E2E 优先读调试 API，不读截图。

"完成"的苛刻定义

  • live code path 真经过

  • focused/full verification 收口

  • 文档同步

  • leftover work 归属明确

  • 独立 closure audit

闭环如何实际运转：一个完整周期

1. 定义吸引子 owner doc 固定基线
2. AI 扩张 代码/方案/文档
3. 实时收敛 lint/check/typecheck
4. 切片收敛 focused verification
5. 记忆沉淀 开发日志
6. 全局收敛 full repo verification
7. 审计收敛 meta-review
8. 关闭判定 独立 closure audit
9. 吸引子更新 更新 owner doc
10. 下一轮 新基线继续扩张

完整实例：先有编译优先和包边界吸引子 → AI 大量实现编译/动作/运行时逻辑 → 审计发现运行时包里仍混有 compiler 与 action 责任 → 最终拆出 flux-compiler 和 flux-action-core → 文档和日志把三层职责重新固定为新基线。

这不是流程图，是系统的呼吸方式。

时间线

三个落地案例

结构收敛：CompiledSchemaNode 被消除，模板实例化成为真实基线

边界收敛：三层拆分，把"能跑"进一步收敛为"边界正确"

审计收敛：deep audit → meta-review → unresolved-only recheck

42 天真正了不起的不是产出快，而是在这种速度下，架构越来越清晰。

吸引子为什么能起作用——它解决了 AI 的三个核心偏差

AI 偏差 表现 收敛机制
乐观偏差 "看起来做完了" Plan 的 Exit Criteria + Closure Audit
上下文漂移 "旧前提不断被复用" 开发日志 + 文档权威链
局部正确全局漂移 "在错误的层做了正确的修复" 架构基线 + owner doc

吸引子模型的核心价值：不依赖 AI 自己发现问题，而是构造外部的力场，让漂移自动被修正。

重点不是让 AI 不犯错，而是让错误不能长期存活。

这套方法的本质：Audit-and-Closure Driven Development

  • 项目最终形成的模式：审计-收口驱动开发

    把实现和完成判定明确拆开：人先定义吸引子，再让扩张和收敛交替发生

  • 把文档、discussion、plan、AI 实现、audit、closure、logs 组织成持续纠偏闭环

  • 默认承认：当前主流 AI 擅长在既有知识边界内高效执行

    但受训练分布和偏置影响，很难自己长出真正新的架构观念
不要指望 AI 自己找到正确设计，要先定义一个不会把系统带偏的吸引子。

可复制结构与核心结论

你很可能必须回答五个问题

谁定义方向？
冲突时信谁？
怎么发现漂移？
怎么保留记忆？
怎么判定完成？

这五个问题都有答案，你就有了闭环；没有答案，系统就很容易在高速推进中偏掉。

可复制的不是表面做法，而是在当前主流 AI 条件下，人先定义新吸引子，AI 再围绕它高效收敛。

真正边界：当前主流 AI 擅长收敛执行，不擅长原创架构

AI 最适合承担的

  • 实现

  • 展开

  • 补全

  • 验证

  • 整理

AI 最不适合单独承担的

  • 核心概念发明

  • 边界重定义

  • 第一性原理层面的架构突破

例子：ActionScope 与 Data Scope 分离并采用词法作用域，这类方案不是让 AI 自由采样平均解能自然长出来的，而必须先由人定义吸引子。

当前主流 AI 最擅长把已有知识执行到极致，但很难自然长出真正新的架构观念。

附录：吸引子模型的关键术语速查

动力学概念 软件工程对应 仓库实例
吸引子 (Attractor) 架构基线——系统应该收敛到的状态 docs/architecture/*.md

记忆 (Memory) 跨 session 可追溯的上下文 docs/logs/、docs/bugs/
