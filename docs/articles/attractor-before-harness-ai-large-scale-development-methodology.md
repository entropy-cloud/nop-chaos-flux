# Attractor Before Harness: AI 大规模开发的方法论

> 在 AI 大规模参与开发的系统里，第一性的问题从来不是“如何约束 AI 的行为”，而是“系统应该向哪里长期收敛”。
> 只有当这个方向被定义清楚之后，harness、guardrail、verification、audit、closure 这些机制才真正有意义。否则，它们只是在更高效地维持一套错误基线。

## 一、为什么现在一谈 AI 工程，大家会先想到 Harness

现在关于 AI 辅助开发的主流话语，最常出现的词是这些：

- `guardrail`
- `verification`
- `review`
- `feedback loop`
- `agent harness`

这很自然。AI 生成速度快，出错也快，于是最直觉的反应就是：给它加护栏、加验证、加流程、加审计，把它管住。

问题在于，这套直觉虽然合理，却默认了一个前提：

**系统正确的方向已经知道了。**

一旦默认这个前提，后面的思路就顺理成章了：

- 怎么限制偏离
- 怎么尽早失败
- 怎么让 review 更严
- 怎么让 agent 不胡来

这套语言在小任务里通常够用。一个脚本、一页 CRUD、一个局部 bug 修复，大多数时候你已经知道“对的结果长什么样”，剩下的确实主要是执行和验证问题。

但大型系统不是这样。大型系统真正困难的地方，往往不是“怎么防止越线”，而是“到底哪条路才是长期正确的结构”。

这就是为什么，AI 大规模开发的第一性问题不是 harness，而是 attractor。

## 二、为什么这个问题在 AI 时代会突然变成主问题

传统开发当然也会有 bug、返工、误解、架构漂移。但 AI 把这些问题放大了，原因不是 AI 更笨，而是它把系统展开状态空间的速度提高了太多。

人类工程师写一个功能，通常是慢慢来的：想一想，写一点，看一眼，再改，再跑，再解释。AI 不是这样。AI 往往会在极短时间内同时给你：

- 一段代码
- 一套类型
- 几个测试
- 一段说明
- 一句“已经完成”的总结

最危险的地方不在于它会胡说八道。最危险的地方在于：**它给出的东西往往局部都很像对的。**

接口有了，类型有了，测试也补了，文档也更新了，解释还特别流畅。你如果只看这些局部信号，会非常容易产生一种错觉：系统正在稳步前进。

但真实情况可能是，系统整体已经开始偏航了。

这就是 AI 时代最典型的失稳模式：

**不是某一步明显做错，而是每一步都像合理，整体却持续漂移。**

规模一上来，这个问题就会突然变得尖锐。一个小脚本、一页 CRUD、一个短期需求，即使中间有些漂移，人也能很快靠局部理解把它拽回来。但一个长期演化的大仓库不是这样。它会同时出现：

- 多 session
- 多 agent
- 多轮计划
- 多层文档
- 多个局部成功信号

于是“局部都像对的”这件事，就不再是小毛病，而会变成系统性风险。

在 `nop-chaos-flux` 的历史里，Plan 76 是一个很典型的例子。那次尝试移除 `array-editor` / `key-value` 的本地状态镜像，结果直接打出 11 个测试失败。暴露出来的不是一个孤立 bug，而是更麻烦的事实：**测试本身也已经绑定在旧实现时序上了。**

换成更通用的场景也一样。团队想把“数据获取”和“页面渲染”拆开，AI 很快补齐了接口、类型和测试，CI 也全绿；但如果这些测试仍然默认旧的耦合时序，那么系统只是把旧结构包上了一层新名字。表面在重构，实际仍在原地打转。

一旦你看见这种现象，就会发现，传统那套“哪一步违规了”“哪条规则没过”“哪个 PR 有问题”的语言已经不够用了。因为问题不只是点上的错误，而是轨迹本身错了。

这时候，更合适的语言不是流程治理语言，而是动力系统语言：

- AI 提供扩张力
- 工程机制提供收敛力
- 仓库历史体现的是轨迹
- 真正的问题是，这条轨迹是不是一直贴着正确的稳定结构在走

## 三、仓库开始承担系统真相：为什么 Harness 会变得必要

很多人一上来就谈 harness，其实是直觉上已经意识到：光靠人脑里那点连续认知，已经接不住 AI 的生成速度了。

这背后真正的变化是：**仓库开始承担系统真相的载体。**

传统开发里，仓库更多是认知的外化。作者脑中有一套相对连续的系统模型，代码、文档、提交记录只是把这套模型表达出来。AI 深度参与以后，这个前提开始失效。下一个 session 能重新读取的，不是作者脑中的完整意图，而是代码、diff、日志、测试和文档。

这会带来一个直接的工程后果：

**生成和验收被真正拆开了。**

因为生成动作可以在同一个上下文里高速完成，但验收已经不能再依赖那个生成上下文本身。你必须回到仓库里的外部证据，重新判断：

- 行为是不是真的落地了
- 当前基线到底是什么
- 哪些材料是权威的
- 这次“完成”是不是只是一种完成感

这就是为什么 harness 不是附属物，而是必要结构。它不是“多加几个流程节点”，而是在做一件更本质的事：**把生成和验收重新接起来。**

从这个角度看，很多看起来像工程治理的东西，角色都会重新变化：

- `test / lint / audit` 更像测量
- `owner doc / plan / closure` 更像约束
- `logs / bugs / discussions` 更像轨迹记录和外部化记忆

所以 harness 的必要性，不是来自“AI 需要更多管理”，而是来自“仓库真相条件下，生成和验收必须被重新组织”。

## 四、先排除三种最容易发生的误解

在继续往下讲之前，最好先把三种最容易发生的误解直接排掉。

第一，**吸引子不是边界**。

边界回答的是“什么不能做”，吸引子回答的是“系统应向哪里长期收敛”。边界定义禁区，吸引子定义稳定区。两者不是强弱关系，而是层级关系。

第二，**吸引子不是更强的护栏**。

很多人一听到这里，会自然把它理解成“原来是更严格的治理、更密的约束、更强的审计”。这仍然是把第一性问题降格了。护栏属于执行层，吸引子属于方向层。

第三，**吸引子不是控制目标的另一种说法**。

控制目标这个词听起来很接近，但它默认控制已经成立了。这里真正要强调的是：如果没有先定义 attractor，那么所谓控制就没有目标，harness、guardrail、verification、audit 这些机制都还没有统一含义。

所以顺序必须保持为：

**状态空间 -> 吸引子 -> 轨迹 -> 控制**

这个顺序一旦被打乱，整套方法就会重新掉回“怎么管”而不是“向哪里收敛”的旧语言里。

## 五、什么是 Attractor

说到这里，先把本文最关键的词说清楚：`attractor`。你可以先把它理解成一句人话：**系统长期会被拉回去的正确结构。**

在动力系统里，attractor 指的是状态空间中的稳定区域。轨迹只要从足够近的地方出发，最终就会被持续拉回，并长期停留在它附近。

把这个概念放到软件工程里，attractor 指的不是某个具体功能，也不是某个单独模块的完成态，而是：

**系统长期应该收敛到什么结构。**

这里最容易犯的第一个错误，就是把 attractor 说成 boundary：

- `boundary` 回答的是：什么不能做
- `attractor` 回答的是：系统应向哪里长期收敛

边界是局部约束语言，吸引子是系统演化语言。边界定义禁区，吸引子定义稳定区。这不是同义词，也不是“更强边界”和“更弱边界”的关系。

更隐蔽的错误，是把 attractor 理解成一张已经完全画好的终局蓝图，好像只要先把最终状态写清楚，后面就只是执行问题。

也不是。

所以，attractor 不是一张画到最后一个像素的终局蓝图，也不是一个孤零零的终点。它更像几条真正抓住系统一阶结构的硬约束：职责怎么分，边界怎么立，哪些变化会把系统重新拖回旧基线。

只要这些约束还在，局部实现可以变化，系统整体仍会朝同一个方向收敛。

如果借一个数学类比，这更像“代数方程定义代数流形”，而不是“枚举所有合法点”。方程并不列出流形上的每一个点，但它定义了什么样的点可以长期存在。工程里的 attractor 也是这样。

## 六、在这个仓库里，Attractor 首先是 `docs/architecture/`

如果 attractor 只停留在抽象层面，那它仍然没有真正的工程意义。真正关键的问题是：在仓库里，什么东西首先承担 attractor。

对 `nop-chaos-flux` 来说，这个答案并不模糊。首先承担 attractor 的，不是泛泛的“架构意识”，而是 **`docs/architecture/` 中带 precedence 的 owner-doc 体系**。

这也是这个仓库最容易被误读、但又最关键的一点。

很多人会自然地说：类型、架构文档、plan、verification、logs 都很重要，它们一起定义了方向。

这种说法比“多加点 guardrail”更接近正确，但它仍然把层级拉平了。真正更硬的说法是：

1. 首先是 `docs/architecture/` 下的规范定义
2. 然后才是 plan、verification、audit、logs 等围绕它组织的收敛机制

在 `docs/architecture/` 内部，这种“方程层”也不是平铺的，而是有明确 precedence 的：

- `docs/architecture/README.md` 负责说明 architecture hierarchy 和 reading order
- `flux-design-principles.md` 负责方向层，解释设计意图和稳定原则
- `frontend-programming-model.md` 负责顶层规范层，定义 primitive identity、macro boundary 和 hard invariants
- `flux-core.md` 负责当前 codebase-wide baseline
- narrower architecture doc 再在各自主题内定义局部 contract

在 `nop-chaos-flux` 里，这个 attractor 也不是抽象的“正确架构”，而是由少数高价值不变量共同定义出来的：七个原语闭合集、编译优先的流水线、Template/Instance 分离、Data/Capability 正交、统一 renderer/hook contract，以及 `flux-core -> flux-formula -> flux-compiler -> flux-action-core -> flux-runtime -> flux-react` 的依赖方向。

它们不是并列的治理材料，而是 `docs/architecture/` owner docs 固定下来的结构方程。架构的价值不在于描述一切，而在于让错误结构无法继续合法存在。

把话说得再硬一点：**在这个仓库里，首先定义 attractor 的不是 plan、lint、audit、logs，而是 `docs/architecture/` 下带 precedence 的 owner-doc 体系。其他机制都在它之后。**

也正因为如此，这个仓库里很多真正重要的收敛动作，最后都会表现成“某个旧结构被排除出合法状态空间”。

`CompiledSchemaNode` 被最终移除，就是一个典型例子。它不是单纯的重构清理，而是说明：模板/实例分离这条新基线成立之后，旧的中间结构虽然还能工作，但已经不再属于正确结构，于是被排除出去。

同样，`flux-compiler` / `flux-action-core` / `flux-runtime` 三层拆分也不是“多拆了两个包”这么简单，而是在说明 attractor 变得更精确了：系统不只是“能跑”，而是被进一步收敛到更稳定的职责结构。

## 七、什么是 Harness

如果 attractor 解决的是“方向是什么”，那么 harness 解决的就是：

**如何通过局部信号，持续地测量、纠偏、更新系统轨迹。**

这里说的 harness，不是狭义的测试 harness，而是更广义的执行支架。它通常包括：

- 上下文路由
- 实现与验收分离
- 计划与关闭条件
- 验证机制
- 审计机制
- 诊断工具
- 外部化记忆

换句话说，harness engineering 关心的是：

- AI 该读什么，不该读什么
- 哪些结论能进入当前基线，哪些只能停留在探索层
- 什么算完成，谁来判定完成
- 漂移如何被尽早发现
- 同一个错误如何不在新 session 中重复出现

所以 harness 不是“几条测试规则”，也不是“再加点 lint”。它是一整套让局部信号持续作用于系统轨迹的工程支架。

放到这个仓库里，至少可以看到五层很清楚的 harness：

### 1. 路由 harness

`docs/index.md` 和 `docs/architecture/README.md` 决定：碰到什么问题先读什么，什么是 current baseline，什么只是 analysis、plan 或 history。

这层的作用不是导航方便，而是在控制进入问题空间的入口。

### 2. Plan harness

`docs/plans/` 解决的不是“系统是什么”，而是“这一轮扩张如何收口”。

`current baseline`、`goals / non-goals`、`exit criteria`、`validation checklist`、`closure audit evidence` 这些字段定义的是局部轨迹怎样才能合法闭合。

Plan 145 在 closure/audit 之后把新确认的跟进面拆到 Plan 146，Plan 143 的 closure assumption 被 fresh audit 连续推翻，直到 live repo 真正过线后才允许关闭，这些都说明了一点：**plan 在这里不是路线图，而是局部收敛器。**

### 3. Verification harness

`lint`、`check`、`typecheck`、`build`、`test` 解决的不是方向定义，而是尽量把高频、明确、可自动化的偏离检测提前到机器层。

这一层不是方程本身，而是把方程投影成早期失败条件。

### 4. Audit harness

并非所有偏离都能被自动化规则抓住。更高层的语义漂移、结构偏差、假完成、局部自洽但整体失真的问题，仍然需要独立审计。

所以审计在这里本身也是一个闭环：发散发现、过滤冲突和伪问题、回看 live repo 再确认。高质量审计不是堆出更多问题，而是尽快排除不成立的问题。

在这个仓库里，最关键的 harness 规则之一就是：不要让同一个上下文既做实现，又做完成判定。完成必须由 fresh session 或独立审计重新回看 live repo；Plan 143 和 Plan 145 的 closure 之所以有意义，正是因为“完成”不是实现者自报，而是独立收敛裁决。

### 5. Memory harness

`docs/logs/`、`docs/bugs/`、`docs/discussions/` 构成了跨 session 的外部化记忆。

这一层特别重要，因为对 AI 来说，“为什么不能那样理解”本身也是系统记忆的一部分。只保留最终结论还不够，很多时候还必须保留：

- 哪个前提已经被证伪
- 哪条路径已经证明会发散
- 哪种术语翻译会把问题降格
- 哪个“已完成”判断后来被 live repo 推翻

没有这层记忆 harness，系统每次都会丢失一部分历史势能。

但这还不是一个静态结构。闭环的最后一步，不只是把轨迹拉回去，还包括把那些被反复证明有效的新约束写回 owner doc，升格为下一轮 attractor 的一部分。也就是说，真正的闭环不是“定义一次 attractor，然后永久执行 harness”，而是“定义 attractor -> 扩张 -> 纠偏 -> 更新 attractor -> 再扩张”。`flux-compiler` / `flux-action-core` / `flux-runtime` 的三层拆分，就是 attractor 被实践校正后再继续扩张的例子。

## 八、为什么不仅是 Attractor Before Harness，而且是 Attractor Before Control

说到这里，其实还需要再把层级说硬一点。

因为真正容易发生的误读，不只是把 attractor 说成 boundary，还会把 harness、guardrail、verification、feedback，甚至 control 语言本身抬到太高的位置。

更准确的层级应该是：

**状态空间 -> 吸引子 -> 轨迹 -> 控制**

这个顺序不能反。

因为：

- 没有 attractor，就没有控制目标
- attractor 定义错了，控制越强，只会让错误状态更稳定
- harness、guardrail、verification、audit、feedback 都属于 attractor 已经被定义之后的控制层语言

把顺序说得再白一点：先有“系统应该收敛到哪种结构”，才谈“怎么持续把它拉回去”。前者是 attractor，后者才是 harness、guardrail、feedback 这些控制手段。方向没定，控制越强，跑偏越稳。

所以 `Attractor Before Harness` 的更完整说法其实是：

- `attractor` 定义长期正确结构
- `trajectory` 是系统真实发生的演化路径
- `harness` 是通过局部信号持续影响轨迹的控制支架
- `guardrail` 只是 harness 的一个子集，负责把明显偏离尽早失败掉

这也解释了为什么很多表面上看起来很像的做法，其实不在同一个层级上。

`plan / lint / audit / closure` 看上去像方法本身，但它们其实都只是测量与纠偏手段，不是这套方法的第一性定义。

## 九、为什么 AI 总会把它理解错

如果这个模型这么关键，为什么 AI 会反复把它讲成 boundary、guardrail 或 control 语言？

原因其实不复杂，主要有三条。

第一，AI 会本能地把陌生高阶概念压缩成熟悉工程语言。`吸引子` 不是常规软件工程教材里的高频词，所以它很容易被自动翻译成更熟悉的词桶：边界、护栏、约束、控制目标。

第二，AI 会默认把可执行的机制当成更本质的东西。`lint`、`test`、`audit`、`closure` 这些机制更容易落地，于是它会反过来把 attractor 理解成这些机制背后的另一种说法。

第三，AI 会为了通俗化而主动降格第一性概念。把问题压缩成“先定边界，再高速生成，再用验证拉回正轨”确实更容易懂，但代价是把“先定义长期正确方向”偷换成了“执行层如何防错”。

所以，真正的困难不仅是把 attractor 讲出来，还要防止它在解释过程中被重新翻译回旧语言。

## 十、为什么新 Attractor 通常不是 AI 自己长出来的

到这里还有一个问题必须说清楚。

既然我们已经有了 harness，AI 能不能在高速迭代中自己慢慢找出新的 attractor？

至少在当前主流模型的训练分布和偏置下，通常不能指望这件事自然发生。

更准确地说：

- 当前主流 AI 很擅长围绕既有 attractor 高速展开和收敛
- 但它通常会回到自己见过的平均方案
- 真正新的概念切分、边界重定义、结构语言，通常仍然需要人先给出

这不是在抱怨 AI 不够聪明，而是在承认一个工程现实：

**不能把新 attractor 的发明权默认外包给 AI。**

这也是为什么在大型框架开发里，人和 AI 的分工不能简单理解成“AI 写代码，人做 review”。更真实的分工是：

- 人定义新的 attractor
- AI 围绕既定 attractor 高速展开
- harness 持续把轨迹压回去

像 `ActionScope` 和 `Data Scope` 分离、词法作用域这类真正改变系统结构语言的东西，并不是让 AI 自己自由采样平均方案就会自然长出来的。它们更像是新 attractor 先被提出，然后 AI 才能在这个新基线上做大规模展开。

## 十一、结语

AI 大规模开发真正困难的地方，不是让 AI 多写一点代码，而是让系统在高速扩张中仍然向正确结构收敛。

所以面对 AI 大规模开发，真正应该先问的不是“测试怎么补、流程怎么卡、agent 怎么管”，而是“这个系统最终要被什么结构拉住”。

这个问题答不清，后面的 harness 只会放大噪声；这个问题答清了，harness 才会把每天的局部修改，慢慢压回同一个长期方向。

AI 时代真正稀缺的，不是更多会写代码的 agent，而是能先回答“系统该向哪里收敛”的人。
