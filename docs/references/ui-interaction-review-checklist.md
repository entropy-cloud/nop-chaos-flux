# UI 交互检查规范

## 1. 目的

本规范用于发现和减少 UI 中“明显不合常规”的交互问题，例如：

- 次级面板不能收起或调宽，持续挤压主工作区。
- 控件位置和它实际作用的区域不就近。
- 高频操作路径过深，日常使用成本异常。
- 当前状态、保存状态、错误状态不可见，用户只能猜。

它适用于：

- 后台管理页
- 设计器、编辑器、工作台
- 表单、列表、弹窗、抽屉
- 工具栏、多栏布局、复杂面板式界面

这不是视觉风格规范，也不是某个产品的专项需求文档；它是一般项目都可以使用的通用交互质量基线。

## 2. 这类问题一般怎么处理

成熟团队通常不会把这类问题只当成“审美问题”，而是按下面的方式处理：

1. 用固定检查清单替代临场感觉。
2. 用核心任务场景走查，而不是只看静态稿。
3. 给问题分严重级别，明确哪些必须阻断发布。
4. 对重复出现的问题抽象成共享布局组件或组件约束，而不是每个页面各做一套。
5. 把检查动作前置到设计评审、PR 自查、提测和回归，而不是等上线后靠反馈发现。

如果同类问题在不同页面、不同组件中反复出现，通常说明缺的不是“多提醒一次”，而是：

- 缺少统一交互基线
- 缺少可复用的布局壳组件
- 缺少 PR 或 QA 的显式验收项

## 3. 严重级别

| 级别 | 定义 | 处理建议 |
| --- | --- | --- |
| P0 | 主任务无法完成，或主工作区被明显挤压到难以使用 | 阻断合并或发布 |
| P1 | 任务能完成，但明显违背常规，频繁增加认知负担或误操作风险 | 进入当前版本修复范围 |
| P2 | 一致性、可发现性、效率存在问题，但不阻断主要任务 | 进入近端迭代 |
| P3 | 细节优化或体验打磨项 | 视资源安排 |

对设计器、报表、IDE、流程编排、画布类工作台 UI，面板伸缩、主工作区优先级、工具控件就近性通常默认按 `P0/P1` 处理，而不是当作普通 polish。

## 4. 从复杂应用归纳出的交互基本原则

结合设计器、会话工作台、列表/详情页、以及带 `sidebar` / `tabs` / `fullscreen` 的后台应用壳层，可以把反复出现的交互问题归纳为以下原则。

评审时不要只盯某一个具体问题，而要先判断它违反了哪条原则。原则找对了，后续才能举一反三。

### 4.1 主任务优先，辅助区让位

页面首先要回答“用户此刻最主要的任务是什么”。画布、详情编辑区、会话正文、大表格通常是主任务区；`palette`、`inspector`、筛选栏、上下文区、日志栏、帮助栏通常是辅助区。

当空间不足时，应该优先牺牲辅助区，而不是把主任务区压缩到不可用。

评审问法：

- 当前页面的主任务区是哪一块？
- 当屏幕变窄或多面板同时打开时，谁应该先让位？
- 用户是否可以自己收起、调尺寸、恢复或重置辅助区？

### 4.2 上下文连续，状态不要靠记忆

用户在任何时刻都应知道：自己在哪个页面、正在操作哪个对象、当前处于什么模式、是否有未保存修改、当前筛选/排序/分页/选中状态是什么。

切换标签、打开面板、加载更多、移动端切换为 `Sheet` / `Drawer` 后，上下文、选中对象、滚动位置、当前会话不应无故跳变。

评审问法：

- 用户离开 30 秒再回来，能否立刻看懂当前上下文？
- 当前对象、当前模式、当前脏状态是否在界面上可见？
- 加载、切换或返回后，界面是否保持连续，而不是让用户重新找位置？

### 4.3 控件就近，作用对象映射清楚

控制某个区域的按钮，应尽量放在该区域的标题区、边缘或主工具条内。局部对象的局部操作应靠近对象本身，全局操作才放在页头或全局工具栏。

用户不应该先在 A 区看内容，再跑到远处的 B 区找控制它的按钮。

评审问法：

- 这个按钮控制的是哪一块区域、哪个对象？它是否就近？
- 收起、展开、关闭、返回、保存这些动作的空间归属是否清晰？
- 图标方向、弹出方向、面板方向是否彼此一致？

### 4.4 高频直达，复杂度渐进展开

80% 的高频任务应放在第一层。低频、高风险、专家级配置可以放入二级面板、更多菜单、`advanced` 区域或上下文区。

快捷键、双击、右键、悬浮工具条适合做效率加速，但不应成为唯一入口。

评审问法：

- 高频动作是否需要多次点击才能到达？
- 低频复杂项是否挤占了默认界面的大量空间？
- 新手能否找到功能，熟手又能否快速完成任务？

### 4.5 一个动作只表达一个含义，避免交互冲突

很多明显“不顺手”的问题，本质不是样式问题，而是事件语义冲突。例如：整行可点击同时又内嵌复选框和更多菜单；拖拽、单击、双击相互打架；输入框里按 `Enter` 与页面快捷键互相抢控制权。

复杂界面里必须显式处理交互边界。

评审问法：

- 点击、双击、拖拽、右键、快捷键、`hover` 之间是否互相冲突？
- 行点击是否会误伤 checkbox、menu、inline action？
- 输入组件聚焦时，全局快捷键、发送快捷键、提交逻辑是否会劫持用户输入？

### 4.6 反馈及时，结果可感知

加载中、保存中、已保存、流式输出、失败、禁用、空状态、无选中状态都必须可见。用户不应靠猜测来判断“系统有没有收到我的操作”。

对于异步操作，除了结果反馈，还要避免重复点击、重复提交和无响应错觉。

评审问法：

- 这个操作进行中时，用户能否看出系统正在处理？
- 成功、失败、停止、中断的结果是否明确？
- 禁用态是否只显示“不能点”，却不说明原因？

### 4.7 数据安全边界清晰，修改应可恢复

只要用户付出了输入、编辑、拖拽、排序、配置等成本，就必须考虑“离开时怎么办、误操作时怎么办、回滚时怎么办”。

不同页面可选的手段不同，但至少要有明确边界：未保存离开保护、撤销重做、区块恢复、草稿、确认对话框、版本回退等。

评审问法：

- 路由跳转、标签切换、刷新、关闭弹层时，未保存修改会不会静默丢失？
- 高成本操作是否有撤销、恢复或明确确认？
- 恢复的作用范围是否清晰，用户能否知道自己会回退哪一部分？

### 4.8 多端等价，不做桌面界面的压缩版

桌面端能通过侧栏完成的任务，在移动端应有等价入口，例如改成 `Sheet`、`Drawer`、底部面板或分步流，而不是简单把桌面布局硬挤窄。

同样，不能把关键能力做成纯 `hover` 或纯鼠标路径。

评审问法：

- 移动端如何打开桌面侧栏、属性区、上下文区？
- 关键路径能否用键盘完成？
- 纯 `hover` 能力在触屏和键盘环境中是否有替代方式？

### 4.9 全局壳层与局部页面要分层，不要互相打架

后台应用常有全局侧栏、标签页、全屏、移动端抽屉、系统级快捷键。这些壳层能力必须服务页面，而不是破坏页面。

页面内部的编辑语义、脏状态、快捷键范围，要和壳层的导航语义明确分层。

评审问法：

- 页面的离开保护能否覆盖壳层导航、标签切换、返回操作？
- 壳层快捷键是否会在输入框聚焦时误触？
- 页面标题、当前 tab、返回路径是否与用户实际上下文一致？

### 4.10 偏好可以记忆，但必须可恢复

布局宽度、面板折叠状态、展开菜单、分页大小、工作区全屏状态等偏好可以记忆，但不能把用户困在坏状态里。

所有持久化的 UI 状态都应有边界、限幅和恢复默认值的能力。

评审问法：

- 记忆下来的布局值是否有最小/最大边界？
- 当历史偏好不再适配新版本界面时，是否仍可恢复？
- 是否提供一键 reset，而不是要求用户手工清本地状态？

## 5. 高频风险场景

| 场景 | 常见风险 | 优先检查 |
| --- | --- | --- |
| 三栏工作台 / 设计器 | 辅助面板固定挤压主画布；收起入口不就近；拖拽/选中/双击冲突 | 主任务优先、控件就近、交互冲突 |
| 会话 / 聊天 / 日志工作台 | 会话列表和上下文区抢空间；发送/停止/换行语义混乱；历史加载后滚动跳变 | 上下文连续、反馈及时、多端等价 |
| 列表 / 表格 + 整行点击 | 行点击与 checkbox、菜单、行内按钮冲突；筛选排序分页状态不可见；批量与单行操作混杂 | 作用对象映射、交互冲突、状态可见 |
| Master-detail / 大表单 | 区块脏状态不透明；恢复边界不清；弹窗/抽屉关闭后丢输入 | 数据安全、上下文连续、可恢复性 |
| 带 App Shell 的后台应用 | `sidebar`、`tabs`、`fullscreen`、页面快捷键互相干扰；移动端入口不清 | 壳层分层、偏好恢复、跨端等价 |

## 6. 通用检查清单

### 6.1 结构与布局

- [ ] 主任务区和辅助区边界清晰。
- [ ] 默认布局把最大、最稳定的空间留给主任务区。
- [ ] 影响主任务的辅助区可收起、可调尺寸、可恢复。
- [ ] 用户调整后的布局可记忆，但也可 reset。
- [ ] 不存在双重滚动、隐藏滚动或内容裁切后无恢复路径。

### 6.2 上下文与状态

- [ ] 当前页面位置、活动对象、活动会话/选区、当前模式可见。
- [ ] 当前筛选、排序、分页、选中数、保存态、加载态、流式态可见。
- [ ] 无选中、空结果、空数据、空会话时有明确下一步指引。
- [ ] 切换、加载、打开/关闭面板后，上下文和滚动位置不会无故跳变。

### 6.3 操作组织与冲突隔离

- [ ] 高频操作在第一层可达，低频复杂项进入 `advanced` 或 overflow。
- [ ] 局部操作靠近局部对象，全局操作放在页头或全局工具栏。
- [ ] 行点击不会与 checkbox、dropdown、行内按钮冲突。
- [ ] 拖拽、点击、双击、右键、快捷键、`hover` 的语义不互相覆盖。
- [ ] 输入组件聚焦时，全局快捷键和提交逻辑不会劫持用户输入。
- [ ] 破坏性操作与高频安全操作有清晰区隔。

### 6.4 反馈与数据安全

- [ ] 异步操作有进行中与结果反馈，并避免重复提交。
- [ ] 禁用态在必要时说明原因。
- [ ] 未保存修改在刷新、关闭、路由切换、tab 切换、关闭弹层时受到保护。
- [ ] 高成本修改具备撤销、恢复、版本回退或至少明确确认。
- [ ] 区块级编辑可以看出脏状态范围，恢复边界清晰。

### 6.5 跨端与可访问性

- [ ] 笔记本宽度、桌面宽度、浏览器 `125%` 到 `150%` 缩放下主任务可完成。
- [ ] 移动端对关键侧栏、属性区、上下文区提供等价入口。
- [ ] 关键路径可通过键盘完成，焦点可见，弹层焦点管理正确。
- [ ] 纯 `hover` 能力在触屏或键盘环境下有替代路径。
- [ ] 调尺寸拖柄、折叠按钮、纯图标按钮具备清晰可发现性和可访问名称。

### 6.6 壳层协同

- [ ] `sidebar`、`tabs`、`fullscreen` 等壳层能力不会打断页面内部任务。
- [ ] 页面脏状态与离开保护能覆盖壳层导航行为。
- [ ] 壳层级布局偏好有校验、限幅和 reset 默认值。
- [ ] 当前标题、当前 tab、返回路径与用户实际上下文一致。

## 7. 最低发布门槛

出现以下任一情况，建议至少按 `P1` 处理；对设计器、工作台、编辑器类产品通常应按 `P0` 处理：

- 主任务区被固定布局压缩到明显不可用。
- 当前对象、当前模式、未保存状态不可见，容易导致误编辑。
- 行点击、拖拽、双击、快捷键等交互存在明显冲突，容易误操作。
- 未保存内容会在刷新、切页、切 tab、关闭弹层时静默丢失。
- 关键能力只支持鼠标、`hover` 或桌面布局，没有移动端/键盘等价路径。
- 异步操作没有进行中或结果反馈，用户无法判断系统状态。
- 记忆化布局把用户困在坏状态里，且没有恢复默认值的入口。

## 8. 推荐评审流程

1. 先定义主任务和壳层边界。
说明页面主任务是什么，哪些属于辅助区，哪些属于全局壳层能力。

2. 走查高频路径和离开路径。
至少真实走一遍创建、编辑、保存、取消、删除、切换、返回、刷新、报错等路径。

3. 画出冲突矩阵。
把点击、双击、拖拽、右键、键盘、`hover`、移动端入口逐项对照，确认没有互相抢语义。

4. 做边界场景检查。
至少检查一次笔记本宽度、浏览器缩放、移动端入口、键盘路径和空状态。

5. 决定是局部修补还是系统治理。
如果同类问题反复出现，应升级为共享壳组件、布局规范、快捷键规则或 PR 验收项，而不是继续单页修补。

```md
- 场景:
- 违反原则:
- 当前行为:
- 期望常规:
- 影响:
- 严重级别:
- 局部修复建议:
- 是否需要沉淀为共享组件或规范:
```

## 10. 常见反模式

- 辅助面板默认全开，且既不能收起也不能调尺寸。
- 当前选中对象、当前模式、当前脏状态只存在于内部状态里，界面上看不出来。
- 控件位置和它控制的区域不就近，用户必须来回找入口。
- 整行点击、复选框、更多菜单、行内按钮同时存在，但没有事件边界。
- 拖拽、单击、双击、快捷键混在一起，导致误触和误操作。
- 高频操作藏在二级菜单里，低频配置却长期占据主界面空间。
- 只在 `hover` 时暴露关键能力，没有触屏和键盘替代路径。
- 只对浏览器刷新做未保存保护，却不处理应用内导航、tab 切换或弹层关闭。
- 记忆用户布局偏好，却没有限幅和 reset 入口。
- 全局壳层快捷键或全局按钮打断页面内部编辑流程。

## 11. 落地建议

- 对复杂工作台 UI，优先建设统一的 `Workbench Shell`、`Split Panel`、`Inspector Shell`、`Filter Shell` 等共享壳组件。
- 对整行点击的表格/列表，建立统一的行内交互边界约定。
- 对高成本编辑页，统一提供脏状态、离开保护、区块恢复、撤销/确认策略。
- 对全局快捷键建立作用域规则，默认跳过输入类控件。
- 在 PR 模板、提测单、设计评审中纳入本规范，而不是把它当成上线后的体验返工清单。

## 12. 适用边界

本规范提供的是通用基线，不替代具体业务的领域规则。

具体产品可以在此基础上追加更严格的专项规范，例如：

- 流程设计器面板与画布交互规范
- 表格与 master-detail 交互规范
- 会话工作台输入、流式输出与上下文区规范
- 应用壳层导航、tabs、fullscreen 与快捷键规范
