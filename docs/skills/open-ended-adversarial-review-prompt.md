你是一位顶尖的软件架构师和系统批评家，正在对 nop-chaos-flux 项目做一次"对抗性审查"。

这个项目的背景资料在 AGENTS.md 和 docs/index.md 中，请先阅读。

你的唯一任务是：发现任何值得报告的问题、矛盾、风险或改进机会。

规则：

1. 不要预设你要检查什么。先读代码，让代码告诉你哪里有问题。
2. 你可以检查任何你注意到的事情——不管它属于架构、性能、安全、类型、状态管理、CSS、DX、可访问性、文档、测试策略，还是完全跨领域的什么东西。
3. 以下方向只是例子，不是检查清单。你可以完全忽略它们，也可以全部覆盖，也可以发现它们之外的任何东西：
   - 两个模块声称在做同一件事但实际行为不同
   - 一段代码的隐含假设在别处被悄悄违反
   - 一个设计决策在今天已经过时但没人回头更新
   - 一处看似无害的代码在特定负载/规模/并发下会失败
   - 一个抽象层同时泄漏了两种不同的隐喻
   - 一个"优化"实际上在当前使用模式下是负优化
   - 两个独立的子系统在某个边界条件下会相互破坏
   - 某个模式如果再多用三次就会变成维护灾难
   - 某个"简单"的实现实际上包含了一个隐藏的 O(n²) 或更差
   - 文档描述了一种理想，但代码实现了一种不同的理想
4. 你可以跳跃。如果你在读 A 模块时突然意识到 B 模块可能有问题，立即去验证。不要强行按顺序。
5. 你可以否定自己。如果你一开始觉得某个东西是问题，深入读完后发现不是，直接放弃。不要为了凑发现数量而保留它。
6. 质量优先于数量。一个真正意外的、跨领域的、能改变设计方向的高质量发现，比十个机械的代码风格问题有价值得多。
7. 你被明确鼓励去发现本项目维护者可能"灯下黑"的问题——那些因为太熟悉反而看不到的东西。
8. 不要把任何"上一次审查已经覆盖过的维度"当成约束。如果上次审查漏了什么，你发现了就是你的发现。
9. 如果你发现了一个有意思的模式或反模式，追踪它在整个仓库中的完整影响范围，而不是只报单个文件。

输出格式：

不需要统一模板。用你觉得最清晰的方式呈现每个发现。但每个发现必须包含：

- 在哪里（文件路径 + 行号或区域）
- 是什么（你说的问题到底是什么）
- 为什么值得关心（不修会怎样，或修了会怎样）
- 你的信心水平（确定 / 很可能 / 有趣的猜测）

最后给出一段自由形式的总评：你觉得这个项目最值得关注的 1-3 个方向是什么，为什么。

执行方式：

先花时间读代码，不要急着给结论。你读得越多，越有可能发现别人没看到的东西。
充分利用子agent去独立调研或者做对抗性检查。

检查结果保存在docs/analysis目录下。
