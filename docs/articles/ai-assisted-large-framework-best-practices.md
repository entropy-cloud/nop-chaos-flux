# AI 辅助开发大型框架：最佳实践与背后的原因

## 引言

AI 已经可以持续生成代码、补测试、改文档、做批量重构，甚至参与架构讨论。真正的问题不是"AI 能不能写"，而是：**怎样让 AI 长期参与大型框架开发，而不把系统带入抽象漂移、边界坍塌和错误自洽。**

**AI 不会自动降低大型框架开发的难度，它只会把既有方法论的优点和缺点同时放大。** 方法对，极大放大产能；方法错，极大放大破坏力。

本文判断来自第一性原理推导——以一个大型框架仓库在 AI 深度参与下的完整开发历史为校验材料，吸收历史中反复被修正且原理上能解释其必要性的做法，不采纳历史中反复出现但原理上站不住的做法。

两点背景。第一，项目主要在极高 AI 依赖度下推进——单人、文档驱动。文档权威链、owner plan、closure audit 等机制替代的是传统工程中靠人脑记忆和多人协作交叉校验来完成的工作，但治理本身也必须始终被 live code path 和可复查证据持续校准，不能演化成文书上的秩序。第二，大型框架站在历史系统和真实产品压力之上演化；前代框架的能力引入本质上是在新设计原则下对概念边界的压力测试。

## 一、大型框架与普通应用的根本区别

关于 AI 开发的大量经验，默认场景其实是普通应用——页面、接口、CRUD、少量业务状态、局部交互。这类系统也复杂，但主要风险通常是局部功能错误。

框架的核心资产不是某几个页面或某组 API，而是：

- 一组长期稳定的抽象边界。
- 一套可以被不同模块共同依赖的术语系统。
- 一个尽量闭合、尽量不随局部需求膨胀的概念模型。
- 对上层调用者可预测、可解释、可迁移的合同。

AI 在框架开发中的主要风险不是"把一段代码写错"，而是：

- 在错误的层里实现了正确的功能。
- 用模糊命名覆盖了本应严格区分的概念。
- 为了解决局部症状，新增了会永久污染模型的机制。
- 让文档、示例、代码、测试分别对应不同的系统现实。
- 把"接口已经出现"误判为"语义已经落地"。

所以，大型框架上的 AI 最佳实践不能只是"怎么提高生成效率"，而必须首先回答：**怎么保护系统抽象不被高频局部最优侵蚀。**

以上 17 条原则按关注点归并为五个大组（原文各节保持原顺序不变）：

- **一、边界与架构**：原则1(边界先行)、2(AI定位)、15(诚实命名)、16(升格审查)
- **二、文档与上下文**：原则3(权威链)、4(文档分离)、12(漂移治理)、上下文持久化(§五)
- **三、执行纪律**：原则6(计划可关闭)、7(语义切片)、5(需求澄清)
- **四、验证与完成**：原则8(验证体系)、9(语义测试)、10(closure audit)、11(接口vs行为)、13(调试基建)
- **五、复杂度与抽象质量**：原则14(窄合同)、17(探索性原型)

## 二、第一原则：先决定边界，再让 AI 生成实现

AI 最突出的优势之一，是能在很短时间里给出大量局部可行解。大型框架最突出的危险之一，恰恰是局部可行解太多，系统迅速失去统一边界。

第一原则不是"先写起来"，而是：

**任何新增能力，都先判定它属于哪一层，再讨论怎么做。**

这在框架里特别关键，因为同一个问题通常可以放在很多地方：编译阶段做一次性归一化、运行时做动态解释、宿主/平台层做预处理、具体 renderer 做局部兜底、调试/验证层做约束。

如果不先回答"它属于哪层"，AI 默认会沿最近上下文扩张——当前文件、当前 bug、当前命名和当前调用链，而不包含系统真正的所有权边界。表面上高效，实际上在持续透支未来。

这个仓库的架构材料中，边界意识贯穿始终。`frontend-programming-model.md` 反复强调 closed primitive set、authoring model 与 execution model 的边界，以及"能在结构变换层解决的问题不要提升到 runtime surface"；`flux-runtime-module-boundaries.md` 把"代码到底应该放在哪个 owner 层"当成正式架构问题，而不是实现时顺手决定的细节。这些做法恰好验证了第一性原理的判断：**框架开发里，正确的层次选择本身就是主要工作，代码实现只是随后发生的事情。**

## 三、第二原则：把 AI 当作"方案展开器"，不要当作"自动架构裁判"

AI 擅长三件事：快速枚举方案、快速外显隐含前提、快速指出每个方案的失败模式和迁移代价。

AI 不稳定的地方在于，它并不天然拥有长期架构判断。大型框架里的关键决策本质上不是"哪种实现更聪明"，而是"哪种边界最值得长期维护"——一种带有价值判断和长期成本判断的裁决。

最有效的用法不是问 AI"请直接给我最好的架构"，而是要求它：

1. 给出 2 到 4 套边界不同的方案。
2. 明确每套方案的概念中心。
3. 说明哪些复杂度留在核心，哪些推出到宿主层。
4. 说明最可能在半年后后悔的点。
5. 说明验证成本和迁移成本。

这不是形式主义。框架设计真正难的地方不是"想不到办法"，而是"看不清每个办法会把什么代价长期固化进系统"。AI 恰好特别擅长把代价外显出来。**AI 更稳定地适合做架构推导，不适合独占架构裁决。**

这个仓库的 discussion、analysis、architecture 文档体现了类似的工作法：不直接奔向唯一答案，而是不断澄清术语、比较层次、收紧边界。AI 更像一个高带宽推导器，而不是自动完成架构设计的黑盒。

## 四、第三原则：文档不是"知识仓库"，而是"权威链和语义路由系统"

AI 项目常见的误区是以为"文档越多越好"。真正的问题不在于多少，而在于**文档是否形成了清晰的权威链**。

AI 的基本工作方式是读取上下文、综合多个来源、生成一个自洽解释。如果这些来源没有明确优先级，AI 就会做一件很危险的事：自动把不同阶段、不同状态、不同用途的材料平均起来，拼成一个看似合理的"综合现实"。这在普通项目里已经危险，在框架里更危险，因为框架高度依赖稳定术语和边界一致性。

适合 AI 的大型框架文档系统，至少要回答四个问题：

1. 当前系统的正式基线由哪些文档拥有？
2. 当不同文档冲突时，谁优先？
3. 哪些材料在描述当前现实，哪些只是在探索未来方案？
4. 新进入上下文的 agent 应该先读什么？

这个仓库在这点上做了系统化的工作：`docs/index.md` 被定义为全局路由入口；`docs/architecture/README.md` 明确了 governing principles、normative architecture、platform extension architecture、focused subsystem docs 的层次；不同文档之间有 precedence model。这些做法击中了 AI 场景下的核心问题：**AI 不是怕材料少，而是怕材料没有权威关系。**

最佳实践应当是：

- 用极少数文档定义当前真实基线。
- 用显式的 precedence 模型定义冲突处理顺序。
- 用 docs router 为人和 AI 同时提供最短阅读路径。
- 允许 top-level owner doc、local owner doc、reference doc 分层共存，但不允许它们在同一问题上并列冲突。

## 五、第四原则：严格分离"现状文档""探索文档""执行文档""历史日志"

这是大型框架 AI 开发里最容易被低估的一条纪律。

这几类材料回答的是完全不同的问题：

- 现状文档：系统现在认可什么。
- 探索文档：我们在比较什么。
- 执行文档：这次要把哪块收敛到什么程度。
- 历史日志：最近发生了什么，为什么这么做，下次从哪继续。

混在一起，AI 会产生几种典型误判：把未落地的探索方案误当现行合同；把已经失效的旧 baseline 当作仍然有效的前提；把某个切片的 landing 当成整个主题已经完成；把历史日志里的临时决定误读为长期规范。

这个仓库对这点的收敛很明显：`docs/architecture/README.md` 要求架构树只保留 current-design rationale；`docs/plans/00-plan-authoring-and-execution-guide.md` 明确 plans 是 execution docs，不是 architecture 替代品；`docs/logs/index.md` 明确 development log 不是 source of truth，而是 lightweight context；`docs/references/maintenance-checklist.md` 把"改了什么代码要同步哪些文档"制度化。

这套分层不是文书工作，而是 AI 时代的基础设施。只要文档状态混乱，AI 就会高效地放大这种混乱。

这里还要补一个原则：**上下文必须能持久化，而不能只存在于单次会话里。**

AI 的工作不是线性的——它会中途修正判断、放弃路线、改术语、拆计划、补 successor plan。如果这些信息只停留在一次性对话里，后来的 AI 或审查者几乎不可能准确恢复上下文。`docs/logs/` 这样的执行记忆之所以有价值，不是因为它记录了多少历史，而是因为它把关键判断、关键链接和下一步动作持续沉淀进仓库，让系统的有效记忆不再依赖某个会话窗口是否还在。

## 六、第五原则：需求澄清的目标不是"理解功能"，而是"消灭语义歧义"

### 需求澄清与概念守门

AI 让编码速度大幅提高，很多团队因此会弱化需求澄清，觉得可以"先写出来再收敛"。这在框架上非常危险。

AI 会非常快地把模糊需求具体化。一旦具体化方向错了，错误会同时进入命名、类型、实现、文档、示例和测试，之后再修正的成本远高于普通应用。

框架需求澄清真正应该问的不是"要做什么功能"，而是：

- 这个能力的最小语义单位是什么？
- 它和已有概念的边界在哪里？
- 它是新 primitive、现有 primitive 的派生系统，还是宿主/平台的责任？
- 它的默认行为是什么，哪些才是配置项？
- 它失败时应该显式暴露什么，而不是静默兜底什么？

**框架需求澄清不是功能拆分，而是概念守门。**

因此，在实质编码前，值得强制 AI 输出：

- 它对需求的复述。
- 它识别出的隐含前提。
- 它认为存在歧义的术语。
- 它判断的边界归属。
- 它最担心的返工点。

### 历史系统能力引入与概念压力测试

还有一个在大型框架里尤其关键、但容易被忽略的现实：很多"需求"其实不是来自一句新功能描述，而是来自历史系统的既有能力集合。比如当你决定是否引入 `object-field`、`variant-field`、`recurse`、`crud` 这类节点时，真正要回答的问题通常不是"要不要多一个组件"，而是：已有框架和真实场景已经证明这类问题长期存在，那么在新的 primitive set、owner boundary 和 runtime contract 下，它到底应该以什么形式重新进入系统。**组件引入本身常常是在做概念压力测试，而不是单纯补产品功能。**

还有一个值得保留的动作：**把比较研究当成压力测试，而不是论证材料。** 仓库里很多 comparison/analysis 文档的真正价值，不是证明"当前路线最先进"，而是逼迫设计回答为什么不走别的成熟路线、当前方案的真实代价是什么、哪些只是理念优势、哪些已经是工程事实。对 AI 来说，这能有效防止它把当前局部实现误当成唯一自然答案。

前代框架和成熟产品本身，就是候选抽象的外部测试集。AMIS 不是只能被当作迁移包袱或类型来源；它的大量组件族、字段形态、CRUD/workflow/布局 authoring 方式，本身就在告诉我们哪些问题在真实低代码系统里会反复出现。Budibase、NocoBase、Formily 这类系统也不只是拿来"比较谁更先进"，而是拿来持续追问：

- 这个能力在成熟系统里为什么会出现？
- 它在那里属于 core primitive、组合模式，还是 product-specific convenience？
- 如果我们不保留它，是否真的消灭了问题，还是只是把问题换了一个名字重新出现？
- 如果我们保留它，应该以当前框架自己的 owner/runtime/host 语义落地，而不是连带旧系统的整套边界一起搬进来？

对大型框架来说，比较研究、旧框架能力盘点、AMIS baseline matrix、以及一类组件是否值得进入 retained family，并不是"外围文档工作"，而是决定 closed primitive set 是否诚实的重要证据来源。

把这件事说得更具体一点：AMIS 这样的前代框架，至少同时承担三种角色：

1. **迁移来源**：它告诉你下游作者、既有 schema、既有心智模型里已经存在什么能力族，哪些名字和 authoring 方式会反复出现。
2. **压力测试集**：它逼迫新框架回答——这些能力在新的 primitive set 和 owner boundary 下到底怎样重新落位，哪些需要 retained，哪些应该 absorbed/renamed，哪些必须明确 not retained。
3. **反例库**：它也提醒你不要把旧系统里出于历史兼容、命名噪声或宿主耦合而存在的表面形态，不加区分地带入新 baseline。

比如，一个成熟做法不是简单地说"AMIS 有 `crud`，所以我们也保留同名同形态 contract"，而是更细地判断：哪些能力要 capability mapping 进入新 schema，哪些字段名不该继续成为 formal DSL 的一部分，哪些 whole-node lowering 应该由 compiler/runtime owner 承担，而不是把旧 authoring surface 整包搬过来。类似地，`object-field`、`variant-field`、`array-field` 这类能力也不能只按"旧框架里存在"来保留，而要进一步看它们在新系统里暴露出来的究竟是新 owner family 的必要性，还是路径绑定、projected runtime、validation addressing 等底层能力仍需收敛。**这才是历史能力集作为压力测试场的真正价值：它逼迫新框架区分"问题真实存在"和"旧解法应被原样继承"这两件事。**

## 七、第六原则：计划必须是"可关闭的执行文档"，不是 ideas dump

大型框架开发里，AI 最容易把问题推向两种极端：没有计划，直接沿着上下文一路改；或者有计划，但只是宏大 roadmap，不能指导真正执行，也不能判定何时完成。

真正有用的 plan，必须回答：

- 当前 baseline 到底是什么。
- 这次明确要收敛什么。
- 这次明确不做什么。
- 每个 slice 完成的 exit criteria 是什么。
- 要经过什么验证才算真的落地。
- 关闭这份计划之前，谁来独立审计。

`docs/plans/00-plan-authoring-and-execution-guide.md` 给出了非常成熟的约束，几条特别值得注意：

- 写计划前先核对 live repo，再写 current baseline。
- 一个计划只负责一个明确结果面；过宽就拆 successor plan。
- 必须有 goals 和 non-goals，防止 scope drift。
- `completed` 必须来自独立 closure audit，不能在实现最后一个 slice 时顺手宣布。
- 必须明确区分 contract surface 已出现和 contract semantics 已落地。

从第一性原理看，这几条之所以重要，是因为 AI 特别容易：沿用旧计划和旧结论，不重看 live repo；把一个 landing 误认成整个主题结束；把接口、类型、方法名出现误认成行为完成；在没有明确非目标的情况下不断顺手扩 scope。

**AI 辅助大型框架开发时，计划系统不是装饰，而是一种执行收口机制。**

## 八、第七原则：切片必须按"语义边界"和"owner 边界"来切，而不是按文件数量来切

很多团队说要小步迭代，但真正困难的是"怎么切才不会切坏"。

在框架里，错误的切法通常有两种：按文件机械切，结果一个语义改动跨很多文件但没有统一收口；按主题写超大计划，结果没有任何 slice 真的可验证、可关闭。

更好的切法是按两种边界切：语义边界（一次只收敛一个概念）和 owner 边界（一次只动一个负责层）。

一个问题如果本质上是 compiler/runtime 边界问题，就不应该被拆成多个 renderer 小修；如果本质上是 host protocol 问题，就不应该顺手在 core primitive 层引入新概念。

**小步的意义不是更保守，而是让归因更准、收口更稳。**

还有一个容易被忽略、但对框架尤其关键的点：切片不仅要按语义边界切，也要按性能与响应式热路径切。以下规则值得被当成硬约束：

- render phase 不能写状态。
- reactive render reads 必须通过订阅或 selector，而不是命令式 snapshot。
- scope identity 变化必须谨慎，不能让局部 state 和父子 reactivity 无意重置。
- field-state 这类热点读路径要走 per-path subscription，而不是整 store 广播。
- 异步入口必须显式声明 overlap policy，包括 abort、supersede、timeout、stale guard。

AI 很容易写出"功能上看是对的，但热路径纪律被破坏"的代码，而这类问题在框架里会随着节点数、字段数、交互频率成倍放大。

## 九、第八原则：验证体系必须比普通项目更强，而且要区分"定向验证"和"全局验证"

AI 让代码产出速度远远超过人工 review 速度，大量错误不能再留到人工 code review 时发现，而必须前移到自动化约束中。

但光说 typecheck、build、lint、test 还不够。大型框架里更重要的是：**哪些验证本身是架构守门，而不是普通质量检查。**

这里至少还有一层经常被低估的前置验证：**编译期结构诊断。**

对于 schema 驱动框架，很多错误不应该等到 runtime、E2E 或人工 review 才暴露，而应该在 compile 或 validate 阶段就被拒绝——未知字段或冲突字段、renderer 契约不满足、host contract 或 capability manifest 不匹配、某类 schema 结构在当前 execution model 下根本不成立。`schema-file-validator.md`、`capability-projection-manifest.md` 等文档，就在逐步把这类验证前移。AI 最容易批量制造"看起来像对的 schema 或 contract surface"，而编译期结构诊断恰好是最便宜、最稳定的第一道防线。

具体来说，有三类做法值得纳入最佳实践。

第一类是**性能红线**。`performance-design-requirements.md` 把 compile once/execute many、identity stability、fine-grained subscription、immutable updates、deterministic async cancellation、hot path 禁止模式都写成了正式设计要求。AI 很容易写出"功能对了、性能模型却错了"的代码，而这类错误在框架里会被成百上千个节点放大。

第二类是**安全红线**。`security-design-requirements.md` 明确了 no runtime permission semantics、no eval/new Function、effect authority boundary、observable failure paths 等要求，把"不要这样做"提升成系统级边界，而不是依赖 reviewer 的临时记忆。

第三类是**bug 派生 guardrails**。`architecture-guardrails-from-bugs.md` 把历史 bug 收敛成 review checklist——reactive render reads 必须订阅、render phase 不可写、async overlap 要有并发策略、source tree 不能混入构建产物。这类 guardrail 特别适合 AI 场景，因为 AI 最容易重复模式级错误；只要某种错误出现过两三次，就值得制度化，而不是每次重新讲一遍。

验证顺序也很关键。先做 focused verification，再做 full repo verification。定向验证回答"这次改动自身是否成立"，全局验证回答"这次改动有没有破坏系统别处"。这两个问题都必须回答，但不应该在同一个低信息密度阶段混成一团。

## 十、第九原则：测试要保护"语义合同"，不要保护偶然实现

AI 很会补测试，但默认倾向是沿当前实现写测试，因为这最容易自动生成。对框架来说，这种测试很危险。

框架内部会持续重构。如果测试绑定的是实现组织而不是对外合同，AI 为了让测试继续通过，就会不断保留偶然实现，最后让测试反过来绑架模型演化。

框架测试更应该保护：某个边界承诺了什么；输入输出语义是否稳定；跨层职责分配有没有被破坏；某个历史 bug 是否已经拥有长期回归保护。

一个值得采纳的做法是：很多验证不追求"大而全"，而是针对具体 contract 或具体 regression 写 focused tests。这种方式比大面积快照式测试更适合框架，因为它保护的是行为闭环，而不是内部实现布局。

框架测试至少应分三层：

- owner-module direct tests：直接验证某个模块或 owner 层自己的合同。
- bug-derived focused regressions：把已经出现过的高价值 bug 固化成最小回归保护。
- behavioral E2E：验证端到端合同，但避免退化成"页面稳定""元素可见"这类弱断言。

E2E 不是越多越好。弱 smoke 测试常常会制造"看起来覆盖了很多"的错觉，但真正的 contract drift 并没有被保护住。

## 十一、第十原则：完成标准必须经过独立 closure audit，不能让实现者自我宣布完成

这是 AI 大型框架开发里最重要的治理纪律之一。

如果同一个上下文同时负责理解问题、设计方案、写代码、补测试、更新文档，再判断"是不是完成了"，那它极容易形成一种流畅但错误的自洽叙事。

计划指南对此要求得非常严格：`completed` 必须来自单独的 closure audit；必须重读整份 plan；必须核对每个 slice 的 exit criteria；必须区分 interface presence 和 behavior landing；必须由独立 reviewer 或独立子 agent 在 fresh session 中执行。

从第一性原理看，实现与验收本来就是两种不同认知任务：实现关注"怎么把它做出来"，验收关注"有没有不自觉地漏掉什么"。AI 在前一种任务上很强，在后一种任务上如果复用同一上下文就会明显变弱——它会偏向证明自己是对的，而不是积极寻找自己没做完的地方。

**独立复核不是额外流程，而是抵消 AI 自洽偏差的必要机制。**

多视角审查真正重要的不是"并行开几个 agent"，而是最终能否收敛成经过复核的共识。更可靠的做法不是把首轮输出直接汇总，而是把审查组织成有顺序的复核链：先产生候选发现，再做独立的维度级复核，再对高价值结论逐项回看 live repo。多模型、多上下文、多角色提示的价值，不只是多找意见，而是降低单一上下文偏差，再通过一轮单独的 reconciliation 或 meta-review 过滤掉证据不足和架构洁癖式结论。**AI 时代的大型框架评审，更像同行评审，而不是并行搜索。**

### 接口存在不等于行为落地

这条值得单独强调，因为它是 AI 项目里最常见的假完成来源。

AI 辅助开发中，特别容易出现：类型定义补上了、对外接口导出了、文档里写到了、示例里能看见字段了、测试里有个壳了——然后系统就被叙述成"已经支持"。

计划指南明确把这件事当成高频历史错误，甚至专门写进 minimum rules 和 closure audit rule。框架消费者会把这些表面元素当成正式承诺；如果它们背后没有完整语义闭环，那不是"暂时没做完"，而是在对外制造错误能力印象。

完成定义必须至少包含：

- live code path 是否真的经过这段逻辑。
- 核心行为是否有 focused verification。
- 关键边界条件是否被验证。
- 文档描述的是 current baseline，而不是目标愿景。
- leftover work 是否真的被显式移出当前 plan 范围。

如果框架已有下游消费者，还必须再加一层约束：**对外合同变更要有显式迁移纪律。** 内部把概念收紧、命名修正、边界重构，并不自动等于可以直接把旧合同删掉。对于已经被外部 schema、插件、宿主或业务代码依赖的 surface，最佳实践应当包括：明确 deprecation 信号、版本或阶段说明、迁移路径、兼容窗口，以及最终删除条件。否则 AI 会因为"内部模型已经更正确了"而过早清理旧入口，把正确的架构演进变成对下游的突发破坏。

## 十三、第十二原则：认知漂移不是"文档小问题"，而是一等故障

普通项目里，文档滞后一点往往还能忍。大型框架不同——框架高度依赖稳定术语、稳定边界和一致心智模型，一旦认知漂移出现，后果会很快扩散。

几类常见的漂移：owner doc 和 live repo 不一致；plan status、completion note 和实际落地彼此不一致；旧 audit 还在被引用，但 live repo 已经变化；示例代表的是过渡状态，而不是现行合同。

从第一性原理看，这些都应该被视为输入损坏。AI 基于上下文工作，只要上下文损坏，它就会高效地产生更多损坏结果。

漂移治理不应该被看成"有空再补文档"，而应该制度化：

- 改动导出合同、运行时行为、schema 语义、包边界、示例时，必须同步回看相关文档。
- 为常见改动建立 maintenance checklist，而不是靠人记忆。
- 对历史 bug 中反复出现的模式，沉淀 review guardrails。

`maintenance-checklist.md` 和 `architecture-guardrails-from-bugs.md` 在这方面提供了很好的范本——它们把"同步哪些文档""警惕哪些已知回归模式"从口头经验变成了显式执行面。

## 十四、第十三原则：大型框架里的"调试能力"本身就是 AI 协作基础设施

很多团队谈 AI 开发时，只想到代码生成和测试修复，却忽略了另一个关键点：**AI 只有在系统能被结构化观察时，才能高质量参与调试、审计和回归定位。** 诊断能力本身是 AI 协作基础设施，不只是给人类开发者看的调试面板。

`debugger-runtime.md` 明确把 debugger 定义为 framework diagnostics infrastructure，而不只是 playground log panel；并强调 automation-facing API、structured event model、inspectByCid、bounded event budget、AI/E2E 优先走结构化 API 而不是抓 panel DOM。

如果调试系统只对人类友好而对机器不可读，AI 在遇到复杂问题时就只能盲猜、到处 grep、解析截图、从非结构化日志里拼凑状态，极大降低有效性。

对 AI 深度参与的大型框架来说，最佳实践不是"出了问题再让 AI 看日志"，而是预先建设：

- 结构化事件模型。
- 可查询的自动化诊断 API。
- 明确的 inspect contract。
- 有界的事件预算和 snapshot discipline。
- 让 E2E/AI 优先消费机器可读诊断面，而不是页面表象。

在更操作的层面上，这还意味着几条简单但高价值的纪律：

- 优先使用结构化 debugger 或 automation API，而不是抓面板 DOM。
- 遇到 UI 问题优先看 DOM、computed style、事件和状态，而不是依赖截图解读。
- 等待条件优先走 event-driven waits，而不是固定 sleep。
- timeout、abort、supersede、retry 这类并发与时序策略要显式建模，不能留给调用方猜。

与之配套的，还有一个更底层的要求：**诊断证据、修复边界、验证路径必须保持闭环。** 框架里的 bug 修复不应该只是"看到失败就改代码"，而应该尽量先确认问题的可复现边界、找到最小回归保护、再做最小语义修复，并用 focused verification 证明修复确实击中了根因。对跨层、跨包、容易重复出现的问题，最长期有价值的往往不是 diff 本身，而是诊断方法：为什么难定位、排除了哪些假设、最后用什么直接证据确认根因。这些内容经常比修复代码更能帮助后续 AI 少走弯路。

## 十五、第十四原则：复杂宿主边界必须收紧为少数窄合同

大型框架一旦开始承载复杂宿主——设计器、报表编辑器、画布、表格引擎、外部会话系统——就会出现一个常见诱惑：为了方便，直接把宿主内部对象、bridge、controller、会话状态暴露给框架 runtime 或 schema 可见层。

短期上很诱人，调试和接线会显得简单；长期上通常会把宿主复杂度反灌进核心，破坏抽象边界。

这个仓库在 host boundary 设计上的高价值经验，是尽量把复杂宿主收敛成几类窄合同：

- host projection 负责只读快照暴露。
- capability 或 namespaced actions 负责受控写入。
- bridge 保持宿主私有，不进入 schema-visible scope。
- manifest 或 versioned contract 负责静态校验和升级约束。

从第一性原理看，这正是让复杂宿主仍然服从框架核心抽象的办法：**把跨边界通信收紧成少数可审计、可验证、可升级的通道。**

## 十六、第十五原则：命名、类型和包边界必须诚实

大型框架很容易出现一种高级但有害的倾向：为了显得类型更漂亮、接口更抽象、结构更统一，反而把系统真实约束藏起来。

`refactoring-guidelines.md` 提出了一个重要观点：某些 `Record<string, any>` 在动态 schema 系统里不是失败，而是最诚实的类型表达；某些看似相似的逻辑不该过早提取；某些共享纯函数必须放在更底层包里，否则会制造反向依赖。

这些观点指向同一条原则：**框架设计要优先忠实表达真实约束，而不是为了表面整洁制造额外失真。**

具体体现在三个方面：

- 命名要诚实：不同概念不要用近义词糊成一团。
- 类型要诚实：动态系统里不要用伪精确类型掩盖真实不确定性。
- 包边界要诚实：共享纯逻辑就应该放在依赖链正确的位置，而不是被"某层应该只放类型"的审美偏好牵着走。

这在 AI 场景下尤其关键。AI 对名称、类型和边界非常敏感，一旦表层符号不诚实，它会沿着错误符号继续高效推理，最后把整个系统拖向更大的误解。

术语收敛本身就是架构工作。概念改名往往不是为了措辞更好看，而是在修正认知边界。对 AI 来说，词汇稳定性直接影响后续计划、文档、代码和审计能否继续指向同一个对象。

在大型边界迁移或低代码系统重构中，兼容层并非一定错误，但必须被当成明确的过渡接缝，而不是永久藏在系统里的模糊历史负担。这个仓库里多次可以看到 compatibility layer 被保留一段时间以完成迁移，但随后要么被 successor plan 接管，要么在 closure audit 中被明确归类。这种做法承认现实迁移成本，但不允许"临时层"偷偷成为长期架构。

## 十七、第十六原则：复杂度治理比功能推进更重要，新增机制必须经过"升格审查"

AI 让新增代码变得很便宜，而大型框架最怕的恰恰是"新增一个东西太便宜"。

没有强约束，人和 AI 会一起滑向机制膨胀：多一个配置项、多一个 hook、多一个 runtime 特例、多一个 compatibility layer、多一个"先放这里以后再收"的临时结构。久而久之，框架失去模型闭包。

这个仓库的架构文档一直在做一件事：不断问"这到底是不是一个应该被提升到核心的概念"。无论是 closed primitive set、promotion rules，还是 derived runtime system 的划分，都在做复杂度治理。

每当出现一个新机制，至少要问：

1. 它是新原语，还是旧原语的派生形式？
2. 它解决的是长期跨域问题，还是单点症状？
3. 它会不会把宿主/平台复杂度反灌进核心？
4. 六个月后如果要删掉它，会不会很痛？

如果这些问题回答得不够硬，就不应该急着让它成为正式机制。

## 十八、第十七原则：探索性原型不是附属演示，而是语义裁决机制

前面的原则主要解决"怎么让治理体系运转"。这条原则回答的是另一个关键问题：**怎样防止治理结构脱离 live repo，变成只在叙事层成立的上层秩序。**

如果说 owner doc、architecture baseline、execution plan、closure audit 负责约束系统不要失控，那么探索性原型、playground、scope-debug、automation-facing diagnostics 和 benchmark 承担的就是防止这些治理结构脱离可执行现实的角色。

下一代低代码底层框架里最难判断的问题，往往不是"代码能不能写出来"，而是：

- 这个概念到底是不是稳定语义单位。
- 这个 owner 边界到底能不能长期成立。
- 这个抽象在热路径、并发、响应式和调试面上到底是不是闭环。
- 这个命名、类型和 contract 到底只是纸面上整齐，还是 live code path 真的经过。

这些问题单靠 discussion、analysis、architecture doc、owner plan 和 closure audit 并不能完全回答。它们仍然必要，但如果缺少可执行现实的反压，项目就可能进入危险状态：治理完备、叙事漂亮，语义落地却停留在文书层。

大型框架里的最小原型，不应被理解为"先随便做个 demo 看看"，而应被理解为：**对候选抽象进行最小语义审判。**

这种审判不只发生在全新想法上。对历史系统能力的重新引入（如 `object-field`、`variant-field`、`recurse` 等），同样是一种原型和压力测试——它测试的不是"能不能做出同名组件"，而是"新架构是否足以诚实承载这类长期问题"。（关于历史能力引入的概念压力测试，§六已有展开。）

它的作用不是证明"这个功能大概能跑"，而是更严格地回答：

1. live code path 是否真的经过这段机制，而不是只有接口表面存在。
2. 关键边界条件是否已经拥有 focused verification。
3. 关键热路径、异步重叠、身份稳定性、状态传播是否能被直接观察。
4. debugger、inspect API、playground 场景和 benchmark 是否能把它暴露成可复查证据。

只有通过这种最小可执行语义样本验证之后，一个候选机制才适合被继续提升为 current baseline、owner-level architecture contract 或 primitive promotion 候选。反过来说，如果一个想法只能在文档、计划和审计里自洽，却没有最小运行闭环，那它仍然只是候选模型。AI 深度参与的场景里这种风险被进一步放大：AI 很擅长把一个想法补全成流畅叙事，却不天然保证这个叙事已经穿过真实运行路径。

**探索性原型不是对严肃治理的削弱，恰恰相反，它是治理体系中的实验法庭。**

从这个角度看，一个足够成熟的 AI 辅助大型框架方法论，不应只有文档权威链、owner plan、focused verification 和独立 closure audit，还应把可执行探针正式纳入方法论本身。因为对于底层模型探索来说，真正危险的从来不是"想法不够多"，而是"想法太容易在没有经过可执行裁决的情况下被升格进核心"。

**在大型底层框架里，不能被 playground、focused regression、structured diagnostics、live code path 和 benchmark 直接验证的"架构进展"，不应轻易升格为正式基线。**

真正稳定的方法论不是"先治理、后实现"，而是让治理、实现、验证和可执行观察始终互相反压、共同收敛。

## 十九、一套更可靠的 AI 辅助大型框架工作流

综合以上原则，一个更可靠的工作流大致是这样：

### 1. 建立权威链

- 有全局 docs router。
- 有明确的 architecture hierarchy。
- 有 current baseline owner docs。
- 有探索、执行、历史材料的显式分层。

### 2. 在编码前先做边界判断

- 先判断属于 compile、runtime、host、renderer、diagnostics 的哪一层。
- 先判断是新概念还是旧概念的派生。
- 先判断 default semantics 和 failure model。
- 先判断这是全新需求，还是旧框架/成熟产品中早已存在的问题在当前 baseline 下重新出现。
- 如果它来自历史系统，先说明是要 retained、absorbed、renamed、deferred，还是明确 not retained。

### 3. 让 AI 做方案展开和歧义暴露

- 输出多套方案。
- 输出隐含前提。
- 输出失败模式。
- 输出最可能返工的点。

### 4. 以 owner plan 执行，而不是沿上下文一路改

- 写 current baseline。
- 写 goals / non-goals。
- 写 exit criteria。
- 把剩余工作归属清楚。

### 5. 执行中持续外部化上下文

- 重要切片 landing 后及时写执行日志。
- 记录 changed what、key decision、next step。
- 对关键判断附 code/doc/plan 路径，而不是只留在会话里。

### 6. 复杂问题走多源审查，再收敛到共识

- 先让多个独立认知源从不同上下文做分析。
- 汇总重叠结论、冲突结论、证据不足项。
- 再做一轮单独的 reconciliation 或 meta-review。

### 7. 先定向验证，再全局验证

- focused tests / package verification 先行。
- 全仓 typecheck/build/lint/test 收口。
- compile 或 validate 阶段的结构诊断尽量前移。
- 性能、安全、bug guardrails 进入 review checklist。

### 8. 用可执行探针裁决候选抽象

- 对可能升格为 primitive、owner seam、runtime substrate、compiler/host contract 的能力，先构造最小可执行语义样本。
- 优先使用 playground、scope-debug、automation-facing diagnostics、structured inspect API 暴露关键状态；遇到性能与热路径问题时补上有针对性的 benchmark 证据。
- 确认 live code path 真实经过、关键边界条件可回归、热路径与并发语义可观察之后，再决定是否升格为正式基线。
- 如果只能在文档、分析和计划里自洽而没有可执行闭环，就继续保留为候选模型。

### 9. 把历史系统和对照实现当成外部测试集

- 对 AMIS、NocoBase、Budibase、Formily 等历史/同类系统，不要只做优缺点评分；要把它们当作候选抽象的外部压力测试来源。
- 对已有能力先判断它在旧系统里到底是 primitive、组合模式、宿主特例还是历史包袱，再决定新系统如何吸收。
- 当 `object-field`、`variant-field`、`recurse`、`crud` 这类节点被重新引入时，默认把它视为一次概念回归测试：验证新 owner/runtime 语义是否真的能承载这类长期问题。
- 任何"为了兼容旧能力而保留"的机制，都必须明确它是 retained baseline、过渡 compatibility seam，还是仅供迁移的临时入口。

### 10. 建立独立 closure audit

- fresh session，并强制回看 live code path、focused verification 结果以及相关 playground / diagnostics 证据。
- 重读完整计划。
- 检查行为闭环，而不是表面元素。
- 明确 leftover work 是否已移出本 plan。

### 11. 把诊断基础设施建设出来

- 结构化 debug events。
- automation APIs。
- inspect contract。
- 机器可读的 failure context。
- DOM-first、event-driven 的操作级诊断纪律。

### 12. 把漂移治理制度化

- 维护 checklist。
- bug-derived guardrails。
- 示例与文档同步校验。
- 定期重审 active baseline。
- 对复杂跨层问题保留 bug note，尤其是 diagnostic method。

## 二十、什么才算真正的"最佳实践"

最佳实践不是"让 AI 产出更多代码"，也不是"把仓库写满文档"。

真正的最佳实践应该满足这些判断标准：

- 新 agent 进入仓库后，能快速找到当前真实基线。
- 同一概念必须有明确 owner；局部主题可以有 local precedence owner，但不能并列冲突。
- 任何新增能力都能先说清属于哪一层，再开始实现。
- 面对历史系统里早已存在的能力时，项目能明确说明是吸收、重命名、收紧、延后，还是拒绝保留，而不是含糊地照搬或隐式消失。
- 系统有足够强的性能、安全、回归 guardrails，能在机器层拦住高频错误。
- 测试保护的是语义合同，而不是偶然实现。
- 候选抽象在升格为正式基线之前，必须经过最小可执行语义样本的验证，而不是只在文档、计划和讨论中成立。
- 前代框架和成熟产品经验被当成外部压力测试集使用，而不是被简单神化或简单切断。
- 完成判定依赖行为闭环和独立审计，而不是接口存在。
- 调试和诊断面对机器同样友好，而不是只对人类可读。
- 项目能持续删掉坏抽象，而不是只会叠加新机制。

如果这些条件都成立，AI 就不只是一个"写代码助手"，而会成为一个能够长期参与框架建设的稳定工程节点。

## 结语

AI 辅助开发大型框架，最难的从来不是 prompt，不是模型参数，也不是一次性能生成多少代码。真正困难的是：**如何用工程结构抵消 AI 的失真倾向，同时放大它的高带宽推导能力。**

好的结果从来不是因为 AI "自己变聪明了"，而是因为团队把路由、文档权威、计划收口、多阶段独立复核、性能红线、安全红线、诊断基础设施和漂移治理一层一层建立了起来。这里被替代掉的，不只是部分编码动作，而是传统工程中相当大一块"靠人直接读代码、靠多人彼此提醒、靠长期手工维护上下文"来完成的控制能力。很多看上去偏重的机制，在这种模式里其实是最低限度的支撑结构；没有它们，AI 确实可以继续产出代码，但系统很难继续维持抽象稳定、语义收敛和长期可演化性。

但这篇文章真正想说的不是"照抄某个仓库怎么做"，而是更普遍的一点：

**大型框架的 AI 最佳实践，本质上就是一套保护抽象、约束语义、控制复杂度、强化验证、抵消自洽偏差的方法论。**

AI 不能替代这套方法论。它只会把它的效果放大。
