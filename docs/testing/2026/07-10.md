## 3. 非搜索型 Select 未选中时 trigger 不可见（高度仅 16px、无边框无背景）

- **所属组件**: Select (flux-renderers-form / @nop-chaos/ui)
- **问题描述**: 在 `http://localhost:5173/#/lab/select` 页面中，非搜索、非多选的 Select 控件在未选中任何选项时 trigger 按钮仅显示一个 16px 的下拉箭头图标，没有边框、内边距、高度或背景色，看起来像宽度为 0。根因是 `ComboboxTrigger`（`packages/ui/src/components/ui/combobox.tsx`）的 base className 仅有 `flex items-center justify-between gap-1.5`，缺少任何视觉样式；在 `ComboboxInput`（搜索型）场景中 trigger 借助 `InputGroupButton` 获得样式，但独立使用（非搜索型）时 trigger 裸露无样式。`SelectRenderer`（`packages/flux-renderers-form/src/renderers/input-choice-renderers.tsx`）的非搜索分支传入的 `className="flex-1 justify-between"` 也未包含任何视觉类。
- **预期行为**: 非搜索型 Select 的 trigger 应具有与其他输入控件一致的边框、高度（h-8）、内边距、背景色和聚焦环。
- **发现方式**: 人工测试 + Playwright programmatic inspection（`getBoundingClientRect()` 确认 width=918px / height=16px）
- **状态**: 已修复（`input-choice-renderers.tsx` 的非搜索型 `ComboboxTrigger` className 补充 `h-8 rounded-lg border border-input bg-transparent px-2.5 pr-2 text-sm outline-none transition-colors focus-visible:border-ring focus-visible:ring-3 focus-visible:ring-ring/50 disabled:cursor-not-allowed disabled:opacity-50 aria-invalid:border-destructive data-placeholder:text-muted-foreground`）

## 4. 非搜索型 Select 未选中时下拉箭头在左侧，选中后跳到右侧

- **所属组件**: Select (flux-renderers-form / @nop-chaos/ui)
- **问题描述**: `ComboboxTrigger` base class 用 `justify-between`，children 为 `{ComboboxValue}` + `<ChevronDownIcon>`。未选中时 `ComboboxValue` 不渲染 DOM（无文本节点），flex 只有 1 个子元素（icon），`justify-between` 将其放在左侧；选中后 `ComboboxValue` 渲染文本，变为 2 个子元素，icon 被推到右侧。导致箭头位置不一致。
- **预期行为**: 箭头始终在 trigger 最右侧，与是否选中无关。
- **发现方式**: 人工测试 + Playwright `innerHTML` 检查（未选中时 `<span class="flex-1 truncate text-left"></span>` 为空，仅 icon 可见）
- **状态**: 已修复（`ComboboxValue` 包裹在 `<span className="flex-1 truncate text-left">` 中，空 span 仍占据全部可用空间，将 icon 始终推至右侧）

## 5. 搜索型 Select 选中后重新打开下拉列表只显示 1 条选项（过滤残留）

- **所属组件**: Select (flux-renderers-form)
- **问题描述**: 搜索型 Select（如 TeamMember `searchable: true`）选中某项后，`onInputValueChange` 将 `inputValue` state 设为所选项的 label（如 "Alice Chen"）。`inputValue` 在 dropdown 关闭时未重置，重新打开时 `query = inputValue` 仍为 "Alice Chen"，`visibleOptions` 被过滤为仅匹配该 label 的 1 条。
- **预期行为**: 重新打开下拉列表时显示全部选项，不残留上次选中产生的过滤条件。
- **发现方式**: 人工测试 + Playwright（选中后 reopen，`optionsAfter.length` 应等于全部选项数）
- **状态**: 已修复（`Combobox` 的 `onOpenChange` 回调在 `open === false` 时 `setInputValue('')`，清空过滤条件）

## 6. 非必选 Select 选中后无法清除选择

- **所属组件**: Select (flux-renderers-form)
- **问题描述**: `clearable` prop 默认为 `false`，非必选 Select 选中某项后无法回到"未选"状态。用户需要额外在下拉列表中添加空条目才能清除选择，体验不佳。
- **预期行为**: 非必选（`required` 未设或为 `false`）的 Select 自动启用 `clearable`，选中后 trigger 旁显示清除 (x) 按钮。必选 Select 不显示清除按钮。显式设置 `clearable` 时覆盖默认值。
- **发现方式**: 人工测试
- **状态**: 已修复（`input-choice-renderers.tsx` 中 `clearable` 默认值从 `Boolean(props.props.clearable)` 改为 `props.props.clearable !== undefined ? Boolean(props.props.clearable) : !props.props.required`）

## 7. Input-Year 控件无法输入

- **所属组件**: Input-Year / PeriodRenderer (flux-renderers-form)
- **问题描述**: `PeriodPicker`（`period-renderers.tsx`）中 year 类型的 `<Input>` value 从 `parsePeriod(value, 'year', 'YYYY')` 派生出 `yearValue`。用户输入部分年份如 "2" 时，`parsePeriod("2", "year", "YYYY")` 解析失败返回 `undefined`，导致 `yearValue=''`，输入被清空，无法逐字输入。
- **预期行为**: 用户可以正常逐字输入年份（如先输入 "2" → "20" → "202" → "2025"），不受中间态解析失败影响。
- **发现方式**: 人工测试 + Playwright（逐字输入后检查 `input.value`，修复前每一步均为空字符串）
- **状态**: 已修复（year Input 的 `value` 从 `yearValue`（parsePeriod 派生）改为直接使用 `value ?? ''`，避免中间态解析失败）

## 10. Textarea rows 属性无效（所有 textarea 高度相同 64px）

- **所属组件**: Textarea (@nop-chaos/ui) / TextareaRenderer (flux-renderers-form)
- **问题描述**: `Textarea` base 组件（`packages/ui/src/components/ui/textarea.tsx`）的 className 包含 `field-sizing-content`（CSS `field-sizing: content`），使 textarea 按内容自动调整高度，完全忽略原生 `rows` 属性。加上 `min-h-16`（64px），所有 textarea 不论 `rows` 值（3/4/5）高度均为 64px。
- **预期行为**: `rows` 属性控制 textarea 的初始可见行数（高度），用户输入超出后滚动。
- **发现方式**: 人工测试 + Playwright `getComputedStyle`（`fieldSizing: "content"`，三个 textarea 高度均为 64px）
- **状态**: 已修复（从 Textarea base 组件移除 `field-sizing-content` class；`field-sizing` 默认为 `fixed`，`rows` 属性正常生效）
