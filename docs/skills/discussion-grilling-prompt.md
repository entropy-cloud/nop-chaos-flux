# 讨论盘问提示词（Discussion Grilling）

> **定位**: 与用户开展**多轮、交互式、一次一个问题**的盘问式讨论，把模糊的需求/方案/设计判断逼问到收敛。讨论过程**逐轮落盘**到 `docs/discussions/`，文件结构严格遵循 `docs/discussions/README.md` 的"文件模板"。本提示词是该 README 缺失的**纪律层**——那份 README 规定"存哪里、怎么排版"，本提示词规定"怎么盘问、怎么边问边写"。

---

## 与相邻工具的边界

三个工具名字/形态接近，按"你要回答的问题"选择，不要按关键词相似度选择：

| 你在回答的问题                              | 用                                                                 | 不要用                                                            |
| ------------------------------------------- | ------------------------------------------------------------------ | ----------------------------------------------------------------- |
| "我想和多轮交互地把一个需求/方案逼问到收敛" | **本文件**（交互式盘问 → `docs/discussions/`）                     | `plan-grilling-question-document-prompt.md`（单次问题包，非交互） |
| "我想一次性生成一份可勾选的决策问题包"      | `plan-grilling-question-document-prompt.md`（→ `docs/analysis/`）  | 本文件（多轮对话，不是单次产物）                                  |
| "需求还很模糊，要数学化量化模糊度再动手"    | `.opencode/skills/nop-deep-interview`（苏格拉底+评分，输出到对话） | 本文件（不评分，产物是落盘讨论文件）                              |
| 明确的 bug 诊断                             | `docs/skills/bug-diagnosis-prompt.md`                              | 本文件                                                            |
| 已经收敛、只需执行                          | 直接执行                                                           | 本文件                                                            |
| 纯文档勘误、单文件改动                      | 直接改                                                             | 本文件                                                            |

**deep-interview vs 本文件的判断准则**：若你关心"需求还要多清晰才能动手"且想要数值化评分 → deep-interview；若你关心"这个设计判断的每个分支该怎么走"且想要一份落库的逐轮记录 → 本文件。两者都一次只问一个问题，但 deep-interview 由模糊度评分驱动，本文件由决策树下行驱动。

## 前置阅读（执行前必读，全文档唯一一处清单）

1. `docs/index.md`
2. `AGENTS.md`
3. `docs/discussions/README.md`（文件命名、模板、纠正规则的权威）
4. 目标 area 的 owner 文档（从 `docs/index.md` 路由）
5. 若讨论触及历史反复 reopened 的设计判断：`docs/references/reopened-design-decisions-and-audit-adjudications.md`
6. 必要时核对相关 live code（涉及包边界/owner 归属时对照 `docs/architecture/flux-runtime-module-boundaries.md`）

## 核心纪律（七条，全文档唯一一处）

任何一条被破坏，讨论就退化为低效闲聊：

1. **对话一次一问，文件全景留痕。** 对话回复里一次只抛一个问题；但每轮文件的 `### 待澄清问题` 要记录该轮识别出的**全部** unresolved 分支。文件是决策树的全景快照，对话是单分支推进器——这是本提示词区别于普通一问一答的关键机制。

2. **推荐答案：对话必带，文件可选。** 对话里抛出的每个问题都必须附带推荐答案 + 一句理由（用户只需"确认/否决/调整"，让讨论收敛而非发散）。落盘到 `### 待澄清问题` 时，推荐答案**可选**——为可读性可只列问题（与既有讨论如 `2026-06-29-crud-loadaction-picker-design.md` 一致；该文件早于本纪律，不作为豁免先例，仅说明文件层风格）。

3. **代码/文档优先于提问。** 一个问题若能读 owner docs 或 live code 直接回答，**先去读，不要问用户**。只有真正 unresolved、不同答案会改变结果的分支，才留给用户。

4. **沿决策树逐分支下行，先解依赖。** 先问"后续问题依赖它"的高层决策；单条链路一问到底再回溯，避免在依赖未定时去问被依赖的细节。

5. **逐轮实时落盘。** 第一轮开始就建文件；每一轮（用户输入 + AI 整理 + 本轮问题）成形即追加。讨论文件是**边走边长的活文档**，不是结束后凭记忆补的纪要。

6. **术语即出即磨，决策即出即记。**
   - 用户用了与 `docs/references/terminology.md`、`quick-reference.md` 或 owner docs 冲突/含糊的词，**当场指出并要求选定**一个规范词。被选定的若是**跨讨论复用的规范词**，当场同步写入 `docs/references/terminology.md`；仅局部一次性术语留在讨论文件内即可。
   - 某个决策结晶（hard-to-reverse + 无背景会让人困惑 + 存在真实取舍，三者同时成立），当场在该轮新增 `### 决策` 子段记成一条；`## 总结` 的 `### 关键决策` 在结束时聚合这些条目，不重复论证。
   - 不要发明概念去显得深刻；术语服务于消除歧义。

7. **纠正优先于保留历史。** 发现此前某轮整理有误/缺漏/过度压缩，**直接修正该轮原文**，并在同轮或专门的 `### 修正说明` 写清"原先错在哪 / 为什么 / 用户如何纠正 / 修正后表述"。不要为"留痕"保留明显错误的文本——讨论文件的目标是**既保留演进过程，又让后续 AI 读到的是准确记录**。

## 落盘规则

### 文件命名

- 路径：`docs/discussions/{YYYY-MM-DD}-{简短主题}.md`（主题 kebab-case）。
- 日期为讨论创建当天；同日多讨论用 README 规定的递增后缀（`-a`/`-b`）区分，不覆盖旧文件。
- 第一轮回复时必须给出保存路径。

### 每轮结构（严格照搬 `docs/discussions/README.md` 的"文件模板"）

**第 1 轮**：

```text
## 第 1 轮

### 用户原始表述
（用户原话或大意）

### AI 复述
（AI 对用户意图的结构化整理）

### AI 分析
（详细分析；引用项目文档/代码路径）

### 待澄清问题
1. <问题>            ← 推荐答案可选；对话里抛出时必带
2. <问题>

---
```

**第 2 轮及以后**：

```text
## 第 N 轮

### 用户回答
（用户对上一轮问题的回应）

### AI 整理
（对用户回答的整理）

### 待澄清问题
1. <问题>

---
```

> 第 2 轮起用 `### 用户回答` + `### AI 整理`，**没有 `### AI 分析`**——这与 README 模板及既有讨论一致。若某轮确需补充分析，并入 `### AI 整理`，不自创标题。`### 决策`、`### 修正说明` 仅在需要时出现。

## 异常与边界

| 情形                               | 处理                                                                                                         |
| ---------------------------------- | ------------------------------------------------------------------------------------------------------------ |
| **用户中途放弃**                   | 写 `## 总结`，`### 待定事项` 列出全部未决分支；文件标题或首行标注 `（未完成）`，便于后续检索清理。           |
| **推荐答案被否决**                 | 在该轮 `### 待澄清问题` 记录用户的替代答案 + 一句理由；若替代引发新的依赖分支，纳入下一轮。                  |
| **新信息推翻已结晶的旧决策**       | 重开受影响的旧轮，加 `### 修正说明` 写清新信息如何改变结论，并把传播到下游轮次的影响一并修正（纪律 7）。     |
| **单轮 unresolved 分支过多（>6）** | 大概率是两个讨论混在一起——考虑拆成两个讨论文件；若确认强耦合，合并但在总结的 `### 关键决策` 里按子主题分组。 |
| **用户主动批量回答多个分支**       | 照单全收并分别记录，**不强制拆成多轮**；但下一轮对话仍只推进一个最高位分支。                                 |

---

## 结束条件与总结

**何时结束**：决策树上每个 unresolved 分支都被确认（确认 = 用户对推荐答案点头，或给出明确替代）。不为凑轮数继续问；也不在还有真实未决分支时提前收尾。

结束时写 `## 总结`，结构遵循 README：

```text
## 总结

### 最终需求
（完整、收敛后的需求/方案描述）

### 关键决策
- （聚合各轮 `### 决策` 条目；标明哪些因 hard-to-reverse + 无背景会困惑 + 真实取舍，值得后续单独记 ADR 或更新 architecture 文档）

### 待定事项
- （确实还无法定的；没有就写"无"）

### 后续行动
- （结论影响的下游：要更新/新建的 docs/architecture 文档、要改的代码路径、要记的 logs 等，尽量给出具体路径）
```

> 若结论影响架构，必须在总结里点名要同步更新的 `docs/architecture/` 文档，并在 `docs/logs/{year}/{month}-{day}.md` 记录——这条直接引用自 `docs/discussions/README.md` 注意事项。
