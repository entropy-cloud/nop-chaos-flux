## P2 Findings (2)

### [NEW-MM-01] `infinite-scroll.tsx`: bare `catch {}` + async `.catch(()=>{})` swallow ALL `onLoadMore` failures with zero dev diagnostic

- **Files**: `packages/flux-renderers-mobile/src/infinite-scroll.tsx:66-75`
- **Dimension**: 06 (async safety) / 19 (error propagation)
- **Evidence**:
  ```ts
  const fireLoadMore = React.useCallback(() => {
    isLoadingRef.current = true;
    try {
      void Promise.resolve(onLoadMoreRef.current?.()).catch(() => {
        /* host owns error surfacing via the `error` prop */
      });
    } catch {
      /* sync throw swallowed; in-flight guard stays until host clears loading */
    }
  }, []);
  ```
- **Severity**: P2
- **现状**: The crash-safety half of MA-14 is genuinely closed (the renderer never throws — `infinite-scroll.test.tsx:304` confirms). But the error-propagation half is fully silent: there is no `console.error`, no telemetry, nothing, in either the async-reject or sync-throw branch. `isLoadingRef` is released only on a `loading` prop transition (`:58-60`); a host that rejects `onLoadMore` and forgets to flip `error` (a very common integration mistake) produces a completely invisible failure with no signal for the integrator.
- **风险**: Hosts that violate the documented error contract leave the list silently dead-locked (host never touches `loading`) or silently thrashing (host toggles `loading` without `error`) with zero diagnostic. This is strictly weaker than the repo's own convention: other renderer packages log action failures as `console.warn`/`console.error` (e.g. `flux-renderers-form-advanced/.../variant-field-controller.ts`, `flux-react/src/renderer-helpers.ts:132`).
- **建议**: Keep the swallow for crash-safety, but add a gated dev-mode diagnostic, e.g. `if (import.meta.env?.DEV) console.error('[flux.infinite-scroll] onLoadMore rejected and host did not set error=true', err);` inside the `.catch`. This matches the repo's existing dev-warning pattern and makes host-contract violations diagnosable without changing runtime control flow.
- **误报排除**: Not a low-code dynamic-boundary allowance (the error contract is explicitly documented and the renderer already half-implements it via the retry `<Button>`). Not the resolved MA-14 crash-safety (that part passes). This targets the residual diagnosability gap.
- **复核状态**: 维度复核 + 子项复核通过 — 保留 (P2). Independent review confirmed the swallow is real, no diagnostic exists, and the documented host-contract is not a sufficient safety net.

### [NEW-MM-02] `infinite-scroll.tsx`: error status row conflates `role="status"` with operability (`tabIndex` + `onClick` + `onKeyDown`) AND nests a `<Button>` — duplicate tab stop + mixed a11y semantics

- **Files**: `packages/flux-renderers-mobile/src/infinite-scroll.tsx:145-189`
- **Dimension**: 09 (renderer contract) / 20 (a11y)
- **Evidence**:
  ```tsx
  <div
    data-slot="infinite-scroll-status"
    role="status"
    aria-live="polite"
    onClick={status === 'error' ? triggerLoadMore : undefined}
    onKeyDown={status === 'error' ? (event) => { if (event.key !== 'Enter' && event.key !== ' ') return; event.preventDefault(); triggerLoadMore(); } : undefined}
    tabIndex={status === 'error' ? 0 : undefined}
  >
    ...
    {status === 'error' ? (
      <Button ... onClick={(event) => { event.stopPropagation(); triggerLoadMore(); }}>{errorText}</Button>
    ) : null}
  ```
- **Severity**: P2
- **现状**: In the error state the outer status `<div>` is simultaneously an advisory live-region (`role="status"` + `aria-live="polite"`) AND an interactive control (`tabIndex=0` + Enter/Space handler + click handler), and it wraps a real `<Button>` with the same label and same action.
- **风险**: Two a11y regressions. (1) Duplicate tab stop for one action: a keyboard user Tabs to the `<div>` (Enter/Space triggers retry), then Tabs to the nested `<Button>` (Enter triggers retry again) — two focusable controls for one semantic action. (2) `role="status"` on a focusable/activatable element is the exact anti-pattern `notice-bar.tsx` was just refactored to avoid — `notice-bar.tsx:112-120` (OA-04): "Mixing role=alert with tabIndex=0 on the same element conflated assertive announcement with operability." Screen readers announce the error text via the live-region AND re-announce it as a button label.
- **建议**: Pick one operability model. Cleanest: drop `tabIndex`/`onClick`/`onKeyDown` from the outer `<div>` and let the inner `<Button>` be the sole retry control (the `<Button>` already supplies the right role, tab stop, and keyboard handling for free). Keep `role="status"` on the `<div>` for non-error announcement only. If a larger touch target is desired, style the `<Button>` to fill the row rather than promoting the `<div>` to interactive.
- **误报排除**: Not OA-10 (OA-10 is the IntersectionObserver auto-fire guard when `error` is set, which IS implemented and tested). This is a separate a11y-semantics concern on the retry UI. Not covered by any prior finding.
- **复核状态**: 维度复核 + 子项复核通过 — 保留 (P2). Independent review confirmed the duplicate-tab-stop and the same `role=status`-on-focusable anti-pattern notice-bar already rejected.

### [NEW-MM-03] `pull-refresh.tsx`: re-entrancy guard reads `statusRef` mirrored via passive `useEffect` — inconsistent with swipe-cell's strictly-safer synchronous mirror

- **Files**: `packages/flux-renderers-mobile/src/pull-refresh.tsx:59-62, 107-115`
- **Dimension**: 07 (lifecycle)
- **Evidence**:
  ```ts
  const statusRef = React.useRef<PullRefreshStatus>('normal');
  React.useEffect(() => {
    statusRef.current = status;
  }, [status]);
  ...
  const handleTouchEnd = React.useCallback(() => {
    touchHandlers.onTouchEnd();
    if (disabled) return;
    const current = statusRef.current;          // read for re-entrancy guard
    if (current === 'loading' || current === 'success') return;
    if (directionalDelta >= threshold) {
      setStatus('loading');                      // ref NOT updated synchronously here
  ```
- **Severity**: P3
- **现状**: pull-refresh mirrors `status → statusRef` inside a passive `useEffect`, so the ref lags the committed state by one effect-flush. swipe-cell solves the identical problem (ref mirror for a re-entrancy guard after MA-02 moved dispatch out of the updater) by updating `openStateRef.current` synchronously inside `closeCell`/`openCell` (`swipe-cell.tsx:121-135`), with a comment explaining why synchronous mirroring is necessary.
- **风险**: There is a theoretical TOCTOU window: `setStatus('loading')` commits, but `statusRef.current` is still `'normal'` until the passive effect runs. If a second `touchEnd` arrived in that window, the guard would pass. In practice native touch events are macrotasks and React flushes passive effects between them, so the window is unreachable by real input — this is **not** a proven live bug. The deeper concern is that the two renderers now solve the same problem two ways, and the re-entrancy test (`pull-refresh.test.tsx:293-327`) passes regardless of mirror timing.
- **建议**: Align with swipe-cell's pattern — set `statusRef.current = 'loading'` synchronously in `handleTouchEnd` right after `setStatus('loading')` (and similarly for the `'success'`/`'normal'` transitions in the promise chain), so the guard is lag-free and the two renderers converge on one pattern.
- **误报排除**: Not the resolved MA-02 (dispatch is correctly in the handler body, not the updater) and not MA-01/MA-12 (reject + unmount guards are present and correct). This is narrowly about the ref-MIRROR timing; impact is nil, hence P3 (consistency), not higher.
- **复核状态**: 维度复核 + 子项复核通过 — 保留 (P3).

### [NEW-MM-05] `swipe-cell` `design.md` documents a `deltaX * 0.3` edge-damping behavior that the renderer does not implement

- **Files**: doc `docs/components/swipe-cell/design.md:95` vs code `packages/flux-renderers-mobile/src/swipe-cell.tsx:88-113`
- **Dimension**: 16 (doc-code consistency)
- **Evidence** (doc):
  ```
  - 拖拽阻尼：接近操作区边缘时减速 `deltaX * 0.3`
  ```
  (code — no damping factor anywhere; finger tracks 1:1 then clamps):
  ```ts
  const activeDragOffset = state.direction === 'horizontal' && state.isTouching ? state.deltaX : 0;
  ...
  if (dragOffset > 0 && canSwipeRight) return clamp(dragOffset, 0, leftWidth || dragOffset);
  if (dragOffset < 0 && canSwipeLeft) return clamp(dragOffset, -rightWidth || dragOffset, 0);
  ```
- **Severity**: P3
- **现状**: `design.md §6` promises edge-damping (`deltaX * 0.3` near the action-region edge), but `swipe-cell.tsx` tracks the finger 1:1 (`activeDragOffset = state.deltaX`) and only hard-clamps at `±leftWidth/±rightWidth`. There is no `0.3` (or any) deceleration factor. (`pull-refresh.tsx:10,84` DOES implement its own `DAMPING_FACTOR = 0.5`, so the damping concept exists in the package — swipe-cell just omits it.)
- **风险**: Doc-code drift. Anyone reading `design.md` to calibrate swipe feel, write a regression test, or build a visual spec will expect rubber-band deceleration at the edge that does not exist; conversely a maintainer "fixing" the clamp may add damping assuming it is missing. Low functional impact (the cell still works), but a real contract inconsistency.
- **建议**: Either implement the documented deceleration (apply a sub-linear scale to `dragOffset` once `|dragOffset|` exceeds some fraction of the region width) or update `design.md §6` to state the content tracks the finger 1:1 and clamps at the region width (no edge damping), whichever matches product intent.
- **误报排除**: Not a low-code allowance (`design.md` is the component contract doc). The rest of `design.md` (touch-action pan-y, touchcancel no-commit, inert, onAction+autoclose) DOES match the code — only the §6 damping line drifts.
- **复核状态**: 维度复核 + 子项复核通过 — 保留 (P3).

### [NEW-MM-06] `styles.css` comment claims tokens derive "from the shared theme layer with package-local fallbacks", but `:root` hardcodes fixed hsl values with no shared-var read

- **Files**: `packages/flux-renderers-mobile/src/styles.css:24-39` (light) + `:42-53` (dark)

- **Evidence**:

  /* MA-06: variant palette now derives from theme tokens with explicit dark
   * overrides ... Tokens use hsl() fragments from the shared theme layer with

- **Severity**: P3

## Rejected findings (1)

### [NEW-MM-07-REJECTED] ~~`swipe-cell` `onAction` drops the native click event, inconsistent with MA-04's Event Passthrough Contract~~

- **原严重程度**: P2 → **驳回**
- **复核理由**: `onAction` is a **semantic gesture event**, not a raw DOM entry point. The Event Passthrough Contract in `docs/architecture/renderer-runtime.md:650-671` lists DOM entry points (`onClick`/`onChange`/`onSubmit`/`onFocus`/`onBlur`) as requiring native-event forwarding, and a second clause explicitly permits: "Non-DOM semantic payloads are still allowed when a renderer emits higher-level interaction data, but those payloads should still carry a meaningful `type` field." `onAction` carries `{ type: 'action', side }` and is structurally identical to the sibling semantic events `onOpen`/`onClose` (`{type,side}`) and `onRefresh` (`{type:'refresh',...}`) — all of which the package deliberately emits as structured payloads. The package's own test suite tags a `{type:'refresh'}` payload as "MA-04" (`pull-refresh.test.tsx:371`), confirming the package treats gesture/interaction events as semantic payloads. notice-bar's `onClick`/`onClose` forwarding is correct because those ARE raw DOM entry points — a different category. The deep-dive conflated DOM-entry-point events with semantic gesture events. Highest-risk false positive; rejected.
