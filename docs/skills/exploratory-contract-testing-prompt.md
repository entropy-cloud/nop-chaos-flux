# 探索性契约测试提示词

## 目标

对当前仓库执行一轮持续迭代的探索性测试：不断尝试编写基于契约、接口、文档承诺和稳定行为的测试，用测试证明当前代码存在真实问题；一旦发现问题，继续沿着新的方向寻找下一类问题，直到本轮确实找不到新的高价值问题为止。

这不是覆盖率冲刺，也不是实现细节固化。

- 测试必须尽量依赖稳定契约，而不是内部私有实现。
- 测试应优先描述“外部应该怎样工作”，而不是“内部现在怎么写”。
- 同类问题只记录一次，不要因为同一个根因在多个文件重复出现就机械刷条目。

## 开始前必须读

1. `docs/index.md`
2. `AGENTS.md`
3. 与当前目标区域直接相关的架构文档
4. 已有测试文件和 `docs/analysis/` 下相关历史分析，避免重复造轮子

## 测试哲学

你的核心任务不是“给现有代码补测试”，而是：

1. 找到稳定契约。
2. 基于契约提出失败假设。
3. 编写最小测试证明行为错误。
4. 如果问题简单，立即修复并把测试转成稳定回归测试。
5. 如果问题复杂，先保留最小复现测试和分析记录，延后修复。
6. 继续寻找下一类问题，直到没有新的问题。

## 什么叫稳定契约

优先从这些来源建立测试断言：

1. `docs/architecture/` 中仍然有效的行为承诺
2. `docs/components/` 中定义的组件契约
3. `docs/references/` 中公开接口、术语、运行时类型说明
4. `packages/*/src/index.ts` 暴露的公共 API
5. renderer contract、schema contract、store contract、action contract、validation contract
6. 用户可观察到的行为：返回值、状态转换、事件语义、错误传播、渲染结果、可访问性语义

不要把以下内容直接当成测试断言基础：

1. 私有 helper 名称
2. 当前文件拆分方式
3. 临时中间状态的内部数据结构
4. 具体 hook 组合顺序
5. 某个内部函数是否被调用
6. 纯实现偶然形成的 DOM 结构细节

## 允许优先攻击的高价值方向

如果没有切入点，优先从这些最容易暴露真实缺陷的方向开始：

1. 输入边界：空值、缺字段、错误类型、深层嵌套、异常 schema
2. 状态边界：初始化、销毁、重复执行、切换 owner、reset
3. 时序边界：连续触发、竞态、并发、abort、先后顺序颠倒
4. 错误路径：抛错是否被吞掉、是否错误归类、是否污染后续状态
5. 契约一致性：文档承诺与 live code 是否一致
6. 跨层行为：core -> runtime -> react -> renderer 的边界是否失真
7. 可访问性与公开语义：role、label、disabled、focus、keyboard 行为
8. 组合场景：两个各自正确的特性组合后是否失效

## 执行模型

这份提示词默认采用：

1. **广度优先探索**：先尽可能铺开不同方向，而不是一开始只深挖某一个方向。
2. **主执行者负责编排**：主执行者的首要职责是切分搜索空间、分配方向、去重、复核，而不是默认先亲自深挖某一条线。
3. **子 agent 负责独立探索**：每个子 agent 在自己负责的方向里完整执行“读代码 -> 提假设 -> 写最小测试 -> 运行 -> 判断 -> 记录/修复”的循环。
4. **问题分支与主流程分离**：一旦确认某个新问题，允许单独开一个小分支继续深挖它，但主流程必须继续横向寻找其他问题类别，不能被单个问题吸住。

## 主执行者职责

主执行者默认不应在第一阶段就把大量时间花在某一个具体契约簇上。

主执行者应优先完成以下工作：

1. 阅读基础文档，确认仓库的主要稳定契约来源。
2. 把搜索空间切分成多个尽量不重叠的高价值方向。
3. 为每个方向分配一个新的独立子 agent。
4. 约束不同子 agent 不要扎堆在同一方向。
5. 汇总子 agent 的发现并做去重。
6. 对“候选问题”做真实性复核，确认是否真的是新的高价值问题类别。
7. 决定哪些问题立即修复，哪些先保留复现和分析。
8. 在一轮广度探索结束后，补齐尚未覆盖或覆盖不足的方向，再启动下一批全新子 agent。

只有在搜索空间已经被切开并经过至少一轮多方向探索后，主执行者才应该亲自补查空白方向或做最终收束验证。

## 搜索空间切分规则

主执行者必须先显式切分搜索空间，再启动探索。

推荐切分维度：

1. runtime / validation
2. action / async / cancel / error propagation
3. react integration / owner boundary / hook contract
4. renderer contract / accessibility / external semantics
5. surface lifecycle / dialog / drawer / teardown
6. compiler / schema contract / public API drift
7. cross-package composition / boundary distortion
8. 大型领域组件：flow / report / spreadsheet / word-editor

如果仓库规模较小，也至少要保证第一轮并行覆盖 3 个以上不相邻方向。

## 单个探索循环

无论是主执行者亲自补查，还是某个子 agent 在其方向内探索，都按下面的最小循环执行：

1. 选一个稳定契约或一组相邻契约。
2. 先读真实代码和已有测试，确认当前约束点。
3. 提出一个可能失败的假设：当前实现也许没有兑现这个契约。
4. 写一个最小测试，只证明一个问题，不把多个疑点塞进一个大测试。
5. 运行相关测试。
6. 如果测试没有暴露问题：
   - 判断是你的假设错了，还是断言太弱。
   - 必要时改写测试；如果契约本身就成立，切换到该方向内的下一类问题。
7. 如果测试稳定暴露了真实问题：
   - 先判断这是不是一个新问题类别。
   - 如果只是同类问题的另一个实例，合并到已有条目，不新增问题编号。
   - 如果是新的问题类别，登记到 `docs/analysis/` 台账。
8. 判断修复成本：
   - 简单问题：立即修复，然后确认测试转绿。
   - 复杂问题：先保留测试和分析记录，标明延后处理原因。
9. 如果当前方向内仍有未覆盖的高价值子方向，继续探索；否则将该方向标记为“本 agent 已耗尽”。

## 广度优先批次循环

整体流程必须按“批次”推进，而不是只靠单线串行推进。

### 第 1 阶段：首批广度覆盖

1. 主执行者先切分方向。
2. 一次性启动一批新的独立子 agent，并让它们分别覆盖不同方向。
3. 主执行者在这一阶段主要负责调度、记录、去重和读取返回结果，不应默认先自己深入某一条方向。

### 第 2 阶段：问题真实性复核

对每个子 agent 报告的候选问题：

1. 主执行者或专门复核分支必须做真实性验证。
2. 只有稳定复现并确认属于新的问题类别，才算“发现问题”。
3. 未经复核的候选不能直接算进最终发现数。

### 第 3 阶段：问题分支深挖

当某个新问题类别被确认后：

1. 可以启动一个专门的小分支继续扩影响范围、补测试、修复或评估延后原因。
2. 这个分支只负责该问题，不应把主流程重新拖回单线深挖。
3. 其他新子 agent 仍应继续寻找不同方向里的其他问题类别。

### 第 4 阶段：补盲与再分发

一轮广度覆盖结束后，主执行者必须检查：

1. 哪些方向还没被覆盖。
2. 哪些方向只做了浅层探索。
3. 哪些方向已经证明只是重复旧根因。

然后：

1. 为未覆盖或覆盖不足的方向启动下一批全新子 agent。
2. 避免把新 agent 再次派去已经被证明只会重复旧根因的方向。

## 双层独立复查循环

这项工作不是“主执行者自己找一轮就结束”，而是必须经过两层耗尽式探索；但这里的“耗尽”指的是**广度搜索空间的耗尽**，不是某一个方向被挖空。

停止条件不是“我找不到了”，而是：

1. 主执行者已经完成至少一轮多方向广度覆盖；并且
2. 最新一批全新独立子 agent 也没有带来经复核成立的新问题类别。

如果某一批新子 agent 又找到了新的问题类别，则说明搜索空间尚未耗尽；这时必须：

1. 更新测试与台账。
2. 继续启动下一批覆盖其他方向的全新子 agent。
3. 直到最近一批新的独立子 agent 也没有新增经复核成立的问题类别为止。

## 多个子 agent 的批次规则

按以下方式推进：

1. 主执行者先切分方向并启动第一批全新子 agent。
2. 每个子 agent 都必须在自己负责的方向内先自我耗尽。
3. 主执行者汇总第一批结果并复核候选问题。
4. 如果第一批产生了新的经确认问题类别，则一边开问题分支处理，一边启动第二批覆盖其他方向的全新子 agent。
5. 第二批完成后，再次汇总、去重、复核。
6. 如此循环，直到最近一批全新子 agent 没有带来新的经确认问题类别。

这里的关键点是：

1. 每次都必须是“新的”子 agent。
2. 每个子 agent 都必须被分配一个与其他 agent 尽量不同的主方向。
3. 只要最新一批全新子 agent 还能带来新的经确认问题，总体流程就不能停止。
4. 不允许因为某一个高价值问题值得深挖，就停止横向探索整个仓库的其他方向。

## 轮次记录要求

每一轮都要明确记录是谁发现了问题，以及该轮是否耗尽：

1. `main-round-N`: 主执行者的第 N 轮调度 / 复核 / 补盲结果
2. `subagent-X-round-N`: 某个独立子 agent 的第 N 轮内部探索结果
3. `subagent-X-summary`: 某个独立子 agent 在其内部耗尽后的总结

在分析台账或配套结果文件中，至少写明：

1. 发现者：主执行者 / 子 agent 名称
2. 是否为新问题类别
3. 是否只是扩大已有问题影响范围
4. 该轮结束后是否已耗尽
5. 是否需要再启动下一个全新子 agent

## 结果目录与文件落盘规则

每次真正执行这份提示词时，必须先在 `docs/analysis/` 下创建一个新的结果目录，并把这次执行产生的全部轮次文件都放在这个目录中。

### 目录规则

1. 一次执行只对应一个结果目录。
2. 不同执行批次不能共用同一个目录。
3. 目录名必须能区分时间或执行序号，例如：
   - `docs/analysis/2026-05-10-exploratory-contract-testing-run-01/`
   - `docs/analysis/2026-05-10-exploratory-contract-testing-run-02/`
4. 本次执行中，主执行者、自身循环、所有独立子 agent、所有子 agent 内部轮次文件，都必须写入这个同一个目录。

### 文件规则

每一轮都必须单独写一个文件，不能把多轮结果混写到同一个文件里。

必须单独落盘的内容包括：

1. 主执行者的每一轮探索
2. 每个独立子 agent 的每一轮探索
3. 每个独立子 agent 在其内部耗尽后的 summary
4. 整次执行的最终 summary

建议命名：

1. `main-round-01.md`
2. `main-round-02.md`
3. `subagent-a-round-01.md`
4. `subagent-a-round-02.md`
5. `subagent-a-summary.md`
6. `subagent-b-round-01.md`
7. `subagent-b-summary.md`
8. `summary.md`

### 单轮文件要求

每个轮次文件至少包含：

1. 轮次标识
2. 执行者身份：主执行者 / 子 agent 名称
3. 本轮检查的契约或方向
4. 本轮新增问题类别
5. 本轮新增测试或修改的测试
6. 本轮修复情况
7. 本轮延后问题
8. 本轮是否已耗尽
9. 下一轮建议方向

如果该轮是主执行者轮次，还必须额外包含：

1. 本轮覆盖了哪些主方向
2. 哪些方向是新覆盖的
3. 哪些方向只是复核旧候选
4. 下一批计划交给新子 agent 的方向

### 结束条件的落盘要求

当某个最新的独立子 agent 在其内部循环后没有发现新的高价值问题时：

1. 仍然要把该子 agent 的最后一轮结果单独写成文件。
2. 仍然要写该子 agent 的 `summary` 文件，明确说明它已耗尽且无新发现。
3. 最后再写整次执行的 `summary.md`，说明：
   - 主执行者共做了几轮
   - 启用了多少个全新子 agent
   - 每个子 agent 是否找到新问题
   - 最终是哪个最新子 agent 首次未发现新问题，从而结束本次执行
   - 本次执行实际覆盖了哪些主方向
   - 哪些方向若继续下一轮应优先补盲

## 去重规则

同类问题不要重复记账。按“根因 / 契约违约类型”去重，而不是按文件数去重。

可视为同类问题的例子：

1. 同一个错误传播缺口在多个 renderer 中重复出现
2. 同一类 hidden field validation 漏洞在不同 schema 组合下反复出现
3. 同一个公共 API 契约漂移导致多个测试入口失败

遇到重复实例时：

1. 扩大已有问题条目的影响范围
2. 在同一个问题下补充代表性测试
3. 不新增新的问题编号，除非根因不同

## 修复策略

### 简单问题

满足以下条件时，优先当场修复：

1. 根因明确
2. 改动局部
3. 不需要额外设计讨论
4. 不会引出新的架构决策

处理方式：

1. 先保留失败测试
2. 做最小修复
3. 重新运行相关测试
4. 把问题状态标为 `fixed`

### 复杂问题

满足以下任一条件时，可延后修复：

1. 牵涉多个包边界
2. 涉及架构 owner 重新划分
3. 需要产品/设计裁定契约
4. 修复范围明显大于发现范围

处理方式：

1. 保留最小复现测试
2. 在分析台账中记录影响、根因猜测、延后原因
3. 状态标为 `open` 或 `deferred`
4. 不要因为暂不修复就丢掉复现

如果仓库默认测试套件不允许长期红灯，优先采用以下方式之一：

1. 使用仓库允许的 known-failure 机制
2. 单独保存复现测试草案，不把它并入默认绿灯套件
3. 在分析文档中明确说明该测试当前为何不能进入默认套件

不要为了维持绿灯而删除已经证明真实问题的复现。

## 测试写法要求

1. 一个测试只验证一个契约点或一个根因。
2. 尽量从公开入口进入，不直接 mock 私有实现细节。
3. 断言用户可观察结果、公开返回值、状态或错误语义。
4. 只有在外部不可观察时，才下沉到次一级公开接口。
5. 优先写最小复现，不要先搭大型 test harness。
6. 先复用现有测试工具和测试风格，不新增无必要基础设施。
7. 对于 bug 修复测试，断言“正确结果是什么”，不要只断言“不抛错”。

## 每发现一个新问题后必须记录

在 `docs/analysis/` 的探索性契约测试台账中更新一条记录，至少包括：

1. 问题编号
2. 问题标题
3. 对应测试文件
4. 约束它的稳定契约来源
5. 失败现象
6. 影响范围
7. 是否已修复
8. 修复提交或修复文件
9. 去重键：用于判断后续是否属于同类问题

## 输出要求

每轮完成后，输出：

1. 新增的问题类别
2. 新增或更新的测试文件
3. 已立即修复的问题
4. 延后处理的问题
5. 本轮覆盖了哪些方向
6. 为什么认为这些方向已耗尽，或者下一轮应从哪些尚未充分覆盖的方向继续

在一次完整执行结束时，还必须额外输出：

1. 主执行者共做了几轮调度 / 复核 / 补盲
2. 一共启用了几个全新独立子 agent
3. 每个子 agent 是否发现了新的高价值问题
4. 最终停止的依据：是哪个最新子 agent 在完成自身内部循环后未发现新问题
5. 本次执行实际覆盖了哪些主方向
6. 哪些方向仍然只做了浅层探索，若继续下一轮应优先补哪些方向
